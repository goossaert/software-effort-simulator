# Live sidebar preview of fitted model inputs

The simulator surfaces the *fitted* model inputs ‚Äî **Poisson Œª**, the **Bootstrap pool** size and its per-**T-shirt size** breakdown, the historical and target **Quarter** labels, and the three per-**Scenario** initiative counts (`K_must`, `K_must+should`, `K_must+should+could`) ‚Äî in a live **Data preview** block in the sidebar (`#data-preview`), painted by `renderPreview` (`index.html:2818-2846`) from the `preview` field of `prepareSimulationData`'s return value (`index.html:1779-1786`). The block re-paints automatically on every file-load, every epics-reset, and every multi-select change via the single guarded entry point `tryUpdatePreview` (`index.html:2873-2884`), which silently returns until both CSVs and at least one **Historical quarter** and one **Target quarter** are present. Alongside the preview, two collapsible `<details>` panels ‚Äî the static **T-shirt size reference** band table and the **Column-detection debug** JSON pre-block ‚Äî make the simulator's other internal model state auditable from the same sidebar location.

We chose to surface the fitted inputs **before Run** ‚Äî in the sidebar, at file-load time and on every quarter-multi-select change ‚Äî rather than only in the post-Run results panel because the cost of getting the *inputs* wrong is silently absorbed by the **Run** if the user has no way to spot the error before pressing it. A wrong **Historical quarter** selection produces a Œª fit on the wrong epic pool, a sensible-looking distribution, and a stats table the user has no reason to distrust; a wrong **Target quarter** selection produces a `K` count that gives the same illusion of correctness. The post-Run chart and table verify that *the engine ran on something*; they cannot verify that *the engine ran on the right thing*. The live preview is the surface that closes that loop: the user reads `Poisson Œª 4.32` and `Epic samples 160` and can confirm ‚Äî *before paying the iteration cost* ‚Äî that those numbers match the team's intuition for the selected quarters. A first-time user catches a column-detection error here (`Initiatives (hist.) 0`, `Poisson Œª 0.00`) instead of three Runs later. An experienced user catches a typo-in-quarter-selection here (`Hist. quarter Q2 2025` instead of `Q2 2026`) instead of after a long Run.

We chose to **share the engine's fit pipeline** (`prepareSimulationData`) rather than write a separate `computePreview()` helper because the preview and the engine must agree by construction: any divergence would let the user see one fit in the sidebar and a different one in the chart. The single function returns *both* the simulation inputs (`lambda`, `epicSizingDist`, `kMust`, ‚Ä¶) and the `preview` payload from the same in-scope epic and initiative walks, so there is only one source of truth. The cost is real ‚Äî every multi-select edit invokes the full fit, even though no Run is imminent ‚Äî but the fit is `O(rows)` over both CSVs, cheap enough on every laptop we have tested on, and the alternative (a slimmed-down preview-only path) would invite the precise drift we are guarding against. A future revision could memoise on `(histQs, targetQs, parsedInitiatives, parsedEpics)`, but the natural cache key is itself fragile (deep equality on `parsedInitiatives`) and would pay for itself only on absurdly large CSVs.

We chose to keep the preview **hide-by-default and unhide-on-first-success** ‚Äî rather than rendering a skeleton with `‚Äî` placeholders on every cell from page load ‚Äî because a half-painted preview reads as a *defect* to a first-time user ("why is Œª a dash?"), whereas a hidden surface reads as a *not-yet-configured* state that matches the user's mental model ("I haven't picked a quarter yet, so there's nothing to show"). The hide is therefore a UX affordance, not a guard. Once unhidden, the surface is *monotonic* ‚Äî it never re-hides on subsequent edits ‚Äî because mid-edit flicker is more disorienting than a stale value briefly persisting until the next successful paint. The same monotonicity applies to `#debug-details`: once shown, it stays shown for the session.

We chose to make `tryUpdatePreview` **silent on failure** ‚Äî return early on missing inputs, catch and log exceptions with the `[preview]` prefix on `prepareSimulationData` errors, no alerts, no toasts ‚Äî because the user has not yet pressed **Run** and is in *configuration mode*, not *commit mode*. An alert dialog on every CSV with a typo-in-header would noise up the file-load happy path; the Run-button click handler is where surface failure belongs (the user has committed to a Run there, and an alert is appropriate). The preview's job is to show what *is* working, not to dramatise what is not; a missing key column produces a hidden preview and a hidden debug panel, and the user discovers the failure by *noticing the surfaces did not appear* ‚Äî the same way they would notice a CSV not loading by the absence of the green check icon.

We chose the preview's **specific field set** ‚Äî `Hist. quarter`, `Initiatives (hist.)`, `Poisson Œª`, `Epic samples`, the per-size breakdown, `Target quarter`, and the three `K` rows ‚Äî to mirror the *upstream-of-the-engine* inputs and nothing else. Notably absent: `Capacity`, `Iterations`, mean effort, percentile estimates. The **Run knobs** ([ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md)) are visible in their own sidebar fields and adding them to the preview would create two displays of the same value. Mean and percentile estimates require a Run to compute, and surfacing them in the preview would either (a) require a tiny no-feedback Monte Carlo just for the preview (a second source of truth, and a much higher cost than the fit walk), or (b) display the previous Run's values, which would lie about the current input state. The compact "fit inputs only" rule is what keeps the preview honest about *when in the workflow* it speaks for.

We chose to **co-locate the static T-shirt size reference and the column-detection debug pre-block in the same sidebar group** as the live preview because all three are *trust surfaces* ‚Äî they exist to let the user verify that the simulator is modelling what they think it is modelling. The reference table answers "what does S actually mean in PM?"; the debug pre-block answers "did the parser pick the right columns?"; the live preview answers "is the fit on the data I expected?". Putting them in three different panels ‚Äî a tooltip on the size column, a settings page, a separate diagnostics view ‚Äî would force the user to scan three surfaces in three places when stress-testing a forecast. The single sidebar group is the documented "show me your work" location. The reference table is the user-facing mirror of `T_SHIRT_PARAMS` named in [ADR-0007](./0007-lognormal-effort-distribution.md); the debug pre-block is the user-facing audit surface for [ADR-0005](./0005-content-based-column-detection.md)'s content scan. Both ADRs already point readers to this feature as their trust surface.

We chose to keep the **T-shirt size reference table hand-maintained** rather than generated from `T_SHIRT_PARAMS` at runtime because the table is a *documentation surface*, not a *derived view* ‚Äî and the documentation must speak for the *synthetic* P10/P90 calibration even when the user has switched to **Empirical parameters** ([feature 0018](../../backtracked-features.md#0018)). A runtime-generated table would silently re-render to show empirical bands when the toggle is flipped, telling the user (falsely) that the reference defines the empirical mode. The intentional duplication ‚Äî a seven-row `<table>` whose rows must be kept in sync with `T_SHIRT_PARAMS` by hand ‚Äî pays for itself by removing the question of "which mode does the reference reflect?". The cost is a known bug surface (one place to forget when re-calibrating); we accept that explicitly in [ADR-0007](./0007-lognormal-effort-distribution.md).

We chose to **write the debug pre-block via `textContent` rather than `innerHTML`** because the panel renders the live `detectedCols` map, whose values are user-supplied CSV headers. Headers can contain HTML-special characters (a `<priority>` column header is not the malicious case ‚Äî it is the *plausible* case for a tool that allows non-canonical formats ‚Äî see [ADR-0005](./0005-content-based-column-detection.md)). An `innerHTML` write would parse them as markup and either drop them silently or ‚Äî for a deliberately-adversarial CSV ‚Äî open a script-injection surface. `textContent` is the one-line guard.

This decision pairs with [ADR-0005](./0005-content-based-column-detection.md) (the **Column-detection debug** panel is the trust surface for content-based detection), [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md) (the preview's `Epic samples` and per-size breakdown are the **Bootstrap pool**'s observable shape), [ADR-0007](./0007-lognormal-effort-distribution.md) (the **T-shirt size reference** is the user-facing mirror of `T_SHIRT_PARAMS`), [ADR-0008](./0008-poisson-epic-count.md) (the `Poisson Œª` row is this ADR's fit), and [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (the three `K` rows are this ADR's per-Scenario initiative counts). The implementation is `renderPreview` and `tryUpdatePreview` (`index.html:2818-2884`), the sidebar markup at `index.html:923-957`, and the inline CSS at `index.html:269-358`.

A future revision could surface the currently-reserved `epicCountMin` / `epicCountMax` / `targetInitCount` fields from the preview payload (additive ‚Äî would not re-open this ADR), add a "copy preview to clipboard" affordance for slide screenshots (additive), or generate the **T-shirt size reference** table from `T_SHIRT_PARAMS` *only after* the parameter sets are unified into a single calibrated map with no synthetic/empirical split (would re-open this ADR and [ADR-0007](./0007-lognormal-effort-distribution.md) jointly). Replacing the live preview with a post-Run-only summary, or moving the debug pre-block into a separate diagnostics page, would also re-open this ADR ‚Äî the "all three trust surfaces in one sidebar group, available before Run" choice is the load-bearing one.
