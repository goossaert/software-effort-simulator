# Three-tier risk colouring (ok / caution / warn) at 25% and 50% cuts on probability of exceedance

The org-level and per-team stats tables colour the **Probability of exceedance** cell (`P(effort > capacity)`) with one of three **Risk tier** classes: `ok` (emerald green `#10b981`) when `pExceed ≤ 0.25`, `caution` (orange `#f97316`) when `0.25 < pExceed ≤ 0.5`, and `warn` (red `#ef4444`) when `pExceed > 0.5`. The classifier is `pExceed > 0.5 ? 'warn' : pExceed > 0.25 ? 'caution' : 'ok'` (`renderStatsTableInto`, `index.html:2361`); the same three tiers and the same thresholds apply to every per-**Marker** `P(effort > value)` row added by [feature 0017](../../backtracked-features.md#0017). The PM percentile cells (P10 / P25 / P50 / P75 / P90 / mean) are *not* coloured — they are reference numbers, not verdicts.

We chose **three tiers** — and specifically the `ok` / `caution` / `warn` triad mapped onto the conventional green / orange / red — because three tiers is the largest set that fits a reader's *"safe / discuss / refuse"* mental model without requiring them to memorise the cut points. A two-tier scheme ("ok / not ok" at a single cut) would hide the meaningful distinction between *"the plan is right at the edge — expect a hard conversation"* (caution) and *"the plan will not fit — refuse"* (warn); operationally those are different responses from different stakeholders and collapsing them would push the user back to reading the numeric cell, which is exactly what the colouring exists to avoid. A four-or-more-tier scheme would force the reader to remember the cuts and would invite a "tier sprawl" where every team negotiates its own borderline-yellow band; we have seen that pattern in internal status dashboards and it consistently degrades into ignored colour. The green / orange / red mapping deliberately matches traffic-light convention so that no legend has to be read.

We chose the cuts at **25%** and **50%** because they correspond to the points where a reasonable stakeholder's response changes. Below 25% the plan is statistically safe — the team can commit without ceremony. Between 25% and 50% the plan is *more likely than not* to fit, but a one-in-three chance of overrun warrants the conversation about *which* initiatives would be the first to slip — the colour is the prompt for that conversation, not the verdict. Above 50% the plan is *more likely than not* to overrun, and the team should either drop scope or raise capacity before committing. The thresholds were chosen so that the **Must Only** **Scenario** is essentially never red unless the plan is genuinely over-committed (i.e. someone has dumped epics labelled `Must` that should have been labelled `Should`), and so that **Must + Should + Could** is *allowed* to be red without that being a tool error — it is the *correct* signal when the user has stuffed every "could" into the forecast, and silently re-tinting it caution would conceal the message [ADR-0010](./0010-three-scenario-moscow-forecasting.md)'s cumulative framing exists to deliver.

We chose to keep the cuts **hard-coded** rather than exposing them as UI knobs. Exposing them would turn the table into *"play with the thresholds until the colour you want appears"*, which is the opposite of what a risk verdict should be. If a particular team disagrees with the cuts in principle (e.g. a regulated team whose acceptable risk is 10%, not 25%), the right response is a future feature that lets them define their own *named* tiers per project, not a slider that lets the same user redraw the same cut for the same Run mid-decision. That feature is not on the roadmap and would re-open this ADR.

We chose to apply the **same** tier scheme to per-Marker `P(effort > value)` rows ([feature 0017](../../backtracked-features.md#0017)) as to the capacity row, rather than letting markers carry their own colour semantics. Markers are user-defined thresholds and the user-readable question on each row is the same ("how often does effort exceed *this* number?"); using different cuts for the capacity row and the marker rows would silently invite the comparison-arithmetic the colour scheme exists to short-circuit. A marker's pill colour on the chart (feature 0017's palette) is a *visual identifier* for the marker line, not a risk verdict — the row's tier class is what the reader should be steering by.

This decision pairs with [ADR-0012](./0012-percentile-summary-and-probability-of-exceedance.md) (which defines the **Probability of exceedance** the tiers grade), [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (whose cumulative three-Scenario framing relies on the warn tier being able to fire on Must + Should + Could without being misread as a tool error), and feature [0017](../../backtracked-features.md#0017) (whose per-Marker rows reuse the tier scheme). The implementation is the one-line classifier in `renderStatsTableInto` (`index.html:2361`) and the three CSS rules `.ok` / `.caution` / `.warn` (`index.html:496-498`).

A future revision could parameterise the cuts per project (a configuration file, not a UI slider) or could introduce a fourth tier for safety-critical contexts. Either would re-open this ADR. Changing the palette to be colour-blind safer (e.g. adding shape or pattern indicators alongside the colour) is *additive* and would not require a new decision; it is a desirable improvement not currently on the roadmap.
