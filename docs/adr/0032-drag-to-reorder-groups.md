# Drag-to-reorder Groups, with the user-set order applied immediately to the chart legend, the Summary Statistics columns, and every cached per-Group array

The **Groups tab** ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)) gains drag-and-drop reordering of its rows. Each Group `<tr>` carries `draggable="true"` plus an inline `⋮⋮` **drag handle** nested in the first cell (alongside the existing Name `<input>`) so the documented six-cell row contract from [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) is preserved unchanged — the handle is a visual affordance, not a seventh column. The five HTML5 drag events (`dragstart`/`dragend`/`dragover`/`dragleave`/`drop`) are wired inline on each row via the same `onevent="…"` attribute style every other Groups-tab control uses; a Y-midpoint test on the row under the mouse selects an **above** or **below** insertion edge and paints an inset `#4f46e5` box-shadow on the prospective drop target. On `drop`, a single helper `reorderGroup(fromIdx, beforeIdx)` (`index.html:3560-3578`) splices `groupsStore` into the new order and *applies the same index permutation in lockstep* to every cached per-Group array — `lastTeamData[*].kPerGroup` and `lastRenderState[*].results.results` — then re-renders the Groups tab and calls `rerenderContext(ctxKey)` for *every* `ctxKey` in `lastRenderState`. The net effect: the Organization Level chart legend and the Summary Statistics columns repaint in the new left-to-right order immediately, no **Run Simulation** press required, and every Team Level chart already on screen repaints in lockstep. No **Iteration** is re-sampled; no **Monte Carlo** scenario is re-run; the operation is pure DOM-and-permutation. The dragged group can be dropped on any other group row (above or below); dropping on itself, or on the boundary that would map back to the source slot, is a no-op (the drop indicator is hidden in those positions for visual feedback). The trailing `+ New group` row carries no `data-group-idx` and is intentionally not draggable.

We chose to **apply the reorder immediately as a pure re-paint** — explicitly departing from the **commit-on-Run** discipline that [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) and [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) establish for every other Groups-tab edit — rather than (a) defer the visual update to the next **Run** like a Name or Color edit does, (b) trigger a full **Run** automatically on drop, or (c) repaint only on the next `tab-switch` to `tab-org`, because reordering Groups is the only kind of Groups-tab edit that *provably cannot change the simulation result*. Renaming a Group changes the legend label and the stats-column header; recolouring a Group changes the histogram dataset's `backgroundColor` and the stats-table column tint; toggling a chip member changes `K` (and therefore the **Scenario**'s distribution); flipping the `isProjection` radio changes which Group drives the **Quick projection Monte Carlo** ([ADR-0020](./0020-team-projections-cross-quarter-view.md)). All four of those edits can move the chart's shape, and so commit-on-Run is the right contract — the user finishes their edits, presses Run, and reads a coherent result. Reorder cannot. It permutes the dataset draw order in the chart and the column order in the stats table; every histogram, every percentile, every `P(effort > capacity)` value is byte-identical before and after. Forcing the user to press Run for what is mechanically a pure paint would be friction earning nothing — the Run is slow (10,000+ iterations × `|groups|` **Scenarios** × per-team simulations) and would re-seed the PRNG, so the chart's bin heights would shift by Monte Carlo noise even though the *answer* would not. Triggering an automatic Run on drop would have the same cost and the same noise. Repainting only on tab-switch would mean a user dragging on the Groups tab sees no visual confirmation that the reorder took effect; the action would feel laggy and the user would not know whether to drag again. The immediate re-paint is the only option that matches what the action actually is: a visual permutation. We pay for this exception with a one-sentence carve-out in the Groups-tab vocabulary entry of [CONTEXT.md](../../CONTEXT.md) — *"edits commit-on-Run, except reorder, which re-paints immediately"* — and the carve-out is sound because the bright line is mechanical, not stylistic: an edit is in the commit-on-Run camp if and only if it can change a **Scenario**'s distribution.

We chose to **propagate the same index permutation in lockstep to every cached per-Group array** — `lastTeamData[*].kPerGroup` and `lastRenderState[*].results.results` — rather than (a) reorder only `groupsStore` and rely on a name-match reconciliation at render time, (b) recompute the per-team `kPerGroup` arrays from `editedInitiatives` on every reorder, or (c) re-run `runSimulation` once per cached context to rebuild a coherent results array, because every consumer of those caches indexes them positionally by Group index and assumes the invariant *"index `i` in `groupsStore` is index `i` in every per-Group array of the same Run"*. The chart's dataset list is built `groupResults.map((r, idx) => …)` ([ADR-0011](./0011-overlapping-histograms-shared-bins.md)); the stats table emits one `<th>` per `groupResults[i]` and one `<td>` per `groupResults[i].stats`; the **Effort projection band** reads `groupsStore.find(g => g.isProjection)` ([ADR-0020](./0020-team-projections-cross-quarter-view.md)) but the per-quarter count chips read `kPerGroup[gi]` against `groupsStore[gi].color` — every read is positional. A name-match reconciliation would have broken on duplicate Group names (which [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) explicitly permits under its lenient-validation rule) — two Groups both named `Must` are indistinguishable by name, and the reconciliation would have to fall back to either an ordering heuristic or a synthetic per-Group id ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) deliberately keeps Groups identifierless and positional). Recomputing `kPerGroup` from `editedInitiatives` on every reorder is correct but redundant — the bucketing did not change, only the order of the buckets did — and it would re-walk every initiative on every drag drop, an O(rows × groups) sweep where an O(groups) splice suffices. Re-running `runSimulation` once per cached context is by far the worst option: it re-pays the entire **Monte Carlo** cost of the Run for what was a paint operation, and the PRNG re-seed would change the bin heights as discussed above. The lockstep-permutation choice keeps the cached arrays *positionally* coherent with `groupsStore` at all times; every existing consumer reads them the same way it always has, with no awareness that a reorder happened; and the work is two array splices per cached array, dominated by the chart re-render itself.

We chose to **re-paint every cached `lastRenderState` context immediately** — `org` plus every `team-{idx}` whose chart was rendered by the most recent Run — rather than (a) repaint only the org-level context (the one the user explicitly named), (b) repaint only the currently-visible tab, or (c) defer per-team repaints until the user switches to the Team Level tab, because the **Team Level tab**'s charts ([ADR-0019](./0019-per-team-independent-simulations.md)) inherit the *same* per-Group dataset and per-Group stats-column shape as the org chart and would otherwise be silently stale until the user pressed Run. Repainting only `org` would have produced a tab where switching from `tab-org` to `tab-teams` reveals charts in the *old* legend order while the Groups tab and the Org chart are in the new order — three internally-consistent surfaces and one inconsistent one. Repainting only the visible tab would have shifted the inconsistency to whichever tab the user happened to be on at drag time. Deferring per-team repaints until the user navigates to the Team Level tab would have required a "dirty flag per context" mechanism and a re-render gate on tab-switch — a new state-machine for a problem the immediate re-paint already solves. The immediate per-context repaint is cheap because `rerenderContext(ctxKey)` (`index.html:3247-3261`) is pure DOM painting: it calls `renderChartOnCanvas` (which destroys and rebuilds the Chart.js instance from the already-computed `sorted`, `stats`, and `hist` payloads) and `renderStatsTableInto` (which rebuilds two `innerHTML` strings). No **Iteration** is sampled; the work scales with `|groups| × |teams|`, which is bounded by the soft cap of ~5 Groups ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)) and the team count of the loaded CSV.

We chose to **deliberately not re-paint the Team Projections tab** ([ADR-0020](./0020-team-projections-cross-quarter-view.md)) — accepting that its stacked-bar `Initiatives by Quarter` count chart and its **Initiative matrix** per-quarter count chips retain their pre-reorder legend order until the next **Run** — rather than (a) cache `projData` so the projection charts can be repainted in lockstep, (b) trigger a partial recompute that rebuilds `projData` and re-renders the projection charts only, or (c) destroy the projection-tab DOM and surface a "stale, press Run to refresh" placeholder, because `projData` is not cached anywhere today and adding a cache introduces a new module-scoped variable with its own staleness contract (when is it valid? when is it invalidated by other edits? does it survive a CSV reload?). The Team Projections tab is the simulator's heaviest tab — `|teams| × |quarters| × 2` Chart.js instances under wide CSVs — and `buildTeamProjections` (`index.html:2042-2152`) runs a per-(team, quarter) **Quick projection Monte Carlo** inline; rebuilding it on every drop would be a noticeable pause for negligible gain on the dominant action ("drag, look at the org chart, drag again"). Surfacing a stale-state placeholder would introduce a new UI mode and a new mental model for a problem that disappears the moment the user presses Run. The **commit-on-Run** carve-out for *reorder specifically* therefore has a sub-carve-out: reorder re-paints the org and team-level cached results immediately, and the Team Projections tab catches up on the next Run — same contract as every Groups-tab edit that *does* change a Scenario's distribution. This is documented as an accepted limitation rather than as an exception: a future revision that caches `projData` would close the gap without re-opening this ADR (see future-revisions below).

We chose to **place the drag handle inside cell 0 alongside the existing Name input**, wrapped in an inline `display: flex` `<div class="group-name-cell">`, rather than (a) introduce a seventh column dedicated to the handle, (b) make the entire row draggable with no visible handle at all, or (c) overlay the handle as an absolute-positioned element on the row's left margin, because the six-column row layout — `[Name | Color | Members | Projection | Duplicate | Delete]` — is a load-bearing contract from [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md), enforced both as documentation prose and as a strict-equality test assertion in `tests/acceptance/phase-2-groups-tab.test.js` (AT-5: `expect(tds).toHaveLength(6)`). A seventh column would have re-opened the column contract in [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) for what is fundamentally a UI affordance, not a new piece of data — the handle has no `groupsStore` field behind it. A no-handle, drag-from-anywhere model would have left the affordance entirely to the cursor (`cursor: grab` on the row), which is a recognised pattern but invisible to first-time users until they happen to hover; the explicit `⋮⋮` glyph is a one-character cost for substantially better discoverability. An absolute-positioned overlay handle would have introduced a layout dependency on the row's geometry (it needs to follow row height changes from chip-strip wrapping, which a flexbox-nested handle gets for free). The flex-wrapped handle preserves [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)'s contract verbatim — the `<input type="text">` is still in `tds[0]`, AT-5 passes — while giving the user an unambiguous visual cue. The `cursor: grab` / `cursor: grabbing` pair on the handle reinforces the affordance per the W3C drag-and-drop UA convention.

We chose to make the **entire `<tr>` draggable** — `draggable="true"` on the row, not on the handle — rather than (a) make only the handle draggable and have it programmatically initiate a row drag, (b) gate `draggable="true"` via a `mousedown` handler on the handle that toggles the attribute on / off the row, or (c) implement drag without the HTML5 API entirely (pointer events + manual ghost element), because the browser's default behaviour for *children of a draggable element* already gives the right interaction model: clicking inside an `<input>` initiates text selection, not a drag; clicking a `<button>` fires the click handler, not a drag; clicking the bare row area or the handle initiates the drag. Modelling the handle as the only draggable surface and programmatically driving a row drag would have required either a `data-drag-target` lookup at `dragstart` to find the ancestor row (extra code, extra fragility against future row-markup changes) or a manual drag-ghost via `setDragImage` (more code than the entire current implementation). Toggling `draggable` on `mousedown` would have introduced timing windows where a click and a drag both fire in unexpected order across browsers. Skipping the HTML5 API entirely would have meant re-implementing the drag-image rendering, the auto-scroll-near-viewport-edge behaviour, and the cross-element handover semantics — all of which the browser provides for free when the row is `draggable="true"`. The cost of "whole row draggable" is that a user trying to drag-select text inside the Name input might occasionally find the row starts dragging instead; in practice the browsers we target (modern Chromium, Firefox, Safari) all defer to text selection when the user mousedowns inside the `<input>`, and the verification tests confirm the Name input is still in cell 0 and still receives `change` events as before.

We chose **Y-midpoint above/below insertion semantics** — the prospective drop edge flips at the vertical midpoint of the hovered row, and `beforeIdx = targetIdx` for an above-drop or `targetIdx + 1` for a below-drop — rather than (a) drop-on-row-means-displace (the dragged row takes the target row's slot; the target shifts down or up), (b) explicit drop-into-gap zones rendered as thin spacer elements between rows, or (c) treat the entire row as a single drop target with a fixed "always above" convention, because Y-midpoint gives the user precise placement with no extra DOM and no precision tax. Drop-on-row-displace introduces an asymmetry the user has to learn: "if I drop on row 3, does row 3 become row 4, or does my dragged row become row 4?" — the answer is one or the other and the wrong default is frustrating. Explicit drop-gap zones would require N+1 spacer elements (one above row 0, one between every adjacent pair, one below row N-1), each draggable-over, each with its own hover state — workable, but five times the DOM and event-binding cost for a result that is no more precise than the midpoint test. A fixed "always above" convention would make placing a row *after* the last row impossible without a separate "drop here at the end" target — which is exactly what drop-gap zones would have to add anyway. The midpoint test is one line of arithmetic (`(e.clientY - rect.top) < rect.height / 2`), the visual feedback is an inset box-shadow on the existing row (one CSS class toggle, no new DOM), and the placement reads naturally to the user: drop on the top half, land there; drop on the bottom half, land just below. Edge cases — dropping on the source row itself, or dropping on the boundary that maps back to the source slot — are detected by `beforeIdx === _groupDragFromIdx || beforeIdx === _groupDragFromIdx + 1` and suppress the marker; the drop, if it lands, no-ops via the same equality check in `reorderGroup`.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the drag-and-drop handlers, the `reorderGroup` helper, the `_spliceMove` / `_applyGroupReorderToCaches` permutation primitives, and the CSS for the handle and the drop indicators all live inline in `index.html`; no library dependency was introduced), [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the chart's dataset draw order *is* `groupsStore` order, and this ADR is what gives the user explicit control over that order), [ADR-0018](./0018-tab-based-results-layout.md) (the drag UI lives on the fifth `tab-panel`, `#tab-groups`; reordering does not change the post-Run "reset to org" rule because no Run is involved), [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (the **commit-on-Run** discipline this ADR carves an exception out of, on the bright-line rule "an edit re-paints immediately if and only if it cannot change a Scenario's distribution"), [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) (the Group data model being reordered, the six-column row contract this ADR explicitly preserves, the lenient-validation rule that allows duplicate names — which makes positional indexing rather than name-matching the only safe propagation strategy, and the future-revision line that anticipated this very change as "additive, would not re-open this ADR"), [ADR-0030](./0030-json-persistence-for-groups.md) (`groups.json` already serialises Groups as an ordered JSON array, so the user-set order survives the Save / Load round-trip with no schema change and no `schemaVersion` bump), and [ADR-0031](./0031-vitest-jsdom-test-harness.md) (the JSDOM harness used to add verification tests in `tests/verification/drag-reorder-groups.test.js` that exercise `reorderGroup` directly — JSDOM does not implement HTML5 drag-and-drop events, so the mouse-driven path is verified manually in a real browser, not by automation). The implementation introduces: a `group-drag-handle` glyph and `group-name-cell` flex wrapper plus drop-indicator classes (`groups-row-dragging`, `groups-row-drop-above`, `groups-row-drop-below`) in the existing Groups-tab CSS block (`index.html:568-575`); five inline drag-event handlers on each row in `renderGroupsTab` (`index.html:3394-3406`); the module-scoped `_groupDragFromIdx` source-row tracker, the `onGroupRowDragStart` / `onGroupRowDragEnd` / `onGroupRowDragOver` / `onGroupRowDragLeave` / `onGroupRowDrop` event handlers, the `_clearGroupRowDropMarkers` helper, and the `reorderGroup` / `_spliceMove` / `_applyGroupReorderToCaches` permutation primitives (`index.html:3489-3596`). No engine code (`runSimulation`, `runScenario`, `computeStats`, `buildHistogram`, `bucketRowsByGroups`, `buildTeamProjections`) is touched.

A future revision could **cache `projData` so the Team Projections tab repaints in lockstep with the org and team-level tabs on reorder** — additive, would close the documented limitation without re-opening this ADR. A future revision could **add keyboard-based reordering** (e.g., `Alt+↑` / `Alt+↓` on a focused row to move it one slot, `Home` / `End` to send it to the boundary) — additive, would not re-open this ADR; the keyboard path would call the same `reorderGroup(fromIdx, beforeIdx)` helper. A future revision could **rebind reorder to commit-on-Run** so the visual order only updates at Run time — would re-open this ADR because the "immediate re-paint" rule is load-bearing. A future revision could **promote the drag handle to a seventh column** for a more conventional table layout — would re-open this ADR jointly with [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) because the six-column row contract is shared and load-bearing. A future revision could **persist the reordered list to `localStorage`** so a page reload preserves the user's order without a `groups.json` save — would re-open [ADR-0002](./0002-client-side-only.md) jointly with this ADR. A future revision could **introduce a virtualised Groups table** that lazy-renders rows above the soft cap — additive but would re-open the "draggable `<tr>`" assumption because virtualised rows are mounted and unmounted dynamically; the drag source could disappear mid-drag. A future revision could **trigger a full Run on reorder** to refresh the histogram's PRNG-driven Monte Carlo noise alongside the legend order — would re-open this ADR because the "no re-simulation, no PRNG re-seed" rule is load-bearing.
