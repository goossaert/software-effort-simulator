# Custom seeded PRNG (Xoshiro128**) over Math.random()

The simulator uses an in-tree implementation of the Xoshiro128** generator (period 2¹²⁸ − 1, passes PractRand and most BigCrush tests; see `Xoshiro128ss`, `index.html:1157`) as the sole source of randomness for every sampler: Box-Muller normals, lognormal effort, Poisson epic count, and bootstrap size pick. The generator is re-seeded at the start of each **Run** with `Date.now()` mixed via SplitMix32 to produce four state words; `Math.random()` is not used anywhere in the modelling code.

We chose a custom PRNG over `Math.random()` for three reasons. First, *statistical quality matters* in Monte Carlo: a Run draws roughly `iterations × K × λ × 2` uniforms (one per Poisson step, plus two per Box-Muller call), which at the default 10,000 iterations and typical K, λ puts us well into the millions of draws per Run. `Math.random()`'s quality is implementation-defined and has historically been weak in some browsers; Xoshiro128** has documented properties we can stand behind. Second, *the state is inspectable*: we can log the seed, snapshot the four state words, and (when we want to) reproduce a Run exactly by re-seeding. `Math.random()`'s state is opaque. Third, *we control the surface*: every sampler in the codebase calls the same `rng.nextFloat()`/`rng.nextInt()` API, which means a future change of generator only touches `Xoshiro128ss` itself.

We chose Xoshiro128** specifically (over Xoshiro256**, PCG, or splittable) because it operates on 32-bit words, which avoids `BigInt` and the performance cliff that comes with it on the JS engines we target. The 128-bit state is more than enough for our draw volume — at 2¹²⁸ − 1 we will not exhaust the period under any realistic Run — and the multiplier-rotate structure produces good output without expensive operations. The cost is that we now carry ~50 lines of generator code; we accept that because the alternative is either weaker randomness or a much heavier library dependency, both of which conflict with [ADR-0001](./0001-single-file-html-app.md).

The Box-Muller spare must be reset whenever the generator is re-seeded — otherwise a cached `_bmSpare` from the previous Run leaks one stale normal into the new Run's first sample. `resetBoxMuller()` is called from `runSimulation` immediately after the re-seed (`index.html:2088-2089`); this invariant is load-bearing and is recorded in the plan for [feature 0003](../plans/0003-monte-carlo-simulation-engine.md) as a forbidden shortcut.

We do not currently expose the seed in the UI, so Runs are not user-reproducible. We could surface "Run with seed X" as a future debug aid without changing this ADR; the machinery to do so is already in place.
