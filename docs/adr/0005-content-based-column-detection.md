# Content-based column detection over header-name matching

CSV column detectors identify the Initiative key, MoSCoW priority, team, name, and Epic→Initiative link columns by *scanning the values* in each column for a known pattern (a Jira-key regex, or a MoSCoW keyword regex), not by looking up headers by exact name. Header-name lookups remain as a **Detection fallback** for the case where no column clears the **Detection threshold** (currently `> 0.5` for Initiative keys and MoSCoW, `> 0.4` for the Epic→Initiative link), and cover both the **Sensible format** (`jira_key`, `moscow`, `building_block`, `teams`) and the legacy **Quirky format** (`teams`, `emoji`).

We made this choice because the historical export tooling that originally fed the simulator labels columns by their visual role, not their semantic content: the column named `teams` carries Jira keys, the column named `emoji` carries MoSCoW priority. A header-only approach would force every user to rename headers before upload, which (a) is annoying enough to push people back to spreadsheets, and (b) does not generalise across Linear, Asana, and various internal exports that drift on header conventions. Scanning values lets the simulator work with whatever export the user can produce, while the header-name fallback gives the **Sensible format** an unambiguous direct path.

The cost is real but bounded: content scans are O(rows × columns) per detector, which is fast enough at the scales we care about (single-quarter exports are typically < 1000 rows); and a column whose values accidentally look like Jira keys — a UUID column, or a numeric ID column with unfortunate formatting — could be misclassified. We accept that: it is rarer than the legacy-format case it solves, and the user can always rename headers to the **Sensible format** as an escape hatch. Misclassifications are also surfaced by the column-detection debug panel (`#debug-pre`), which shows the detected header names on every load.

This decision pairs with [ADR-0003](./0003-csv-input-format.md): because we accept any CSV the user can produce, we must also accept the column-naming conventions those tools impose.
