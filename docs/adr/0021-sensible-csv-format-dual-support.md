# Direct header-name matching for the Sensible format alongside content-scan detection

The **Column detectors** that already implement the **Content scan** strategy from [ADR-0005](./0005-content-based-column-detection.md) gained, in this feature, a *direct header-name match* path for the **Sensible format** headers (`jira_key`, `building_block`, `moscow`, `teams`) — applied at a *detector-specific* position relative to the content scan: *before* the scan for name and team (where there is no content-detectable pattern), and *after* the scan as the **Detection fallback** for the Initiative key and MoSCoW columns (where the content scan is the canonical path and a direct header match is only useful when the scan finds nothing). The legacy **Quirky format** detectors continue to work without change: the **Quirky format** headers (`teams` for Jira keys, `emoji` for MoSCoW priority) are still detected by content, and the positional fallbacks for name (header before the Initiative key column) and team (header before the MoSCoW column, distinct from the Initiative key column) remain in place. Both formats round-trip through the same `loadInitiativesCSV` (`index.html:1503-1516`) and produce identical `detectedCols` semantics; downstream consumers (`prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`, `renderInitiativesTable`) read `detectedCols[xxxCol]` and never see which format produced the header name.

We chose to **support both formats simultaneously**, rather than migrate to **Sensible format**-only and ask users to rename headers on legacy exports, because the **Quirky format** is the *only* output the legacy internal tooling produces and re-exporting through a rename step is exactly the friction that pushes users back to spreadsheets — the same rationale ADR-0005 documented for content scanning in the first place. Migration would mean either (a) breaking every old CSV the user has on disk, or (b) shipping a one-shot converter that the user has to remember to run, or (c) hard-coding a format toggle in the UI that the user has to set before upload. Each of these reintroduces the configuration step the simulator deliberately avoids. Keeping both formats live means the user can upload whatever export they happen to have today and a renamed-to-sensible export tomorrow, and both work without intervention. The cost is the **Detection fallback** lists — `'jira_key' → 'teams'` for the Initiative key, `'moscow' → 'emoji'` for MoSCoW — which a future maintainer must keep in sync with the format definitions, and the slightly more complex per-detector logic (header check, content scan, positional fallback) compared to a pure content-scan or pure header-match approach.

We chose to **prefer direct header-name matches over positional inference for the name and team columns**, by putting the `building_block` and `teams` checks *at the top* of `detectNameCol` and `detectTeamCol` respectively, because the **Sensible format** is the recommended layout going forward and the legacy positional inference (the header immediately before the Initiative key for the name, the header immediately before the MoSCoW column for the team) is a *heuristic against column order* that fails silently when the user reorders columns in their spreadsheet. A user who renames their headers to match the **Sensible format** does so because they want the simulator to "do the right thing without guessing", and the positional fallback's failure mode is invisible — wrong name and team columns are read, the `Initiatives count` in the **Data preview** looks fine, but the **Team Level tab** sections appear under unexpected team names and nobody notices. The direct header match short-circuits the positional inference for any CSV that names its columns the recommended way, eliminating an entire class of silent misdetection. The trade-off: a CSV that happens to have a `building_block` *header* whose values are not the initiative name is now misclassified — but the **Sensible format** is conventionally enforced for the headers it claims to use, and the **Column-detection debug** panel surfaces the choice so the user can spot the error.

We chose the **opposite ordering for the Initiative key and MoSCoW detectors** — content scan first, header-name match only as **Detection fallback** — because both of those columns *are* content-detectable (Jira-key regex `/^[A-Z][A-Z0-9_]+-\d+$/` and MoSCoW keyword regex `/must|should|could|won.t/i`) and the content scan is the canonical path that handles both formats uniformly. A direct `jira_key` header match before the scan would be redundant for the **Sensible format** (the content scan picks the same column) and *wrong* for the **Quirky format** with a misleadingly-named `jira_key` column carrying empty or free-text values. Leaving the header-name match as the fallback preserves the original ADR-0005 invariant — the content scan is the source of truth, the header-name match catches only the empty-CSV and zero-match-ratio degenerate cases — while still giving the **Sensible format** an unambiguous landing if a CSV's content happens to confuse the scan. The two orderings are *deliberately different*; do not unify them into a single pipeline.

We chose to **switch the zero-rows defaults to the Sensible-format header names** (`detectInitKeyCol` returns `'jira_key'` for empty input, `detectMoscowCol` returns `'moscow'`, `detectNameCol` returns `'building_block'`, `detectTeamCol` returns `'teams'`) — rather than to the **Quirky format** defaults — because the zero-rows path is, in practice, only hit by header-only test CSVs that the user is intentionally uploading to probe the simulator. The user constructing such a probe is almost certainly using the recommended layout, so the more useful default is the **Sensible format** name. The behavior is unobservable in normal usage (the **Data preview** is gated on non-empty rows), and the choice does *not* affect any **Quirky format** load because those CSVs always have data rows that the content scan picks up.

We chose to **keep the two CSV formats as an emergent property of the column-detector outputs** — not as a `format: 'sensible' | 'quirky'` enum surfaced anywhere in the code — because the simulator works one column at a time, not one CSV at a time. A real upload can be hybrid: a CSV exported through a rename script that hits `jira_key` and `moscow` but leaves the team column as `teams` (which is *also* the **Sensible format** header for team, but in a **Quirky format** context held Jira keys instead). The format-as-enum approach would force a discrete choice on the union of detected columns and then re-derive each column's header from that choice; the per-column approach lets every column be independently right. The price is that there is no single source of truth for "what format is this CSV" — but the simulator never asks that question; only its readers ask "what header carries the Initiative key" and that question is answered by `detectedCols.initKeyCol` directly. The **Column-detection debug** panel surfaces the per-column outcome, so the user can read the format from the headers selected, not from a flag.

This decision pairs with [ADR-0003](./0003-csv-input-format.md) (CSV is the input format and we accept whatever convention the user's tooling produces), [ADR-0005](./0005-content-based-column-detection.md) (the **Content scan** is the canonical detection strategy; this ADR refines its **Detection fallback** for two detectors and inverts the order for two others), and the existing **Sensible format** / **Quirky format** glossary entries in [CONTEXT.md](../../CONTEXT.md). The implementation is the four detector functions `detectInitKeyCol` (`index.html:1357-1369`), `detectMoscowCol` (`index.html:1376-1388`), `detectNameCol` (`index.html:1396-1408`), and `detectTeamCol` (`index.html:1416-1430`), plus the per-detector fallback lists they encode.

A future revision could **collapse the **Quirky format** support entirely** — once every internal tooling export produces the **Sensible format** headers — by removing the positional fallback from `detectNameCol`/`detectTeamCol` and the `'teams'`/`'emoji'` legacy fallback from `detectInitKeyCol`/`detectMoscowCol`. That would re-open this ADR jointly with [ADR-0005](./0005-content-based-column-detection.md) because the "two formats live together" invariant is the load-bearing decision here. A future revision could **add a third format** (e.g. an Asana or Linear export with its own header conventions) — additive at the detector level, would not re-open this ADR unless the new format's headers collide with an existing fallback. A future revision could **surface a per-detector override** in the **Column-detection debug** panel ("I know best — read column X as the Initiative key") — additive, would not re-open this ADR. Adding more **Sensible format** headers (e.g. a `quarter` direct match in a new detector) is a per-detector edit and does not re-open this ADR; *removing* `building_block` or `teams` from the direct-match list, however, would re-open this ADR because the "name/team prefers Sensible format over positional inference" rule is load-bearing.
