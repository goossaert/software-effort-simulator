# Editable Initiatives tab with `editedInitiatives` as the simulation source of truth

The simulator surfaces a fourth result **Tab** — **Initiatives** (`#tab-initiatives`, `index.html:1027-1029`) — that exposes every row of the loaded **Initiatives CSV** as a wide editable table, with two non-editable identity columns (the detected **Initiative key** and the detected name column), two free-form numeric columns (`added_value_impact`, `added_value_cost_saving`), and every other column rendered as a dropdown whose options are the unique observed values from the *original* parsed CSV (`getUniqueColumnValues(parsedInitiatives)`, `index.html:3124-3134`). Two module-scoped arrays carry the initiative data: `parsedInitiatives` (`index.html:1496`) — the immutable result of `parseCSV`, used *only* for dropdown-option enumeration and for the within-tab table render's column-order seed — and `editedInitiatives` (`index.html:1497`) — a per-row shallow-clone (`parsedInitiatives.map(r => ({ ...r }))`, `index.html:1505`) that *every downstream consumer* of initiative data reads from: `prepareSimulationData` (`index.html:1709`, `1766`), `prepareTeamSimulationData` (`index.html:1806`, `1825`), `buildTeamProjections` (`index.html:1922`, `1935`), and the projection-axis `extractQuarters` call (`index.html:3356`). Inline `onchange` / `oninput` handlers on the dropdowns and numeric inputs write straight into `editedInitiatives[rowIdx][col] = this.value` (`index.html:3170-3171`, `3176`), so an edit takes effect on the *next* press of **Run Simulation** with no re-render of the Initiatives tab itself. The toolbar carries a single `↓ Export CSV` button (`index.html:3154`) that serialises `editedInitiatives` via `Papa.unparse` and downloads `initiatives-edited.csv` (`exportInitiativesCSV`, `index.html:3191-3201`); two columns (`jira_link`, `emoji`) are *hidden in the table view* but still included in the export (`hiddenCols`, `index.html:3146`).

We chose to **keep two parallel arrays — `parsedInitiatives` (immutable) and `editedInitiatives` (mutable)** — rather than (a) editing `parsedInitiatives` in place, (b) holding the edits in a separate `edits: { [rowIdx]: { [col]: value } }` overlay applied on read, or (c) re-parsing the original CSV text on every Run, because the dropdown's *option set* must remain stable across user edits. The dropdown for each column shows the unique observed values from the loaded CSV — if the user edits a `team` cell from `Platform` to `Risk`, the next render of the dropdown should still offer `Platform` (so the user can revert), not silently lose it because no row carries that value anymore. Editing in place would have collapsed the option set to whatever the user had typed last; an overlay would have required every downstream consumer to thread a "resolve the overlay" helper between `parsedInitiatives` and its read, which contradicts the simulator's pattern of treating the parsed array as the input substrate; re-parsing on every Run would have either lost edits (defeating the feature) or required serialising edits back into CSV text and re-parsing them (a strictly worse path than just keeping the in-memory mutable copy). The cost we pay is duplicated row data — roughly `2 × |initiatives| × |columns|` strings in memory — which we accept because real-world CSV sizes are dozens to low thousands of rows and the strings are short.

We chose to **make `editedInitiatives` the simulation's source of truth**, with every downstream consumer (`prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`, the projection-axis `extractQuarters` call) reading from it directly, rather than (a) snapshotting `parsedInitiatives` into the simulator at Run time, (b) requiring an explicit "Apply edits" button before edits take effect, or (c) routing edits through a callback that re-runs the simulation on every keystroke. The "edits flow into the next Run" semantics matches the simulator's whole-app pattern — sidebar inputs (capacity, iterations, parameter mode, **Quarter selector** picks) all stay un-applied until the user presses **Run Simulation**, so the Initiatives tab follows the same contract: edit freely, press Run when ready. Snapshotting `parsedInitiatives` would have made the Initiatives tab decorative — the user could edit but their edits would not feed back into the model. An "Apply edits" button would have introduced a second commit gesture redundant with the existing Run button. Re-running the simulation on every keystroke would have burnt thousands of Runs during normal editing and produced flickering charts. The cost is that a user who has made edits but not yet pressed Run will see a Stats table reflecting the *previous* edit state (or none), with no visual cue that the table is out-of-date; we accept that because every modelling-config knob has the same property (capacity / iterations / parameter mode also lag until the next Run) and a uniform "edits pending" indicator is a future revision called out below.

We chose to **render dropdowns populated from `parsedInitiatives`'s unique observed values for most columns**, rather than (a) a free-form text input on every cell, (b) a fixed allow-list per column (e.g. `Must`/`Should`/`Could`/`Won't` for the MoSCoW column), or (c) only allowing edits on a curated subset of columns, because the dropdown communicates *the universe of valid choices* — and that universe is "whatever values exist in this CSV", not a generic vocabulary the simulator could hard-code. The MoSCoW column happens to use a fixed vocabulary, but the `teams` column varies per organisation; the `key_result` column varies per planning cycle; the `quarter` column varies per dataset. Hard-coding allow-lists per column would have required either column detection (which the simulator does for *which* column carries MoSCoW, not for what MoSCoW values look like, see [ADR-0021](./0021-sensible-csv-format-dual-support.md)) or a maintenance burden of keeping vocabularies up to date. A free-form input would have lost the type-safety of "you can only pick a team that exists in your data" — a typo turning `Platform` into `Platfomr` would have silently split the team across two **Projection sections**. Restricting edits to a curated subset would have left the most common edit case (correcting a misclassified MoSCoW, moving an initiative to a different quarter) unavailable. The dropdown's "include the current cell value even if absent from the unique set" branch (`allOpts.includes(current) ? opts : (current ? [current, ...opts] : opts)`, `index.html:3175`) handles the edge case where a row's cell holds a value no other row uses — that value is preserved as a selectable option for that row only.

We chose to **carve out `added_value_impact` and `added_value_cost_saving` as free-form numeric `<input type="number">` cells** (the `numericCols` set, `index.html:3148`), rather than rendering them as dropdowns or treating them like every other column, because those two fields are *quantitative* — they are monetary impact estimates that the user adjusts continuously, not categorical labels the user picks from a vocabulary. A dropdown over the unique observed numeric values would have made it impossible to enter a new value not already present in the CSV (the most common edit case on a numeric field); a text input would have allowed non-numeric garbage. `<input type="number" step="any">` is the minimal commitment — the browser enforces numeric input, `step="any"` allows decimals, and the value still lands in `editedInitiatives` as a *string* (the same shape as a freshly-parsed CSV cell), so downstream consumers do not need a type-discriminator. We considered making the numeric carve-out data-driven (e.g. "any column whose values are all numeric") and rejected it at landing because the simulator currently does not consume those two fields anywhere — they exist in the CSV and the export but are not read by `prepareSimulationData` or any rendering surface — and a data-driven heuristic would create surprise the first time a numeric `quarter`-like column appears in someone's CSV. Hard-coding the two field names is acceptable because they are documented contract with the team's CSV schema.

We chose to **keep `jira_key` and the detected name column non-editable** (`nonEditable = new Set([detectedCols.initKeyCol, detectedCols.nameCol].filter(Boolean))`, `index.html:3144`), rather than letting the user edit them as text inputs or making them dropdowns of observed values, because the **Initiative key** is the *primary key* threading **Initiatives** → **Epics** (every Epic links to its parent via the initiative key) and the name is human-readable identity. Letting the user edit the initiative key would silently break the join: the **Bootstrap pool** and the per-initiative epic counts depend on epics matching `parsedEpics[i]._initiative_key === init[detectedCols.initKeyCol]`, and editing the initiative side without editing the epics side would orphan every linked epic. Letting the user edit the name would not break joins but would invite confusion ("I renamed it but the chart still shows the old name in a tooltip somewhere"). The non-editable cell class (`.init-readonly-cell`, `index.html:547`) communicates the constraint visually — plain text rather than a control. Allowing edits to the team column, the MoSCoW column, the quarter column, and the **Key Result** column is safe because none of those participate in cross-table joins; the team and quarter columns *partition* the Initiative set (changing them moves an initiative between **Projection sections** or between **Target quarter** / **Historical quarter** scope) but they do not break references.

We chose to **hide `jira_link` and `emoji` from the table view** (`hiddenCols = new Set(['jira_link', 'emoji'])`, `index.html:3146`) **while keeping them in the CSV export**, rather than rendering them as columns or omitting them from the export, because both are *legacy artefacts* of the **Quirky format** ([ADR-0021](./0021-sensible-csv-format-dual-support.md)) — `jira_link` is a URL that adds horizontal scroll for no editing affordance, and `emoji` historically carried the MoSCoW priority before the **Column detector** family was rich enough to ignore it ([feature 0002](../../backtracked-features.md#0002)). Showing them as editable columns would have wasted table real estate and tempted the user to edit a column the simulator already ignores. Dropping them from the export would have broken round-trip fidelity — a user who exports, opens externally, and re-uploads the file would lose two columns of metadata for no gain. Keeping them in the export means the user's CSV stays shape-compatible with the original; the export is "your CSV with your edits applied", not "a simulator-internal projection of your CSV". We considered allowing the user to toggle which columns are visible (a column-chooser) and rejected it as scope creep — the two hidden columns are the only ones the simulator deliberately suppresses, and the rule is short enough to encode as a literal set.

We chose to **render the Initiatives tab once per Run, in the same render pass as the other three tabs**, rather than lazy-rendering on tab activation, because the same render-once-per-Run pattern is what [ADR-0018](./0018-tab-based-results-layout.md) established for the other tabs and the cost of pre-rendering is small (one table render, no Monte Carlo). Lazy-rendering on tab click would have introduced a perceptible delay on first tab visit and would have required the tab-switch handler to know about each tab's render function — coupling the tab framework to the content. The render call sits inside the run-button handler (`renderInitiativesTable()`, `index.html:3362`), one line below the Team Projections render, and reads from `editedInitiatives` — which means a user who edits a cell *after* the Run but *before* re-running will see the post-edit value persist in the dropdown (the edit went into `editedInitiatives`) but will see the *previous* simulation's charts and stats. A user who then re-runs gets a fresh simulation against the new edits and a re-rendered Initiatives tab that still reads from the now-further-mutated `editedInitiatives`. The Initiatives tab is therefore *self-consistent* (the table always shows the live `editedInitiatives` state) but the *other tabs* lag until Run is pressed; this matches the "edits flow into the next Run" semantics.

We chose to **export `editedInitiatives` verbatim via `Papa.unparse`** — no column reordering, no header normalisation, no schema transformation — rather than (a) emitting a fixed sensible-format schema, (b) splitting hidden and visible columns into two files, or (c) preserving the original parsed-CSV column order, because the user's CSV is the user's CSV: `Papa.unparse` reads the keys of the first row's object and uses them as the header set in iteration order, which is the same order `Papa.parse` produced from the original file. The shallow-clone-per-row at load time (`parsedInitiatives.map(r => ({ ...r }))`, `index.html:1505`) preserves that key order because the spread retains the source object's property-insertion order. The exported file is therefore round-trip-compatible — re-uploading `initiatives-edited.csv` produces the same `parsedInitiatives` shape as the original (with edits applied). The filename is hardcoded as `initiatives-edited.csv` rather than prompting the user, matching the **Marker CSV** convention ([ADR-0025](./0025-per-context-marker-system.md)). We considered post-processing the exported CSV to emit only the **Sensible format** ([ADR-0021](./0021-sensible-csv-format-dual-support.md)) header names (so a user editing a **Quirky format** file would receive a sensible-format export) and rejected it because column renaming on export would silently break the round-trip — a user re-uploading the exported file would get a different `detectedCols` map than they started with.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the tab markup, the CSS, the data model, the render function, and the export helper all live inline in `index.html`), [ADR-0002](./0002-client-side-only.md) (no edits persist across browser sessions other than via the user-driven CSV export), [ADR-0018](./0018-tab-based-results-layout.md) (the Initiatives tab is the fourth result tab, pre-rendered per **Run**, sharing the existing tab-bar / tab-panel skeleton), [ADR-0021](./0021-sensible-csv-format-dual-support.md) (the hidden `emoji` column is a **Quirky format** legacy artefact; the dropdowns and the non-editable key/name columns honour the format-agnostic detector outputs), [ADR-0022](./0022-optional-key-result-column.md) (the **Key Result** column appears as an editable dropdown when present in the CSV — the table reads `Object.keys(parsedInitiatives[0])`, not a curated column list), and [ADR-0025](./0025-per-context-marker-system.md) (the `↓ Export CSV` toolbar button follows the same `.add-marker-btn` styling as the chart-card export buttons, by intentional convention). The implementation is the tab-panel markup (`index.html:1027-1029`), the table-style block (`index.html:539-548`), the `editedInitiatives` declaration and load-time clone (`index.html:1497`, `1505`, `1619`), the migration of every downstream initiative-reader to `editedInitiatives` (`index.html:1709`, `1766`, `1806`, `1825`, `1922`, `1935`, `3356`), `getUniqueColumnValues` (`index.html:3124-3134`), `renderInitiativesTable` (`index.html:3136-3189`), `exportInitiativesCSV` (`index.html:3191-3201`), and the call site in the run-button handler (`index.html:3362`).

A future revision could **add a "Reset to original" button** that copies `parsedInitiatives` back into `editedInitiatives` — additive, would not re-open this ADR. A future revision could **surface an "edits pending — press Run to apply" indicator** in the sidebar or the other tabs' headers — additive, would not re-open this ADR. A future revision could **persist edits in `localStorage` across browser sessions** — would re-open this ADR jointly with [ADR-0002](./0002-client-side-only.md) because the "no implicit state across sessions" rule is load-bearing across both. A future revision could **let the user edit `jira_key` and the name column** (with a confirmation step explaining the orphan-epic risk) — would re-open this ADR because the "non-editable identity columns" rule is load-bearing. A future revision could **make the numeric carve-out data-driven** (any column whose unique values parse entirely as numbers) — would re-open this ADR because the "hardcoded `added_value_impact` / `added_value_cost_saving` set" is load-bearing. A future revision could **add a row-level Delete button** to remove an initiative from the Run — additive at the data-model level, would not re-open this ADR, but would need a corresponding "Add row" affordance for symmetry. A future revision could **let the user pick which columns are visible** in the table (a column-chooser dropdown) — additive, would not re-open this ADR; the export contract (all columns, including the user-hidden ones) would be preserved. A future revision could **re-render the Initiatives tab on tab activation** (lazy, not pre-rendered with the Run) — would re-open [ADR-0018](./0018-tab-based-results-layout.md) because the "render-once-per-Run" rule is shared across all four tabs. A future revision could **change the export's filename to include a timestamp or the Run's iteration count** — additive, would not re-open this ADR. A future revision could **make the dropdowns autocomplete-style** for columns with many unique values — additive, would not re-open this ADR.
