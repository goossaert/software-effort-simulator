# Tab-based results layout, with the org level as the resting tab

The simulator's results region (`#results-content`, `index.html:977-1030`) is split into four sibling **Tab panels** — `#tab-org` (the org-level histogram + stats table), `#tab-teams` (the per-team sections), `#tab-projections` (the cross-quarter forecast view), and `#tab-initiatives` (the editable initiatives table) — each gated by a button in a single `.tab-bar` (`index.html:982-987`). One panel is visible at a time; tab switching is a single delegated click handler on `#results-content` (`index.html:3275-3291`) that toggles a `.tab-btn.active` class and flips each panel's `style.display` between `flex` (for the chosen tab) and `none` (for the others). After every **Run**, the run-button handler resets the active tab to `org` and re-shows the results region (`index.html:3367-3375`); the user can then click any tab without re-Running.

We chose a **tabbed layout** — rather than a single scrolling page that stacks all four views vertically — because the four views are read at different *moments in the workflow*, not in a single top-to-bottom narrative. The org-level chart is the "is the plan plausible?" first read; the team level is the "where does the risk concentrate?" follow-up; the projections view is the "how does this plan trade against the next two quarters?" planning conversation; the initiatives table is the "let me edit this and re-Run" what-if surface. A scrolling page would require the user to scroll past the org chart on every team-drill-down and to scroll past the team sections to reach the projections — a constant tax on a flow that already has a Run latency. Tabs let each view occupy the entire results region so the chart fills the viewport, the stats table fits without competing for vertical space, and the team sections have room to render their own chart-plus-table groups without nesting scrollbars. The cost is a click per view-switch, which is cheap because all four panels are pre-rendered in the same **Run** (no per-tab lazy load).

We chose the **org level as the resting tab** — both at first paint and after every subsequent Run — because the org-level histogram is the canonical first read: the three overlapping **Scenario** distributions and the capacity line answer "what is the probability the plan fits" before the user has any reason to drill down. Resetting to `org` after each Run is intentional: the user has just changed an input (capacity, iterations, an initiative's MoSCoW), and the relevant question is "did the org-level answer move?" — not "did *this team*'s answer move?". A user mid-team-investigation who wants to stay on the team tab after a Run is the minority case; switching back to it is one click. The opposite default — preserve the last-active tab across Runs — would silently land the user on a stale tab with results from a now-superseded input set, an easy way to read the chart wrong.

We chose to **pre-render every panel inside the same Run**, rather than re-rendering on tab switch, because every panel reads from the *same* in-memory data the Run produced — `lastTeamData`, `projectionChartInstances`, the global stats payload — and re-rendering on switch would either need a cached payload (a snapshot the engine has to keep alive past Run completion) or a re-execution of the engine (multiplying the wall-clock by 4 for a four-tab switch sequence). Pre-rendering on Run makes the tab switch O(1) DOM cost and keeps the engine's life-cycle simple: the Run is the *only* compute moment; tab switches are pure paint.

We chose to **resize charts on tab visibility change** — the click handler invokes `c.resize()` on the now-visible tab's chart instances (`index.html:3285-3290`) — because Chart.js cannot compute canvas dimensions while its container is `display: none`. A chart drawn while its panel was hidden would have a zero-sized canvas; the resize on first show is what makes the chart usable. The resize is idempotent and cheap; running it on every show (even after the first one) is simpler than tracking a "first show" flag per chart, and the perf cost is negligible at the team-tab cardinalities we have measured.

We chose to **delegate the click handler at `#results-content`** — rather than attach one listener per `.tab-btn` — because the tab bar is rebuilt as static markup at page load (no dynamic tab additions) but the *click target* is small (the button's text), and a delegated listener via `e.target.closest('.tab-btn')` cleanly handles clicks that landed on a child of the button while requiring no rebind on tab markup changes. The four tabs are static today; a future fifth tab is markup-only.

We chose the **`data-tab` attribute on each button**, paired with `id="tab-<value>"` on each panel, as the binding between button and panel. The handler reads `btn.dataset.tab` and toggles `p.id === `tab-${tab}`` to pick the visible panel. The contract is "button's `data-tab` matches the panel's `id` suffix", which is observable from the markup alone without reading the JS. A `data-target` attribute that holds the full panel `id` would have been more general but is unnecessary at four tabs; the suffix convention is shorter and self-documenting.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the tab-switching code is ~15 lines of vanilla DOM, no router, no framework), [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the org tab's three-Scenario histogram is the canonical first read the resting tab surfaces), [ADR-0013](./0013-three-tier-risk-colouring.md) (every tab that renders a stats table inherits the same **Risk tier** colour rules, regardless of which tab is visible), and [ADR-0019](./0019-per-team-independent-simulations.md) (the Team Level tab's per-team independent simulations live in the second tab; this ADR explains *why* they have their own tab rather than sitting inline below the org chart). The implementation is the markup at `index.html:977-1030`, the CSS at `index.html:515-537`, and the click handler at `index.html:3275-3291`.

A future revision could persist the last-active tab in `localStorage` so a page reload lands the user on the tab they were on (additive — would not re-open this ADR but would override the "reset to org on Run" rule, which would re-open this ADR if extended to Runs). A future revision could lazy-render heavy tabs (the projections tab's 80+ chart instances under wide CSVs) only on first show — additive, would not re-open this ADR but would re-open [ADR-0019](./0019-per-team-independent-simulations.md) because the team tab's pre-render guarantee would change. Replacing the tabs with a single scrolling page would re-open this ADR.
