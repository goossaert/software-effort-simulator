# Capacity rendered as an auto-managed chart Marker (red dashed line)

The configured **Capacity** is visualised on every chart in a **Run** as a red dashed vertical line at the corresponding x-coordinate, labelled with a small pill reading `Capacity: {value} PM`. The line is rendered by [feature 0017](../../backtracked-features.md#0017)'s `markersPlugin` `afterDraw` hook (`index.html:2174-2243`) — the *same* code path that draws every user-added **Marker** — keyed off a special entry in `markerStore[contextKey]` with `isCapacity: true`, `id: 'cap-{contextKey}'`, `label: 'Capacity'`, and `color: '#ef4444'`. The entry is *auto-managed* by `ensureCapacityMarker(contextKey, capacityValue)` (`index.html:2986-2994`): called once per chart context per Run, it either creates the entry (first Run on a context) or updates its `value` in place (subsequent Runs), preserving any user edits to `label` and `color` from the marker dialog. The user cannot delete the capacity marker — feature 0017's dialog hides the delete button when `isCapacity === true`.

We chose to render the **Capacity** on the chart at all — rather than letting it live only as a number in the stats table — because the table tells the user *how much* of the distribution exceeds capacity, but only the chart tells them *where* the threshold sits relative to the distribution's *shape*. A `P(effort > capacity) = 32%` cell is informationally complete but reads as an abstract verdict; a vertical line drawn through the chart's right shoulder tells the reader at a glance whether overruns are concentrated in the heavy-tail outliers (a manageable risk profile) or in the centre mass (a structural problem). The two surfaces are complementary, not redundant: the table is the verdict, the chart is the diagnosis. A future revision could add a shaded "danger zone" to the right of the line as an additive visual cue without re-opening this ADR; replacing the line with a tabular-only view would re-open it.

We chose to render the capacity line as a **single chart Marker** sharing the unified marker store and rendering path — rather than as a bespoke `afterDraw` plugin written just for capacity — because every property a "capacity line" needs (a labelled vertical line at a numeric x-coordinate, drawn on top of the chart, hit-testable for click-to-edit, contributing a per-threshold `P(effort > value)` row to the stats table) is *exactly* the property model of [feature 0017](../../backtracked-features.md#0017)'s general Marker. The pre-0017 implementation drew the capacity line in a dedicated plugin; once the general marker system landed, keeping the capacity line as a separate code path would have produced a chart with *two* sets of dashed lines, drawn by *two* different `afterDraw` hooks, racing each other for label-collision avoidance and producing visually-distinct pill styling for what the user reads as "the same kind of thing". Folding capacity into the marker store unified the rendering, the click-to-edit hit-test, the label-pill collision avoidance, and the stats-table per-threshold row — at the cost of one boolean flag (`isCapacity`) that gates the dialog's delete button. We accept that asymmetry: the capacity marker *is* a marker for visualisation purposes, and is *not* a marker for "the user can delete it" purposes. The flag is the one-line cost of that asymmetry.

We chose to make the capacity marker **auto-managed** — `value` is overwritten by `ensureCapacityMarker` on every Run, but `label` and `color` are preserved across Runs — because the two field categories have different sources of truth. The `value` is owned by the sidebar `#capacity` input ([ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md)); a user who recolours the line to yellow from the marker dialog is *not* asserting "the capacity should be at a different PM" — they are personalising the visual. Conversely, a user who edits `#capacity` from `120` to `200` *is* asserting a different capacity, and the line must move. The auto-management makes those two sources of truth coherent: sidebar → `value`, dialog → `label`/`color`. The alternative — making `value` editable from the dialog and *not* auto-synced from the sidebar — would force the user to remember to update it in two places after every team-roster change. The simpler alternative — making `label`/`color` *also* reset every Run — would silently revert the user's personalisation on every Run, which we have seen frustrate users on internal dashboards that adopted that pattern.

We chose to keep the capacity marker **always present** on every chart context — `ensureCapacityMarker` is called once per chart, every Run, and there is no path to remove it — because the chart without a capacity line is a chart whose verdict the reader has to reconstruct mentally from the stats table. Allowing the line to be optional would either (a) hide it behind a sidebar toggle (clutter for no benefit, see [ADR-0001](./0001-single-file-html-app.md)'s "no abstractions" tone) or (b) let the user delete it from the dialog with no easy recovery path (the user would have to know to refresh the page). The hidden delete button on `isCapacity === true` markers is the prevention; the auto-management is the recovery (a Run after any state confusion re-creates the marker if it has somehow been removed, e.g. via the DevTools console).

We chose to **co-locate the capacity colour with the warn-tier colour** (`#ef4444`) by default — rather than picking a neutral indicator colour like grey or a chart-friendly accent — because red is what the reader's eye is already trained to find by [ADR-0013](./0013-three-tier-risk-colouring.md)'s warn-tier colouring on the `P(effort > capacity)` cell. Aligning the chart-line colour with the table-cell-tier colour reinforces "this is the budget; crossing it is the warn condition" without requiring a legend. The user can recolour the line from the dialog and the choice will persist; the *default* signals the relationship to the stats-table verdict.

We chose to ensure the line is **clamped to the chart edge** when the configured capacity sits outside the chart's clipped x-range (the **Global histogram range**, see [ADR-0011](./0011-overlapping-histograms-shared-bins.md)) rather than letting it disappear off-screen. The plugin computes a `fraction = clamp((value - globalMin) / range, 0, 1)` (`index.html:2188`) and draws at the clamped x. A configured capacity of `1000` PM on a chart whose right edge is at `250` PM produces a line at the chart's right edge with the pill reading `Capacity: 1000 PM` — the user still sees *that there is a capacity*, and the off-chart number on the pill is the cue to scroll up to the sidebar and re-evaluate. The alternative (drawing off-screen) would silently hide the line, leaving the reader to wonder whether the simulator forgot to render it.

This decision pairs with [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the chart whose x-range the line is drawn into), [ADR-0013](./0013-three-tier-risk-colouring.md) (which gives the line its default red), [ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md) (the source of the `value`), and feature [0017](../../backtracked-features.md#0017) (whose plugin actually draws the line and whose dialog allows label/colour edits). The implementation is `ensureCapacityMarker` (`index.html:2986-2994`) and the marker entry shape it produces; the drawing path is owned by feature 0017.

A future revision could allow the capacity to be hidden via a sidebar toggle (would re-open this ADR — the "always present" rule is load-bearing), let the user delete the capacity marker from the dialog (would re-open this ADR — the protection on `isCapacity === true` is what keeps the chart's diagnostic intact), or render a shaded danger zone instead of a line (additive — would not re-open this ADR). Reverting the capacity to a bespoke `afterDraw` plugin separate from the marker system would also re-open this ADR; the unification with the general Marker model is the load-bearing choice.
