# Capacity and iterations as user-configured per-Run sidebar inputs

The simulator exposes two **Run knobs** in the sidebar: **Capacity** (`#capacity`, a person-month budget, default `120`, no upper bound) and **Iterations** (`#iterations`, the Monte Carlo iteration count, default `10000` in the HTML placeholder and `1000000` as the empty-field fallback inside the run-button handler, clamped to `[1000, 10000000]`). Both are bare `<input type="number">` controls — no presets, no slider, no derivation from team-roster data, no auto-fit from historical actuals. They are read *exactly once* per **Run**, at the top of the run-button click handler (`index.html:3310-3312`), and passed by argument into every downstream consumer; there is no module-scoped global that stores them between Runs, so the `<input>.value` attribute is the single source of truth.

We chose to make **Capacity** a *user-typed* PM number — rather than deriving it from a team-roster CSV, headcount × quarter-weeks × utilisation, or a "team-size" preset — because the only honest answer to *"how much can we deliver?"* is the team's own commitment, not a formula. Every derivation we considered hides a value judgement: utilisation is a knob with no canonical value, "ramp-up" subtracts a different fraction in every org we have looked at, and parental-leave / holiday calendars sit in HR systems the simulator deliberately does not talk to ([ADR-0002](./0002-client-side-only.md)). Surfacing those knobs as UI would push the user toward *"play with the inputs until capacity matches the plan"* — the opposite of the tool's purpose, which is to confront a *committed* capacity with a *forecast* distribution. The committed PM number is also what the team is held accountable to externally (a roadmap conversation with leadership, a slide); the simulator should consume that same number verbatim, not a derived approximation. A future feature could *suggest* a starting capacity from a loaded team-roster CSV, but the user would still have to accept it as the input — the suggestion would never be the source of truth.

We chose to make **Iterations** a *user-tunable* precision knob — rather than fixing it at a single value or auto-fitting it to a target standard error — because the same user, on the same plan, has two distinct use modes: a *quick scan* (under a second, ~10k iterations) while iterating on the MoSCoW bucketing, and a *commit Run* (a few seconds, ~1M iterations) when the team is locking the plan and the **Probability of exceedance** at the published capacity needs to settle to two significant figures. A fixed iteration count would force one mode to be wrong: 10k is too noisy for the second mode (the right-tail `pExceed` flickers Run-to-Run); 1M is too slow for the first (a five-second wait on every MoSCoW edit breaks the exploratory rhythm). Auto-fitting to a target standard error would be statistically principled but would couple the iteration time to the input data — a plan with heavier tails would take longer for the same convergence — surprising the user whose mental model is "iterations = the time-cost dial". The numeric input lets the user move along that cost/precision curve explicitly. The default of `10000` is calibrated to the *quick-scan* mode (sub-second on a typical laptop) so first-time users get a fast first answer; the empty-field fallback of `1000000` is calibrated to the *commit-Run* mode and is the value that materialises if the user clears the field entirely (a "give me the slow precise answer" shortcut). The discrepancy between the two defaults is intentional and documented in feature [0008](../plans/0008-configurable-capacity-and-iterations.md).

We chose the iteration clamp `[1000, 10000000]` because the lower bound is where Monte Carlo noise visibly dominates the **Stats** table (`pExceed` at 100 iterations can flip class on a re-Run, see [ADR-0013](./0013-three-tier-risk-colouring.md)'s tier thresholds, which would erode trust in the tool) and the upper bound is where a single Run starts to lock the main thread for tens of seconds on a typical laptop — a UX cliff worse than the precision gain. The clamp is enforced silently (`Math.min(10000000, Math.max(1000, ...))`) rather than via a modal because a clamp dialog on every paste-from-Slack would be noise; the user can see the actually-used iteration count in the chart's bar-count sum if they care to check.

We chose to read the inputs *once per Run* — at the top of the run-button handler, before the `setTimeout` that defers the heavy work — and to plumb the values by function argument from there. This makes the **Run** boundary defined in [CONTEXT.md](../../CONTEXT.md) literal: mid-Run sidebar edits do not affect the in-flight Run, only the next press. The alternative (re-read inside `runSimulation` or inside the render functions) would let a half-Run be configured with two different capacities — the engine sampling against one, the table labelling against another — and the resulting `P(effort > {capacity} PM)` row could disagree with the chart's capacity line. The plumb-by-argument pattern is the same discipline applied to `lambda`, `epicSizingDist`, and `fixedEffort` and keeps every Run reproducible from a single snapshot of inputs.

We chose to *not* store the inputs in any module-scoped variable between Runs. The `<input>.value` attribute is the source of truth; the read-and-clamp expression is the access path. The earlier draft of this feature briefly considered a `lastCapacity` / `lastIterations` module global to support a *"replay the last Run with new markers"* path — that path exists today as `lastRenderState[contextKey]` ([feature 0017](../../backtracked-features.md#0017)) and stores the *capacity-as-used* value alongside the results payload, *not* the live sidebar value. The separation is intentional: a marker-only re-render uses the `lastRenderState` snapshot's capacity (so the marker hit-test math stays consistent with the chart that is actually drawn); a fresh Run reads the live sidebar value (so the user's edit takes effect). Conflating the two would let a sidebar edit to capacity silently take effect during a marker-edit re-render, surprising the user whose mental model is "I clicked OK on a colour change, why did my numbers move?".

This decision pairs with [ADR-0002](./0002-client-side-only.md) (capacity cannot come from a server-side roster system this app does not talk to), [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (the same capacity is compared against all three **Scenarios** — there is no per-Scenario capacity), [ADR-0012](./0012-percentile-summary-and-probability-of-exceedance.md) (the capacity number is the threshold the table's `P(effort > capacity)` row reads), and [ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md) (which decides *how* the capacity value is visualised on the chart). The implementation is the two `<input>` elements (`index.html:897-905`) and the read-and-clamp block in the run-button handler (`index.html:3310-3312`).

A future revision could add a "suggested capacity" preview computed from a loaded team-roster CSV (additive — would not re-open this ADR, but would re-open [ADR-0002](./0002-client-side-only.md) if the roster were fetched server-side), expose an "auto-fit iterations to standard error" mode (additive UI, would not re-open this ADR), or introduce per-team capacity overrides (would re-open this ADR because the "one capacity, all charts" rule would break). Replacing the bare number input with a slider or a presets dropdown would also re-open this ADR.
