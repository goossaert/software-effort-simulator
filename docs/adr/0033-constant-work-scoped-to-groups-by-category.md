# Constant work scoped to Groups by Category

> Amends [ADR-0023](./0023-constant-work-csv-deterministic-shift.md). The deterministic mechanism ADR-0023 established — closed-form lognormal-mean effort via `tshirtToPersonMonths`, added as a post-sort shift that preserves the Monte Carlo sort order, following the **Synthetic** ↔ **Empirical** parameter toggle — is **retained unchanged**. What this ADR changes is the *scope* of the shift: from one global scalar applied to every **Scenario** to a per-**Group** vector determined by **Category** membership.

We chose to **bucket constant-work effort per Group by Category, rather than apply a single `fixedEffort` scalar uniformly to every Scenario**. Under [ADR-0023](./0023-constant-work-csv-deterministic-shift.md) the constant-work sum was applied to *every* Group's distribution identically — the **Category** field on a constant-work row was read for the **Initiative matrix** badge but never influenced the engine, so a row categorised `Backend` shifted the `Frontend` Scenario by the same amount as the `Backend` one. As of [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) a **Group** is a user-defined set of Category values and is the sole determinant of which **Initiatives** enter its **Scenario**; leaving constant work outside that model made it the one body of work the user could not target. The new rule makes constant work a first-class, Category-addressable commitment: for each Group, sum the effort of constant-work rows whose Category ∈ `group.members` (compared `trim()` + case-insensitively, the same convention as Initiative bucketing and [ADR-0028](./0028-category-as-generalized-moscow.md)) and apply *that* sum as the Group's shift. One global scalar becomes a per-Group vector aligned index-for-index with `kPerGroup` — computed alongside it in the same bucketing pass (`bucketRowsByGroups` is the natural home) and threaded into `runSimulation` as a `fixedEffortPerGroup` array in place of the scalar `fixedEffort`. A Group whose members match no constant-work Category gets a shift of `0`.

We chose to **keep constant work out of the K / Poisson count**, exactly as before. Constant work is *guaranteed, deterministic* effort; the user already knows it will ship. Feeding it into the stochastic epic-count or sizing machinery would reintroduce the Monte Carlo noise the deterministic shift exists to avoid, and would double-count. So the per-Group vector is purely additive effort; `kPerGroup` continues to count Initiatives only. The phrase "add items from both initiatives and constant work for each group" resolves to: stochastic Initiatives drive the sampled distribution, deterministic constant work shifts it — the two are summed, not pooled.

We chose to **apply this scoping on all three simulation surfaces** — the org-level headline **Run**, the **Team Level tab**, and the **Team Projections tab** — rather than only the headline chart. All three already bucket by Group (`bucketRowsByGroups` at org and team scope; the **Projection group**'s membership at projection scope) and all three previously summed *all* in-scope constant work regardless of Category. Scoping only one surface would mean the same constant-work row behaves differently depending on which tab the user reads — an indefensible inconsistency. The existing scope filters compose with the new Category filter by **conjunction**: org level sums over (Category ∈ members) across all teams in the target quarters; the Team Level tab adds (team matches, case-insensitive); the **Quick projection Monte Carlo** scopes each (team, quarter) cell to the **Projection group**'s members. A direct consequence on the Team Projections tab: a cell now shows only the constant work whose Category is in the Projection group's members — constant work in other Categories no longer appears in that cell's band or in the appended soft-green **Initiative matrix** rows. This is the intended reversal of ADR-0023's "all constant work, every cell" behaviour.

We chose to **exclude constant work whose Category is in no Group, but surface the exclusion rather than drop it silently**. The reversal's sharp edge is that guaranteed effort can now fall through the cracks: a constant-work Category the user forgot to add to any Group is counted *nowhere*, whereas ADR-0023 always counted everything. Two mitigations make this safe. First, the auto-default `All` **Group** (created on first load with no **Groups JSON**) takes its members from every unique Category across *both* `editedInitiatives` **and** `editedConstantWork` (plus the **(Blank) sentinel**), so a fresh import captures all constant work by default and the drop can only happen once the user *deliberately* narrows Groups. Second, the **Data preview** surfaces the allocation: each per-Group row reports the constant-work PM folded into that Group beside its `K`, and a dedicated line reports any constant work outside all Groups (e.g. "Constant work in no group: 12 PM across 3 rows — excluded"). No **Run** gate, no blocking — pure visibility, consistent with the app's lenient-validation philosophy ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)). **(Blank)**-categorised constant work follows the **(Blank) sentinel** rule like Initiatives: it enters a Group only if that Group's members contain the sentinel.

We chose to **feed constant-work quarters into the target Quarter selector only, not the historical one**. A quarter present in the **Constant Work CSV** must be selectable as a **Target quarter** (and only constant work whose quarter ∈ the selected target set is summed, the existing exact-trimmed match). But constant work contributes nothing to fitting **Poisson λ** or the **Bootstrap pool**, so a constant-work-only quarter is a meaningless *historical* selection. The two selectors therefore diverge in source: historical stays the Initiatives + Epics union; target becomes Initiatives + Epics + constant-work quarters. Selecting a target quarter that exists *only* in constant work yields a pure-constant-work forecast — every Group's `K = 0`, each Group sitting at its deterministic shift.

This decision pairs with [ADR-0023](./0023-constant-work-csv-deterministic-shift.md) (the deterministic-shift mechanism it scopes), [ADR-0028](./0028-category-as-generalized-moscow.md) (the case-insensitive Category comparison and **(Blank) sentinel** it reuses), [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) (the user-defined Group model it extends to constant work), and [ADR-0020](./0020-team-projections-cross-quarter-view.md) (the Projection-group-scoped **Quick projection Monte Carlo** whose constant-work fold-in this now filters). The histogram's shared `globalMin` — held at `fixedEffort` under ADR-0023 because every Scenario shifted by the same amount — must now become the **minimum** per-Group shift across the Run (often `0`), since Scenarios no longer share a single floor; this is the one **Global histogram range** consequence the implementation must handle.

A future revision could **promote constant work to a stochastic-with-tight-CV mode** — would re-open this ADR jointly with ADR-0023, since the deterministic invariant is still load-bearing. A future revision could **surface the per-Group constant-work shift as a vertical chart marker** ("Backend constant work eats X PM") — additive, would not re-open this ADR. A future revision could **add a Run-time warning or gate when constant work falls outside all Groups** — additive over the Data-preview surfacing, would not re-open this ADR. A future revision could **let a Group target constant work by team or quarter independently of Category** — would re-open this ADR, because "Category membership is the sole binding between a Group and its constant work" is the load-bearing rule here.
