# User-defined Groups supersede the cumulative MoSCoW three-scenario forecasting

> **Status:** supersedes [ADR-0010](./0010-three-scenario-moscow-forecasting.md)

The hardcoded three cumulative **Scenarios** of [ADR-0010](./0010-three-scenario-moscow-forecasting.md) — `Must Only`, `Must + Should`, `Must + Should + Could` — are removed and replaced with user-defined **Groups**. A **Group** is `{ name: string, color: string, members: (string | BlankSentinel)[], isProjection: boolean }` — a flat, independent set of **Category** strings ([ADR-0028](./0028-category-as-generalized-moscow.md)) that names one **Scenario** the Run executes. A **Run** now performs `iterations × |groups|` simulations rather than `iterations × 3`; each Group's simulation samples `K = #initiatives whose Category ∈ Group.members`, sharing the per-Run **Poisson λ**, **Bootstrap pool**, **Capacity**, and PRNG seed. The hardcoded exclusion of `Won't` and `unknown` from [ADR-0010](./0010-three-scenario-moscow-forecasting.md) is removed entirely: inclusion is now *purely* a function of group membership, and an initiative whose Category is in no Group is silently excluded from every Run. Exactly one Group has `isProjection = true`; that Group drives the per-(Team, Quarter) **Quick projection Monte Carlo** ([ADR-0020](./0020-team-projections-cross-quarter-view.md)) and the **Effort projection band** in every **Projection section**. Groups are managed via a new fifth **Tab** — **Groups tab** (`#tab-groups`) — that renders one editable row per Group with columns `[Name | Color | Members | Projection | Duplicate | Delete]` and a `+ New group` row at the bottom; the Members cell is a horizontal chip strip with `×` per chip and a `+` button that opens a **MultiSelect** ([ADR-0017](./0017-multi-quarter-selectors.md)) populated from every observed category in `editedInitiatives` plus a `(Blank)` row. Defaults: on first load (no JSON yet, no user-defined groups), the simulator auto-creates exactly one Group named `All` whose members are every unique Category present in the loaded CSV including the **(Blank) sentinel** ([ADR-0028](./0028-category-as-generalized-moscow.md)), flagged `isProjection`. Edits to Groups are **commit-on-Run** — they land in the in-memory group list immediately, but no chart or stats table re-renders until the user presses **Run Simulation** ([ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md)). Validation is lenient: empty names, duplicate names, zero-member groups, and members referencing categories not present in the current CSV are all allowed (the chart legend disambiguates duplicates with a suffix; the engine silently returns `K = 0` for zero-member or no-match groups, falling back to the **Constant work** shift per [ADR-0023](./0023-constant-work-csv-deterministic-shift.md)); the `isProjection` flag is the only strictly-enforced rule (a single-select radio in the Groups tab guarantees exactly one Group carries it). The soft cap is around 5 Groups — not enforced; the chart keeps the overlapping-datasets shape from [ADR-0011](./0011-overlapping-histograms-shared-bins.md) with N user-coloured series and the stats table grows to N columns, both deteriorating in legibility above ~5–6 Groups as a self-evident signal.

We chose to **supersede the cumulative-three model entirely** rather than (a) keep three hardcoded scenarios with user-configurable bucket *labels*, (b) allow N groups but constrain them to be nested-by-construction, or (c) layer the new Group concept on top of the existing three scenarios as a fourth axis, because the cumulative-three shape was a *consequence* of MoSCoW's inherent ordering (`Must > Should > Could`), and arbitrary categories have no inherent ordering. [ADR-0010](./0010-three-scenario-moscow-forecasting.md) explicitly defends the cumulative framing as "the marginal cost of admitting the next priority tier" — that question only makes sense when categories form a priority hierarchy. A user labelling initiatives `KR1` / `KR2` / `KR3` does not want to know "the marginal cost of admitting KR2 *and* KR1 simultaneously"; they want to know "what is the effort forecast if we commit to just KR1, and separately, what is the forecast for just KR2". A user labelling initiatives `Automation` / `Project ABC` does not have a natural cumulative order at all. Keeping the three-scenario constraint and only generalising the labels would have forced the user to lie to the model about ordering. The previous ADR-0010 paragraph "a future revision could expose the bucket set as a user-configurable choice without changing this ADR's cumulative-three shape" anticipated this question — and the answer turns out to be that the cumulative-three shape itself does not survive the generalisation; the cumulative-three semantics is *recoverable* by the user manually defining three overlapping groups `{Must}`, `{Must, Should}`, `{Must, Should, Could}`, but it is no longer the engine's contract.

We chose to **model Groups as flat, independent sets of Category strings** rather than (a) nested-by-construction with an `extends: GroupId` pointer producing an implicit DAG, (b) a stack of mutually-exclusive layers producing N implicit cumulative scenarios, or (c) a tagged-tuple `{ baseGroup, additionalCategories }`, because the flat-set shape is the maximally expressive and minimally surprising option. With flat sets, the user explicitly chooses which Categories appear in which Group; overlap is allowed (and is exactly how the user reconstructs the classic cumulative scenarios); independence is allowed (and is exactly how the user expresses "KR1 only" and "KR2 only" as separate forecasts). A nested-extends model introduces a graph concept — cycles, ambiguous topological order, "did I edit the base or the child?" UI questions — for which the gain is one line of typing per Group. A cumulative-stack model re-imposes the very ordering constraint we just got rid of. A tagged-tuple model is a private API for cumulative semantics that doesn't extend to non-cumulative use cases. The cost of flat sets is that a user who *does* want cumulative scenarios must type the overlapping member lists explicitly (or via Phase-3 JSON load); we accept this because (a) the classic 3-scenario user reconstructs them once and saves to JSON, (b) every other group shape is a strict subset of what flat sets can express anyway.

We chose to **define inclusion as pure group membership**, removing the hardcoded `Won't` / `unknown` exclusion from [ADR-0010](./0010-three-scenario-moscow-forecasting.md), rather than (a) keep the hardcoded exclusion for legacy MoSCoW values, (b) add a global "always exclude" string-set distinct from groups, or (c) keep an implicit "Blank-excluded-by-default" rule, because the previous exclusion encoded an assumption — "Won't is a de-scope decision, unknown is a data-quality signal" — that is true *only when the categories happen to be MoSCoW*. In the generalised model, `Won't` is not a special string; it is just one possible Category label that the user may or may not put in a Group. The user who wants the legacy exclusion behaviour gets it by simply not adding `Won't` to any Group; the user who wants to forecast even the "won't do" scope gets it by adding `Won't` to a Group; the user who labels things `KR1` / `KR2` never had a "won't" concept to exclude in the first place. The **(Blank) sentinel** ([ADR-0028](./0028-category-as-generalized-moscow.md)) follows the same rule — it is just another addressable member-set entry, and the user decides via Group definition whether (Blank)-categorised initiatives are forecast. A global "always exclude" list would have duplicated what the Group editor already expresses; an implicit Blank-excluded rule would have re-introduced the very asymmetry we're removing.

We chose to **carry the projection-Group designation as a `isProjection: boolean` flag on the Group itself**, enforced as exactly-one via a single-select radio in the Groups tab, rather than (a) make the **Quick projection Monte Carlo** ([ADR-0020](./0020-team-projections-cross-quarter-view.md)) run against the union of all Groups, (b) use the first Group in display order, or (c) render N projection bands — one per Group — on the **Team Projections tab**, because the projection band's role on the Team Projections tab is to give the user *the* per-(Team, Quarter) effort estimate, not a per-Group breakdown. The previous behaviour (MSC-only) was a hardcoded "the most inclusive forecast is the projection" — which made sense when the three scenarios were nested. With flat independent Groups, "most inclusive" is not well-defined: union-of-all is one definition, but it would force the projection to include scenarios the user might never combine in practice (e.g., `KR1 ∪ KR2 ∪ KR3` may not be a planning scenario the user wants to project). First-in-display-order ties two UI semantics together — reordering for aesthetics would silently change the projection. N projection bands per quarter on the Team Projections tab would significantly increase visual density on a tab already showing many (quarters × teams) cells. The `isProjection` flag is explicit, visible (a radio column in the Groups tab), and stable across reorderings. Default behaviour: the single auto-created `All` Group on first load is flagged; on deletion of the projection Group, the next-available Group auto-takes the flag; if there are no Groups at all, the projection band falls back to constant-work-only (the same `cwEffort` fallback [ADR-0023](./0023-constant-work-csv-deterministic-shift.md) already specifies).

We chose to **place the Groups tab as the fifth tab, after Initiatives** — `Org Level / Team Level / Team Projections / Initiatives / Groups` — rather than (a) before Org Level as a "configure first, then view" landing, (b) before Initiatives to express "groups are the spec, initiatives populate them", or (c) embed Groups as a sub-section of the Initiatives tab, because the data-flow read order — label initiatives, then group them — matches the left-to-right tab reading order, and putting Groups last preserves the post-Run "always reset to Organization Level" rule ([ADR-0018](./0018-tab-based-results-layout.md)) without surprising the user with the active tab landing on a configuration surface after every Run. Placing Groups before Org Level would have made "press Run, look at the chart" require a tab-switch every time. Placing Groups before Initiatives would have required the user to define groups before they have any sense of what Categories exist in the loaded CSV (Category values are discoverable from the Initiatives tab's table view). Embedding Groups in the Initiatives tab would have cramped a tab already containing a wide horizontally-scrolling table with one column per CSV header. The fifth-tab placement is also additive to the existing tab framework — `<button class="tab-btn" data-tab="groups">Groups</button>` and `<div class="tab-panel" id="tab-groups">` slot in without restructuring the panel-switching logic.

We chose to **render the chart as N overlapping datasets and the stats table as N columns**, with a soft cap around 5 Groups and no hard enforcement, rather than (a) hard-limit Groups to exactly 3, (b) switch to small-multiples (one mini-chart per Group) above some threshold, or (c) require the user to pick which Groups display on the chart via per-Group `showOnChart` toggles, because the existing overlapping-datasets shape ([ADR-0011](./0011-overlapping-histograms-shared-bins.md)) and the existing N-column stats table are familiar surfaces the user already reads correctly, and they degrade gracefully — at 4–5 Groups the chart is still legible, at 6–7 it starts to look crowded, at 10 it is unreadable. The soft cap is *signal*: the user who has crammed 10 Groups onto one chart can *see* that they have crammed 10 Groups onto one chart and act accordingly (consolidate, save to JSON, define a smaller subset). A hard cap of 3 would have re-encoded [ADR-0010](./0010-three-scenario-moscow-forecasting.md)'s constraint in a different vocabulary and rejected legitimate workflows. Small-multiples would have been a substantially larger render redesign (per-Group chart canvas, per-Group axis management, per-Group **Marker store** key). A per-Group `showOnChart` toggle adds UI complexity for a problem the user can solve by deleting groups (or, in Phase 3, saving and curating multiple `groups.json` files). The default Chart.js dataset colours (red, orange, green, blue, purple, etc.) cycle for newly-created Groups; the user can override per-Group via the Color column in the Groups tab — first-class control.

We chose to **default first-load to a single auto-generated Group named `All` containing every observed Category including (Blank)**, flagged `isProjection`, rather than (a) auto-detect MoSCoW values and pre-create the classic three Groups (`Must`, `Must+Should`, `Must+Should+Could`) with red/orange/green colours, (b) start with no Groups and gate the Run button until the user creates one, or (c) auto-create one Group per unique Category, because the simplest and most predictable default is "one Group containing everything" — the user sees a single chart series on first Run, knows exactly what is being forecast, and customises from there. MoSCoW-auto-detection would have been a per-data-shape special case that legacy users get and non-legacy users don't — a split UX that the user must learn (and that gives the wrong default the moment they label a few initiatives `KR1`). Gating the Run on at least one Group would have introduced an explicit setup step that "press Run on a fresh CSV" no longer satisfies. One-Group-per-Category would have produced N chart series on first Run — visually loud and rarely what the user wants. The legacy MoSCoW user gives up the classic three-scenario default on first load — they redefine their three Groups manually once, save via Phase 3 JSON ([ADR-0030](./0030-json-persistence-for-groups.md)), and load on subsequent sessions. The cost is a one-time setup burden for legacy users; we accept it because the alternative is a data-shape-dependent default that confuses every non-legacy user.

We chose to **render the Members cell as a chip strip + MultiSelect popover** ([ADR-0017](./0017-multi-quarter-selectors.md)), with the popover showing every observed Category plus a `(Blank)` row plus a free-text input for adding categories not yet in the CSV, rather than (a) a checkbox column always-visible inside the cell, (b) a comma-separated text input, or (c) a typeahead `<input list>` like the Initiatives tab Category cell ([ADR-0028](./0028-category-as-generalized-moscow.md)), because the existing **MultiSelect** widget already exists in the codebase and gives the user a familiar pattern — chips for current selection, `×` to remove individual chips, dropdown to add more. A checkbox column always-visible inside the cell would have used a lot of vertical space per row (one row per Category in the CSV; 20+ rows easily). A comma-separated text input gives no autocomplete and no error feedback for typos. The Initiatives-tab `<input list>` typeahead is great for the single-value case (one Category per initiative) but does not generalise to picking multiple values at once. The free-text branch of the popover (allowing addition of categories not in the CSV) preserves the legitimate workflow of prepping a Group set ahead of CSV load — a user who saves a `groups.json` referencing `Must`/`Should`/`Could` should be able to define those Groups without first loading a CSV that contains those values.

We chose to **commit edits to Groups immediately to the in-memory group list and defer chart/stats re-rendering until the user presses Run Simulation**, mirroring [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md)'s Initiatives-tab discipline, rather than (a) live-re-running on every keystroke / chip toggle, (b) requiring an explicit `Apply` button on the Groups tab, or (c) auto-running on a debounce, because the simulator's whole-app pattern is "edit freely, press Run when ready" ([CONTEXT.md L93](../../CONTEXT.md)). Live re-running would burn 10,000+ iterations on every chip toggle (heavy for a vanilla JS thread; visible flicker on the chart; user sees a half-edited Group's chart state mid-edit). An explicit `Apply` button on Groups would have introduced a second commit gesture redundant with the existing Run button — and inconsistent with the Initiatives tab. Auto-debounce is a hidden delay the user does not perceive as theirs. Commit-on-Run keeps the mental model uniform across the two editable tabs: edits land in `editedInitiatives` / `editedGroups` immediately and surface in the *next* Run.

We chose **lenient validation** — empty Group names, duplicate names, zero-member Groups, and references to categories not present in the current CSV are all allowed — rather than (a) strict block-on-invalid, or (b) a mixed warn/block policy, because each "invalid" case is a legitimate intermediate state during editing or a legitimate cross-CSV workflow. An empty Group name is "I haven't finished naming this one yet" — blocking would force the user to type a placeholder. Duplicate names happen during duplicate-then-rename workflows; the chart legend appends a disambiguating suffix at render time. Zero-member Groups are valid prep states: the user creates a Group, names it, and is about to add members. References to categories not present in the current CSV are exactly how a user prepares a Group set for a *future* CSV upload (or loads a `groups.json` that was authored against a different CSV). The cost is no inline error feedback for typo'd category names — we accept this because the Group editor's dropdown lists every category that *does* exist, making typos surface visually as separate chip entries. The single strictly-enforced rule — exactly one `isProjection` — is enforced at the UI level (single-select radio), so the data model can never reach an invalid state through normal interaction.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the new tab markup, the Groups data model `groupsStore`, the Groups-tab render function, and the engine changes all live inline in `index.html`), [ADR-0002](./0002-client-side-only.md) (Group definitions are ephemeral in-memory state, persisted only via the user-driven `groups.json` save/load — [ADR-0030](./0030-json-persistence-for-groups.md)), [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the chart's shared `Global histogram range` and shared bin count now span N datasets instead of three), [ADR-0017](./0017-multi-quarter-selectors.md) (the **MultiSelect** widget is reused for the Members popover), [ADR-0018](./0018-tab-based-results-layout.md) (the Groups tab is the fifth result tab, pre-rendered per Run, sharing the existing tab skeleton), [ADR-0020](./0020-team-projections-cross-quarter-view.md) (the **Quick projection Monte Carlo** now runs against the `isProjection` Group's member-set instead of MSC), [ADR-0023](./0023-constant-work-csv-deterministic-shift.md) (a zero-member Group falls back to the `cwEffort` constant-work-only band, same fallback path), [ADR-0025](./0025-per-context-marker-system.md) (the per-context **Marker store** keys remain `'org'` and `'team-{idx}'` — Group selection is independent of marker context), [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (the commit-on-Run pattern, and the principle that an editable tab is the source of truth for the engine), [ADR-0028](./0028-category-as-generalized-moscow.md) (the **Category** strings that populate `members`), and [ADR-0030](./0030-json-persistence-for-groups.md) (the `groups.json` save/load layer on top of this model). The implementation introduces a new module-scoped `groupsStore: Group[]` array, a `renderGroupsTab()` function mirroring `renderInitiativesTable`'s shape, a rewrite of `prepareSimulationData` to emit one K value per Group (`groups.map(g => initiatives.filter(i => g.members.includes(i.category)).length)`), a rewrite of `runSimulation` to loop over `groups` instead of the three hardcoded scenarios, a rewrite of `renderStatsTableInto` and the `markersPlugin` to emit N columns / N datasets, a rewrite of `buildTeamProjections` to read `groups.find(g => g.isProjection)?.members` instead of MSC, and updates to the **Data preview** to surface a per-Group K breakdown instead of the three-scenario `K_must` / `K_must+should` / `K_must+should+could`.

A future revision could **reintroduce a cumulative-by-construction Group type** as an opt-in (a second "Group kind" alongside flat) — would re-open this ADR because the "flat, independent" invariant is load-bearing. A future revision could **allow multiple `isProjection` Groups, rendering N projection bands** — would re-open this ADR because the "exactly one projection Group" invariant is load-bearing. A future revision could **add a per-Group `showOnChart: boolean`** to declutter the chart without deleting Groups — additive, would not re-open this ADR. A future revision could **add a `description: string` field per Group** for free-text notes — additive, would not re-open this ADR. A future revision could **switch the chart layout to small-multiples above N Groups** — additive at the rendering level, would not re-open this ADR. A future revision could **enforce a hard cap on Groups** (e.g., 8) — would re-open this ADR because "soft cap, signal not enforcement" is load-bearing. A future revision could **introduce per-Category colour** alongside per-Group colour — would re-open this jointly with [ADR-0028](./0028-category-as-generalized-moscow.md). A future revision could **make Groups draggable to reorder** — additive, would not re-open this ADR; the chart-legend order and stats-column order would follow the user-set order. A future revision could **expose a "duplicate Group with `Must+Should`-style auto-name" preset** that constructs the classic three MoSCoW Groups from the current CSV in one click — additive, would not re-open this ADR.
