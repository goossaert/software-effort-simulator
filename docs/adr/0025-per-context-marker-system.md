# Per-chart-context Marker system: user-defined thresholds with paired chart line and stats-table row

The simulator surfaces a unified **Marker** model — a user-defined threshold on the effort (PM) axis — that exists in *two* paired surfaces simultaneously: a labelled dashed vertical line on the per-context Histogram chart, and an auto-inserted row in the per-context Stats table reporting `P(effort > <value>)` against every **Scenario**. Markers are stored in a module-scoped `markerStore` (`index.html:2975`) — a flat map from `contextKey` (the string `'org'` for the org-level chart, or `'team-{idx}'` for each team's chart in the **Team Level tab**) to an array of `{ id, label, value, color, isCapacity }` records. The same code path that draws every user-added marker — `markersPlugin`'s `afterDraw` hook (`index.html:2174-2243`) — also draws the auto-managed **Capacity** marker ([ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md)) because the **Capacity** *is* a marker (red dashed line, label pill, contributes a `P(effort > capacity)` cell to the stats table); the only asymmetry is the `isCapacity: true` flag, which (a) makes `ensureCapacityMarker` (`index.html:2986-2994`) overwrite `value` on every **Run** while preserving `label`/`color`, (b) excludes the capacity row from the auto-inserted stats-table row block (because the capacity row is already produced by `renderStatsTableInto`'s hardcoded `P(effort > capacity)` cell), and (c) disables the value-input field in the marker dialog and hides the delete button. Markers are edited via a modal dialog (`#marker-overlay`) opened *either* by the per-context `＋ Add marker` button (`index.html:995`, `2491`) *or* by clicking on the chart within a 10 CSS-pixel hit-target of an existing marker's vertical line (`index.html:2321-2340`). Markers are optionally persisted across browser sessions via per-context CSV export/import (`saveMarkersToCSV`, `triggerLoadMarkers`, `index.html:3204-3265`).

We chose to **store markers per chart context** — `markerStore['org']`, `markerStore['team-0']`, `markerStore['team-1']`, … — rather than as a single flat list shared across all charts, because the question each marker answers is *context-local*: "is the Platform team's effort distribution likely to exceed 80 PM?" has no meaning on the Risk team's chart, where the distribution sits in a different range. Sharing markers across contexts would either (a) draw every marker on every chart (cluttering charts with thresholds that are meaningful elsewhere), (b) require a per-marker "applies to" axis (a configuration burden for what is, conceptually, a one-line annotation), or (c) render markers only on the chart where they were created (which makes the shared store indistinguishable from the per-context store, only with a more confusing data model). Keeping the store keyed by `contextKey` makes each chart's markers a local concern: adding, editing, or deleting a marker on the org chart never touches a team chart, and the capacity marker — *also* a Marker — is naturally per-context (the org's capacity is the org-wide PM budget; a team's capacity is, in the current design, the same number echoed per-team, but the model leaves room for per-team capacity overrides without re-architecting). The contextKey naming convention (`'org'` for the org Run, `'team-{idx}'` indexed by the section's position in `lastTeamData`) is deliberately a string — not an object reference, not a numeric ID — so the same key threads cleanly through `markerStore[key]`, `lastRenderState[key]`, `rerenderContext(key)`, and the CSV filename (`markers-${contextKey}.csv`).

We chose to **pair every Marker with both a chart visual and a stats-table row** — `markersPlugin` for the chart, `renderStatsTableInto`'s `markerRows` block (`index.html:2376-2390`) for the table — because the chart line tells the user *where* the threshold sits relative to the distribution shape, and the stats-table row tells them *how much* of the distribution exceeds it. Either surface alone is informationally incomplete: a chart line with no probability cell forces the reader to estimate the exceedance fraction by eye; a stats-table row with no chart line gives the verdict but loses the diagnostic context. Auto-generating both surfaces from one data entry means the two stay in lockstep — when the user edits a marker's `value` in the dialog, `rerenderContext(contextKey)` (`index.html:3094-3108`) re-runs both the chart's `afterDraw` and the stats table's `renderStatsTableInto`, so the line and the row update together. The `P(effort > value)` is computed via `computePExceed(sorted, threshold)` (`index.html:2996-3002`) — a binary search over the same sorted `Float64Array` already produced by the engine, with no re-sampling — so the per-marker probability is the *exact* empirical exceedance over the completed **Run**'s **Iteration**s, not an approximation. The stats-table row uses the same three-tier **Risk tier** colouring ([ADR-0013](./0013-three-tier-risk-colouring.md)) as the `P(effort > capacity)` row, so the user reads "green ≤ 0.25, orange ≤ 0.5, red > 0.5" uniformly across capacity and custom markers — a single mental model.

We chose to **edit markers via a modal dialog** (`#marker-overlay`, opened by `openMarkerDialog`, `index.html:3020-3039`) — rather than via a sidebar form, inline-on-chart controls, or an "edit mode" toggle — because marker editing is a *transient* interaction (set a value, pick a colour, close) that doesn't need permanent sidebar real estate, and because the value-axis is the chart, not the sidebar. The modal carries three controls: a `<input>` for the label, a `<input type="number">` for the value, and an 80-colour palette grid. The 80-colour palette (`COLOR_PALETTE`, `index.html:2954-2972`) is a Google-Docs-style 8-row × 10-column grid spanning grays, reds, oranges, yellows, greens, teals, blues, and purples — enough to differentiate a dozen markers on the same chart without forcing the user into a colour-picker UI. The dialog is keyed by `_dialogState = { contextKey, markerId }` (`index.html:2977`): a `markerId === null` means "new marker"; a `markerId` matching an existing record means "edit". The dialog's value-input is disabled and the delete button hidden when `isCapacity === true` — the asymmetric protection that keeps the capacity marker auto-managed ([ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md)). The dialog closes on `Save`, on `Delete`, on `Cancel`, or on a backdrop click; on save or delete, `rerenderContext` rebuilds the chart and stats table in lockstep.

We chose to **make the chart itself a marker editor** via a 10-CSS-pixel hit-target click handler (`canvas.addEventListener('click', …)`, `index.html:2321-2340`), rather than requiring the user to find the marker in a list and click an edit button, because the most natural place to re-edit a marker is the place where the user is reading it. The hit-test runs per-context: for each marker in the chart's `getMarkers(contextKey)`, the click handler computes the marker's pixel x-coordinate via the same `(value - globalMin) / range` fraction the renderer uses, then opens the dialog for the marker whose line sits closest to the click *if* the distance is under 10 CSS pixels. Clicks outside the chart plot area are ignored; clicks inside but more than 10 px from every marker are silently dropped (Chart.js's own dataset click-handling is untouched). The 10-px target is a UX trade-off: smaller would make markers fiddly to hit; larger would cause adjacent markers to fight for the same hit-zone (with the closer one winning, which is correct but less predictable). Because the click hit-test reads from `markerStore` directly, the capacity line is also clickable — opening the dialog in "edit capacity" mode with the value-input disabled and the delete button hidden, which preserves the [ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md) protection.

We chose to **lay out label pills above the chart, staggered across multiple rows when they would overlap horizontally** — rather than placing them inline next to their lines, beside the chart, or in a legend — because the chart's only free vertical real estate is the strip above the plot area, the labels need to remain readable when markers cluster (e.g. `Capacity: 120 PM` next to `Stretch: 130 PM`), and a separate legend would dissociate the label from its line at the worst possible moment (when the user is actively comparing several thresholds). The `markersPlugin` runs three passes (`index.html:2185-2241`): pass 1 computes each pill's pixel x-centre and measured text width; pass 2 walks the pills left-to-right (sorted by `value`) and assigns each pill a *row* — incrementing from 0 — whose interval `[leftPx, rightPx]` does not overlap any previously-placed pill on the same row; pass 3 draws first the dashed vertical line (starting *below* the tallest label row so the line never crosses the pill of another marker) then the rounded-rectangle pill at the assigned row's vertical position. The collision-avoidance algorithm is `O(n²)` in the number of markers per chart, but `n` is small (typically `≤ 12`) so the cost is dominated by the canvas draw calls. Pills are clamped to the chart plot area so a marker at the extreme left or right does not bleed off the edge.

We chose to **persist markers via per-context CSV save/load** — `saveMarkersToCSV(contextKey)` exports `label,value,color,is_capacity` rows for one chart context, `triggerLoadMarkers(contextKey)` imports them and re-renders — rather than via `localStorage`, a server round-trip, or a "share link" mechanism, because the persistence story has to match the simulator's whole-app pattern: client-side only ([ADR-0002](./0002-client-side-only.md)), single-file ([ADR-0001](./0001-single-file-html-app.md)), no implicit state across sessions. A `localStorage` cache would silently surface yesterday's markers on today's chart even when the underlying CSVs have changed; a server round-trip would break the offline use case. A user-controlled CSV file is the simulator's existing pattern for every input (Initiatives, Epics, Constant Work) — extending it to markers means the user explicitly chooses when to save and load, can email the file to a teammate, can version-control it alongside the planning artefacts, and never finds the simulator in a state they did not put it in. The CSV roundtrip preserves `isCapacity` so a load can restore the capacity marker's colour/label personalisation; loading a CSV whose `is_capacity` row carries a different `value` from the current `#capacity` input *also* updates the sidebar input and *re-runs the simulation* (`index.html:3251-3260`) — this is the one place a marker CSV load reaches outside the marker model, and it exists because a capacity value that disagrees with the sidebar would silently desynchronise the next Run's `value` overwrite from the persisted intent. The CSV save/load is per-context (`markers-org.csv`, `markers-team-0.csv`, …) because that mirrors the storage model; users who want to bundle multiple charts' markers must export each separately.

This decision pairs with [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the chart whose `globalMin`/`globalMax` the marker x-coordinates project against), [ADR-0012](./0012-percentile-summary-and-probability-of-exceedance.md) (the `P(effort > value)` semantics every per-marker stats row reports), [ADR-0013](./0013-three-tier-risk-colouring.md) (the colour classes the per-marker rows reuse), [ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md) (the source of the capacity `value` that `ensureCapacityMarker` mirrors into the marker store), and [ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md) (the auto-managed capacity marker that lives inside this marker system as a flagged entry). The implementation is `markerStore` and its helpers (`index.html:2975-3002`), the modal dialog (`openMarkerDialog`, `handleMarkerSave`, `handleMarkerDelete`, `index.html:3020-3091`), `markersPlugin`'s `afterDraw` (`index.html:2174-2243`), the canvas click-to-edit handler (`index.html:2321-2340`), the `renderStatsTableInto` per-marker row block (`index.html:2376-2390`), `rerenderContext` (`index.html:3094-3108`), the CSS palette (`COLOR_PALETTE`, `index.html:2954-2972`), the per-context buttons in the HTML (`index.html:993-995`, `2489-2491`), and the CSV save/load helpers (`index.html:3204-3265`).

A future revision could **add a third paired surface — a hover tooltip on the chart showing the per-marker `P(effort > value)` inline** — additive, would not re-open this ADR. A future revision could **share markers across contexts** (a "global threshold" that draws on every chart) — would re-open this ADR because the "per-context store, no cross-talk" rule is load-bearing. A future revision could **persist markers in `localStorage`** as an alternative to the CSV roundtrip — would re-open this ADR because the "no implicit state across sessions" rule is load-bearing. A future revision could **expand the colour palette beyond 80** or replace the swatch grid with a free-form colour picker — additive, would not re-open this ADR. A future revision could **support per-marker hover dimming** of the surrounding distribution (so the user can see the tail being attributed to this marker) — additive, would not re-open this ADR. A future revision could **let the user delete the capacity marker** — would re-open this ADR jointly with [ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md), because the `isCapacity` protection is the load-bearing rule of both. A future revision could **collapse `markerStore` into a property on each chart context's `lastRenderState`** — additive at the data model level, would not re-open this ADR. A future revision could **render markers on the Team Projections tab's per-quarter charts** — would re-open this ADR because the `contextKey` namespace currently has no `'projection-…'` form and the projection charts have a different x-axis semantic (per-quarter band, not per-iteration distribution).
