# Empirical lognormal parameters as a sidebar-toggleable alternative to the synthetic per-size fit

The simulator carries **two** parallel per-**T-shirt size** lognormal parameter tables — the original `T_SHIRT_PARAMS` (`index.html:1229`), whose `(μ, σ)` pairs are derived from the documented `[min, max]` band of each size via the closed-form P10/P90 fit ([ADR-0007](./0007-lognormal-effort-distribution.md)), and a new `T_SHIRT_PARAMS_EMPIRICAL` (`index.html:1253`), whose `μ` is bias-corrected from Q1 2026 realised effort while `σ` is preserved unchanged. Which of the two tables is "live" at any moment is governed by a single module-scoped mutable reference, `activeParams` (`index.html:1264`), that every sampler reads through: `sampleLognormal` (`index.html:1308`), `tshirtToPersonMonths` (`index.html:1273`), and the **Recognised t-shirt size** test in the within-file epic dedup and the **Bootstrap pool** ingestion guards all dereference `activeParams[normalizeSize(label)]` rather than naming a specific table. The sidebar carries a two-option radio group `<input name="param-mode">` (`index.html:907-919`) with values `synthetic` (checked by default) and `empirical`; flipping the radio reassigns `activeParams` to the corresponding table and toggles the `.active` class on the wrapper `<label>` (`index.html:3293-3300`). The toggle is *ephemeral* — it has no persistence layer, no impact on the next **Run** beyond the table swap, and no effect on the sidebar **T-shirt size reference** panel (which always documents the synthetic bands).

We chose to **store the empirical parameters as a parallel constant table**, declared at module-load time alongside the synthetic table, rather than (a) computing them at runtime from a bundled Q1 dataset, (b) loading them from an uploaded "calibration CSV", or (c) replacing the synthetic table in place. The empirical calibration is a *modelling artefact* — `n = 36` epics from Q1 2026, with per-size sample counts as small as `n=3` for `L` — that we want pinned to a specific moment in time and reviewable in the source code, not silently re-derived on every page-load against whatever the most recent dataset happens to be. Bundling a calibration CSV would have forced either an embedded data block (defeating the inspectability goal) or a fetch (which contradicts the single-file [ADR-0001](./0001-single-file-html-app.md) and the client-side-only [ADR-0002](./0002-client-side-only.md)). Replacing the synthetic table in place would have erased the documented-band reference the **T-shirt size reference** panel displays and would have broken the reproducibility contract [ADR-0007](./0007-lognormal-effort-distribution.md) holds — that anyone reading the sidebar's `[min, max]` PM column can recompute the synthetic `(μ, σ)` by hand. Keeping both tables side-by-side, with the empirical entries carrying inline comments documenting their per-size `n` and `avg_ratio`, makes the calibration both transparent and replaceable: the next time someone re-fits empirically (a future revision called out below), they overwrite *only* the empirical table; the synthetic table stays anchored to the documented bands.

We chose a **globally-mutable `activeParams` reference** — assigned once at declaration and reassigned on every radio change — rather than (a) passing the parameter table through every sampler's argument list, (b) reading a "current mode" string and branching at each sample site, or (c) maintaining two sets of pre-computed lognormal samplers and swapping them. The Monte Carlo hot loop runs `iterations × K × E[numEpics]` samples per **Run** (up to ~10⁷ × 100 × 5 ≈ 5 × 10⁹ samples at the upper end of the iteration range), and adding a table-lookup *or* a string compare on the inside of that loop would have measurable cost. The mutable-reference approach pays the cost exactly once — on radio change, outside any `Run` — and the samplers stay one-liners (`const p = activeParams[sizeLabel]`). The risk this trades off is the classic mutable-singleton risk: any code path that reads `activeParams` mid-Run would see the table change under its feet. We accept this because (a) the radio's `change` listener does *not* trigger a `Run` and the sidebar's run button is the only entry point that calls `runSimulation`, so a Run starts and finishes against a single `activeParams` value; (b) the radio is disabled by no surrounding modal state but the **Run Simulation** button itself blocks while the spinner is up; and (c) all consumers of `activeParams` are synchronous functions that complete inside one event-loop tick. Passing the parameter table through every sampler's arguments was considered and rejected as a needless refactor — `sampleLognormal`, `tshirtToPersonMonths`, every `prepareSimulationData`-family function, and the within-file dedup tie-breaker would all have to thread the extra parameter down, and the engine's signatures (`runScenario`, `runSimulation`, `buildTeamProjections`) currently take no per-Run config object that the table would naturally live inside.

We chose **μ-shift-only calibration (preserving σ)** rather than re-fitting `(μ, σ)` jointly from the realised Q1 data, because the per-size `n` is *very* small (`L: n=3`, `M: n=8`, `XS: n=10`, `S: n=14`) and a joint MLE fit of `σ` on three to fourteen samples is dominated by sample noise — the resulting `σ` would oscillate wildly between calibration windows and would communicate confidence the data does not support. The μ-shift formula is `μ' = μ + ln(avg_ratio)`, where `avg_ratio = mean(actual / synthetic_mean)` over the per-size Q1 epics; algebraically, this is equivalent to multiplying the lognormal mean by `avg_ratio` while leaving the *shape* of the per-size distribution (and therefore its log-scale uncertainty) untouched. The shape of effort uncertainty *within* a size is a property of the size category itself, not of the realised quarter — an `XL`'s log-scale spread reflects the bounded variability of "things estimators agree are XL", not the variability of last quarter's specific XL epics — so preserving σ encodes the right epistemic commitment: we have learned something about the *centre* of each size's distribution from realised data, but not enough to revise its *spread*. The price is that the empirical mode cannot widen or narrow the tail of a size; if a future calibration window has enough data to support a σ revision, the lognormal closed-form contract from [ADR-0007](./0007-lognormal-effort-distribution.md) lets either parameter be re-derived independently.

We chose to **carry the synthetic `(μ, σ)` through for every size lacking Q1 data** — `2XS`, `XL`, `XL+` at landing time — rather than (a) excluding those sizes from the empirical mode (which would make them unsamplable when the toggle is flipped), (b) emitting a warning row in the **Data preview** ("3 sizes have no empirical calibration"), or (c) inheriting from the closest-calibrated neighbour. The "carry through" rule preserves the *every size in both tables* invariant — `Object.keys(T_SHIRT_PARAMS_EMPIRICAL)` equals `Object.keys(T_SHIRT_PARAMS)` exactly — which means every downstream guard (`activeParams[size]` truthiness checks in the within-file dedup, the bootstrap-pool ingestion, the **Constant work** effort helper) behaves identically under both modes and no code path needs to special-case the empirical toggle. Excluding sizes would have introduced a silent "this CSV is no longer fully samplable" failure mode the moment the user toggles to empirical with a `2XS`-heavy dataset; the inline comments on the synthetic-carry-through entries (`// no Q1 data; same as synthetic`) document the intent for any reader. Inheriting from a neighbouring size was rejected because it would manufacture confidence (an `XL` empirical fit derived from `L`'s `n=3` sample of realised effort is a *worse* parameter set than the synthetic-band fit, not a better one).

We chose to **leave the sidebar T-shirt size reference panel unchanged when the toggle is flipped**, rather than redrawing it with the empirical bands, because (a) the band column is the synthetic *input* to the lognormal formula — it is what defines `T_SHIRT_PARAMS` and the formula `μ = (ln min + ln max) / 2`, `σ = (ln max − ln min) / (2 · Φ⁻¹(0.9))` is the operative invariant; the empirical table's `μ` shifts do not correspond to a reversed band derivation (the shifted lognormal has a different `[P10, P90]` interval that is no longer a hand-pickable t-shirt band); (b) the panel exists for *sizing* humans on what an `XS` vs an `S` *should* mean during ticket entry, not for previewing the simulator's per-Run sample distribution; and (c) every reader of the panel needs a stable anchor — a panel that re-renders with different numbers on every radio click would read as the simulator quietly changing what `XS` "is" rather than which `μ` the simulator is using for sampling. The cost is that a user looking at the **Empirical parameters** mode cannot read the empirical `[P10, P90]` directly off the sidebar — they would have to evaluate `T_SHIRT_PARAMS_EMPIRICAL` in DevTools, or trust the **Stats** output of a **Run** as the live evidence of where the per-size effort distribution sits. We accept that cost because the panel's contract is *band-as-definition*, not *band-as-current-sampling-window*.

We chose to **make the toggle ephemeral** — no `localStorage`, no URL parameter, no `Run`-input snapshot — rather than persisting it across browser sessions or stamping it onto the **Run**'s output, because the simulator's whole-app pattern ([ADR-0002](./0002-client-side-only.md)) is "no implicit state across sessions, every input the user sees is the one they just chose." A persisted toggle would silently surface yesterday's empirical mode on today's session with a different dataset loaded, and the next Run would silently report numbers the user did not intend. The radio defaults to `synthetic` on every page-load — the documented, reproducible baseline. The cost is that a user who frequently runs the simulator in empirical mode must re-click the radio every session; we accept that because the click is one keystroke and a session boundary is the natural moment to revalidate the modelling commitment. We considered stamping the active mode onto the **Run**'s output (e.g. a `Mode: Empirical` badge in the stats table) and decided against it at landing because the **Run** does not carry any other modelling-config provenance metadata (capacity, iterations, historical-quarter selection are all sidebar-readable) and adding one for the parameter mode would create a single carved-out exception; a future revision could surface a uniform "Run config" panel — additive, called out below.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (both tables, the toggle UI, and the radio handler live inline in `index.html`), [ADR-0002](./0002-client-side-only.md) (no persistence — the toggle resets on page reload, the calibration is in source code), [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md) (the **Bootstrap pool** ingestion guards on `T_SHIRT_PARAMS[size]` truthiness and continues to work because the empirical table preserves the synthetic key set), [ADR-0007](./0007-lognormal-effort-distribution.md) (the synthetic table whose `(μ, σ)` derivation contract is preserved unchanged; the empirical mode is the alternative this ADR introduces), and [ADR-0024](./0024-2xs-t-shirt-size-extension.md) (the `2XS` row in the empirical table is the synthetic carry-through, honouring the "every size in both tables" invariant). The implementation is the `T_SHIRT_PARAMS_EMPIRICAL` constant (`index.html:1253-1261`), the `activeParams` mutable binding (`index.html:1264`), the `activeParams` reads inside `tshirtToPersonMonths` (`index.html:1273`) and `sampleLognormal` (`index.html:1308`), the sidebar radio markup (`index.html:907-919`), its CSS (`index.html:172-199`), and the radio `change` handler (`index.html:3293-3300`).

A future revision could **re-fit the empirical parameters once enough realised data is available beyond Q1** — additive, would not re-open this ADR. The empirical table is the only artefact that changes; the toggle, the storage model, and the read-through semantics are unchanged. A future revision could **fit `σ` jointly with `μ` once the per-size `n` supports it** (the current `μ`-shift-only constraint is data-volume-driven, not principled) — additive, would not re-open this ADR. A future revision could **add a third "Custom" mode** that lets the user upload a calibration CSV (an empirical fit against a window the user picks) — would re-open this ADR because the "two parallel constant tables" rule is load-bearing; a third dynamic table would need a place to live. A future revision could **stamp the active mode onto the Run's output** (a "Mode: Synthetic"/"Mode: Empirical" badge in the stats table or chart title) — additive, would not re-open this ADR. A future revision could **persist the toggle in `localStorage`** as a session-spanning preference — would re-open this ADR jointly with [ADR-0002](./0002-client-side-only.md) because the "no implicit state across sessions" rule is load-bearing across both. A future revision could **let the sidebar T-shirt size reference panel surface the empirical `[P10, P90]` per size when the toggle is set to empirical** (as a separate column rather than a replacement) — additive, would not re-open this ADR. A future revision could **introduce per-team or per-Key-Result empirical parameter sets** (the current single global table folds the org's variability) — would re-open this ADR because the `activeParams` singleton would need to become a function of context. A future revision could **emit a Data-preview line documenting which sizes have empirical calibration vs synthetic carry-through** under empirical mode — additive, would not re-open this ADR.
