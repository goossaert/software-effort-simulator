# Overlapping bar histograms with shared bins and P99.5 outlier clipping

The org-level results view renders the three **Scenarios** (**Must Only**, **Must + Should**, **Must + Should + Could**) as three coloured datasets overlaid on a single Chart.js bar chart. Every dataset is binned against the *same* edge grid — exactly 60 **Bins** spanning a shared **Global histogram range** `(globalMin, globalMax)` — and the x-axis upper bound is clipped at the maximum of the three Scenarios' P99.5 so that heavy-tail **Iterations** do not stretch the axis. `globalMin` equals the constant-work shift (`fixedEffort`; `0` when none is loaded); `globalMax` equals `max(P99.5(MS), P99.5(M), P99.5(MSC), globalMin + 1)`. All three are computed once per **Run** in `runSimulation` (`index.html:2115-2120`) and consumed unchanged by `buildHistogram` (`index.html:2072`) and `renderChartOnCanvas` (`index.html:2159`).

We chose overlapping histograms on a single chart — over small-multiples (one chart per Scenario), box-and-whisker plots, violin plots, or empirical CDFs — because the question users actually arrive with is *"how does the distribution shift when we admit the next priority tier?"*, and that shift is read more directly off bars that line up on a shared x-axis than off three side-by-side mini-charts. A box plot would compress the shape into five numbers and hide bimodality and the right-skew that the **Lognormal effort distribution** ([ADR-0007](./0007-lognormal-effort-distribution.md)) produces; a violin would force a kernel-density choice that does not fall out of the Monte Carlo discipline of [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md); a CDF would be statistically cleaner but is harder to read for the product- and engineering-lead audience the tool targets (see ADR-0006's audience note). Small multiples were rejected because users consistently mis-read scale across separate axes — and forcing equal axes across three mini-charts is equivalent in information content to the overlay, with more chart chrome.

We chose to share Bin edges across the three Scenarios — rather than letting each Scenario compute its own bin grid — because differently-binned histograms are not directly comparable: a bar of height 100 on a 1 PM bin is not the same density as a bar of height 100 on a 5 PM bin, and the reader cannot tell which is which by eye. The Global histogram range is computed once per Run and every Scenario's counts are built against the same fixed grid. The bin count of 60 is a hand-tuned compromise: enough resolution that the right tail's shape is visible, few enough that each Bin still gets a meaningful count at the default 10,000 Iterations. We chose a *fixed* count rather than a Freedman–Diaconis or Sturges rule because rule-based bin counts can change between Runs on the same plan (as the empirical IQR shifts), which would make the chart appear to *move* in ways that have nothing to do with the underlying forecast.

We chose to clip the x-axis at the maximum of the three Scenarios' P99.5 (`index.html:2116`) rather than at the true maximum because Monte Carlo lognormal tails routinely produce one or two iterations in the hundreds-of-thousands of person-months, which would compress the visible distribution into a single near-zero bar. P99.5 keeps 99.5% of every Scenario's mass on-screen and trims the long tail that is not what the user is reading the chart for. We deliberately did not expose the clip percentile as a UI knob: a slider here would push the chart toward *"play with the view until it looks reassuring"* and away from *"answer the question"*. The clipped tail is still summarised in the stats table's P90 column and is the territory custom **Markers** (feature 0017) are designed for. We chose `globalMin = fixedEffort` rather than `0` so that a Run with non-zero **Constant work** does not waste the left half of the x-axis on empty space — the constant-work shift is the new origin of the distribution, not a bar at zero.

This decision pairs with [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md) (which produces the sorted **Float64Array** distributions this view bins), [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (which defines the three-Scenario shape this chart visualises — the number `3` is load-bearing for both the legend and the colour scheme), and [ADR-0001](./0001-single-file-html-app.md) / [ADR-0002](./0002-client-side-only.md) (which constrain the renderer to Chart.js loaded from CDN, with no build step and no server-side rendering). The Chart.js dependency is the second-largest in the page after the file itself; we treat it as a "ship-with" dependency, not a swappable one.

A future revision could split the overlay into small multiples behind a user toggle, switch from histograms to a density estimate, or expose the clip percentile and bin count as power-user knobs. None of those changes would alter the three-Scenario shape, but each would re-open this ADR. Moving away from Chart.js — to D3, to a hand-rolled canvas renderer, or to a server-rendered SVG — would also re-open this ADR and would need to be weighed against ADR-0001's single-file constraint.
