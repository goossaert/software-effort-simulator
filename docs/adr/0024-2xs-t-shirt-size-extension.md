# Extending the t-shirt size set downward with `2XS` (band 0.10–0.25 PM)

The simulator's **T-shirt size** vocabulary gained a seventh size — `2XS`, band `[0.10, 0.25]` PM — appended *below* `XS` on the synthetic parameter table (`T_SHIRT_PARAMS['2XS']`, `index.html:1230`), the sidebar **T-shirt size reference** panel (`index.html:936`), and the **Data preview**'s size-distribution sort order (`renderPreview`, `index.html:2821`). The derivation follows the same `μ = (ln min + ln max) / 2`, `σ = (ln max − ln min) / (2 · Φ⁻¹(0.9))` formula every other entry uses, producing `(μ, σ) = (-1.8444, 0.3575)`. The Monte Carlo engine, the **Bootstrap pool** ingestion, the **Column detectors**, and `normalizeSize` (`index.html:1493`) were *not* touched: `normalizeSize` already returned `'2XS'` for any reasonable input casing because it is a literal `(raw || '').trim().toUpperCase()` with no allow-list filter, and every downstream guard reads `T_SHIRT_PARAMS[size]` so the new key joined the **Recognised t-shirt size** set automatically. The single user-facing error message that enumerated the valid sizes (`"recognised t_shirt_size (XS, S, M, L, XL, XL+)"`, `index.html:3332`) was updated to include `2XS` at the head of the list.

We chose to **extend the band set downward to `[0.10, 0.25]` PM** rather than (a) keep the floor at `XS = [0.25, 0.75]`, (b) widen `XS` to `[0.10, 0.75]`, or (c) introduce a generic "tiny" bucket without a documented P10/P90, because real epic-sizing practice produces a stream of sub-day or one-day commitments — config tweaks, copy changes, dependency bumps — that were being filed as `XS` and dragging the empirical mean of `XS` *down* enough to make the synthetic `XS` parameters systematically over-estimate. Splitting the bottom of the curve preserves the per-size uncertainty principle [ADR-0007](./0007-lognormal-effort-distribution.md) was designed around: a `2XS` is *qualitatively* different from an `XS` (a few hours vs a few days), and smearing them together pulled both lognormal fits away from the underlying realities. Keeping the band continuous with the existing `XS = [0.25, 0.75]` (the new `2XS` top is exactly the old `XS` bottom) means no overlap and no gap — every observable point-estimate of effort still lands inside exactly one band. Approaches (a) and (b) were considered and rejected: (a) leaves the over-estimation problem; (b) widens `XS` to a 7.5x range whose lognormal fit would have an unhelpfully large σ and would understate the right-skew the simulator depends on.

We chose to **spell the new size `2XS`, not `XXS`**, because `2XS` is the existing convention inside the org's tooling and CSV exports — calling it `XXS` would have required either a normalisation rule (`normalizeSize('XXS') === '2XS'`) or two parallel entries in `T_SHIRT_PARAMS`, both of which would have meant touching code the feature otherwise left alone. The `2XS` spelling also sorts cleanly in front of `XS` under any naive lexicographic comparison (`'2XS' < 'XL'` because `'2' < 'X'`), which is convenient for any future reader who walks `Object.keys(T_SHIRT_PARAMS)` and assumes the natural order. The price is that **Data preview**'s explicit size-order array (`['XS','S','M','L','XL','XL+']` → `['2XS','XS','S','M','L','XL','XL+']`, `index.html:2821`) had to be updated by hand because that array previously relied on string-comparison-equals-ascending order and `'2XS'` would have collated *before* `'L'` rather than at the head; we accept that update because the array is the *only* place where the size order is encoded outside of the parameter table's declaration order, and JavaScript object-literal key-order semantics (insertion order for non-numeric string keys) make the parameter table itself the canonical ordering elsewhere.

We chose to **derive `(μ, σ)` from the same closed-form `P10/P90` fit** every other size uses, rather than (a) calibrating from realised effort directly or (b) inheriting `XS`'s `σ` with a shifted `μ`, because the simulator's synthetic-vs-empirical split ([ADR-0007](./0007-lognormal-effort-distribution.md), [feature 0018](../../backtracked-features.md#0018)) is load-bearing: the *synthetic* table is the documented-band fit and is supposed to be reconstructable by anyone reading the band column of the sidebar reference panel and applying the formula. Hand-tuning `(μ, σ)` for `2XS` would have introduced a silent drift between the band column and the fitted parameters, breaking the invariant that the band is the canonical input and the parameters are the derived output. The price is a `σ = 0.3575` that is *slightly* lower than `XS`'s `0.4286` — the 0.10–0.25 PM range is narrower in log-space than the 0.25–0.75 PM range — which is consistent with the principle that smaller sizes carry less *log-scale* uncertainty even when their absolute PM band is narrow. When [feature 0018](../../backtracked-features.md#0018) later introduced the **Empirical parameters** table, it explicitly *did not* attempt to calibrate a `2XS` row (no Q1 data was available for sub-`XS` epics) and instead carried the synthetic `(μ, σ)` through unchanged — preserving the "every size has an entry in both tables" contract — see [feature 0018](../../backtracked-features.md#0018).

We chose to **leave `normalizeSize` and every `T_SHIRT_PARAMS[size]` consumer untouched**, because the feature's design intent was a *zero-engine-change* extension. `normalizeSize` (`index.html:1493`) is `(raw || '').trim().toUpperCase()` — it passes any uppercase token through, so `'2xs'`, `'2XS'`, ` 2xs `, all map to `'2XS'` without any allow-list edit. The bootstrap-pool ingestion (`prepareSimulationData`, `index.html:1761-1762`) and its team-scoped twin (`prepareTeamSimulationData`, `index.html:1874-1875`) gate every epic on `T_SHIRT_PARAMS[size]` truthiness, so a `'2XS'`-labelled epic was unreachable before the table entry was added and became automatically eligible the moment it landed. The within-file epic dedup tie-breaker that reads "the row with a **Recognised t-shirt size** wins" (`index.html:1587-1588`) also became `2XS`-aware automatically. The result is that the entire CSV → Bootstrap pool → sample → Stats pipeline accommodated the new size without a line of engine code changing.

We chose to **update the user-facing error enumeration but not introduce a self-describing list of valid sizes**, because the error message at `index.html:3332` is the *only* place that enumerates the size set in user-visible text — every other surface (the sidebar reference panel, the **Column-detection debug** panel, the Initiatives tab dropdowns) reads from `T_SHIRT_PARAMS` directly or from the parsed data. Hand-editing the one literal `(XS, S, M, L, XL, XL+)` → `(2XS, XS, S, M, L, XL, XL+)` is a single-line maintenance cost that we accept, in exchange for keeping the error string flat and grep-able. A future revision could replace the literal with `${Object.keys(T_SHIRT_PARAMS).join(', ')}` — additive, would not re-open this ADR — but the manual list was preferred at landing time because it makes the error message obvious to anyone reading the page source without first loading the parameter table.

This decision pairs with [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md) (the Bootstrap pool that grew to accept `2XS` automatically), [ADR-0007](./0007-lognormal-effort-distribution.md) (the per-size lognormal contract that the new entry honours), and [feature 0018](../../backtracked-features.md#0018) (the **Empirical parameters** alternative table whose `2XS` row is a synthetic carry-through). The implementation is the `T_SHIRT_PARAMS['2XS']` entry (`index.html:1230`), the sidebar reference row (`index.html:936`), the `renderPreview` sort order (`index.html:2821`), and the run-handler error enumeration (`index.html:3332`).

A future revision could **add an `XL++` or `2XL` band above `XL+`** by mirror-applying this ADR's reasoning to the upper end of the curve — additive, would not re-open this ADR. A future revision could **fit `2XS` empirically once enough sub-`XS` realised data is available** — additive at the [feature 0018](../../backtracked-features.md#0018) table level, would not re-open this ADR. A future revision could **collapse `2XS` and `XS` back into a single band** if the realised distributions converge — would re-open this ADR because the "per-size uncertainty" principle the split encodes is the load-bearing decision here. A future revision could **switch the size-order encoding to `Object.keys(T_SHIRT_PARAMS)`** in `renderPreview` to eliminate the hand-maintained order array — additive, would not re-open this ADR. A future revision could **derive `(μ, σ)` from a per-size empirical fit at landing time** (rather than the closed-form P10/P90 formula) — would re-open this ADR jointly with [ADR-0007](./0007-lognormal-effort-distribution.md) because the "band is the canonical input, parameters are derived" rule is load-bearing across both.
