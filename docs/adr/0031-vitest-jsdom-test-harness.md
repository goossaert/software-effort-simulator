# Vitest + JSDOM test harness as dev-time tooling

The simulator gains a **Vitest** + **JSDOM** automated test harness living entirely under `tests/` and `package.json` / `vitest.config.js` at the repository root, executed via `npm test`. Tests load `index.html` into a JSDOM window with stubbed `Chart.js` and `PapaParse` globals, evaluate the page's inline `<script>` blocks in document order against that window, then assert against the resulting DOM and lexical bindings (engine functions, module-scoped `let groupsStore`, the `BLANK` sentinel, etc.). The harness exposes a small surface — `loadSimulator()`, `read(win, name)`, `typeOf(win, name)`, `evalIn(win, code)`, `execIn(win, code)`, `csv(rows, headers)` — that future test files import from `tests/harness.js`. Acceptance tests for feature 0020 (Phase 1 engine substrate, Phase 2 Groups tab) land under `tests/acceptance/phase-1-engine.test.js` and `tests/acceptance/phase-2-groups-tab.test.js`, one `describe('AT-N: …')` block per scenario in the feature's plan. The harness is **dev-time only**: the production artefact is still the single `index.html` file loaded directly from disk with no build step — `node_modules/` is gitignored, the test scripts are not bundled into the app, and a user opening `index.html` never sees Vitest or JSDOM.

We chose to **introduce automated tests via a parse-and-evaluate JSDOM harness** rather than (a) continuing the manual-only verification convention the plan documents, (b) extracting engine logic into separate ES modules importable directly by tests, or (c) using a headless-browser runner like Playwright or Puppeteer, because the simulator now carries enough behavioral surface (six tabs, per-context Markers, editable Initiatives, Constant Work cascade, soon-to-be user-defined Groups + JSON persistence) that catching regressions by manual DevTools clicks has stopped scaling — feature 0020 alone introduces 63 acceptance scenarios across Phases 1 and 2, and the only way to confirm each scenario across future refactors is to encode them as runnable assertions. Continuing manual-only would have left the project unable to refactor the engine without bespoke per-PR regression testing; the cost compounds with every future feature. Extracting engine logic into separate ES modules would have given clean test ergonomics but violates [ADR-0001](./0001-single-file-html-app.md)'s single-file constraint at the *app* level (the page would have needed `<script type="module" src="…">` references to local files, which break when `index.html` is served from `file://` on some browsers, and which complicate the "open the file, it works" deployment story). A headless-browser runner would have given the most realistic environment but added a much heavier dependency footprint (Chromium download), slower test runs (browser launch per test), and the same JSDOM-like Chart.js / PapaParse availability constraint (those libraries load from CDN — offline test runs would need a local mirror). JSDOM + script-eval keeps the production artefact untouched, runs in milliseconds, and has a footprint of two devDependencies (`vitest`, `jsdom`).

We chose to **stub `Chart.js` and `PapaParse` rather than vendor them** because the test harness does not need real chart rendering or real CSV parsing — it needs *recordable* surrogates so assertions can target what the simulator *did* (which datasets were passed to Chart, which rows came out of `Papa.parse`) instead of what the renderer *drew* (which JSDOM cannot tell us anyway: `HTMLCanvasElement.getContext` is not implemented). A `Chart` constructor that captures `config.data.datasets` and a `Papa.parse` that splits on commas / newlines is enough to drive every engine path the tests cover. The real Chart.js / PapaParse libraries continue to load from CDN at app runtime via `<script src="…">` — the harness strips those CDN tags before passing the HTML to JSDOM, then injects the stub script as the first `<script>` in `<head>` so the page's inline scripts see the stubs as their global `Chart` and `Papa`. Vendoring the real libraries into `node_modules` would have given a more realistic test environment but at the cost of (i) JSDOM-Canvas incompatibility (Chart.js wants a 2D drawing context), (ii) larger dependency footprint, and (iii) brittleness — every Chart.js version bump would change subtle internal call paths the tests might accidentally lock down. The stub is a deliberate test seam.

We chose to **reach lexical bindings (`let` / `const` / `class`) via an inline `__readBinding` helper installed by the stub script**, rather than (a) modifying `index.html` to attach every binding to `window` for test purposes, (b) running each `<script>` block via separate `window.eval()` calls from outside, or (c) requiring the page to use `var` everywhere, because `let` / `const` / `class` declarations in real `<script>` execution land in the global lexical environment — which is shared across same-document scripts — but are *not* reflected as properties on `window`. From outside the realm, `window.groupsStore` is `undefined` even after the page's `let groupsStore = []` runs. Modifying `index.html` to expose bindings for tests would have polluted the production artefact with test scaffolding; running scripts via repeated `window.eval()` from outside creates a fresh lexical environment per call and breaks cross-script references; switching the page to `var` declarations would have changed the page's actual scoping rules and removed the `let` block-scoping guarantees the engine code relies on. Instead, the stub script installs `window.__readBinding(name)` — a function that uses *direct* `eval` (`(0, eval)(name)`) from inside the page realm. Direct eval inside a function defined at top level of a script falls through to the global lexical environment, where the page's `let` / `const` / `class` bindings live. From outside, tests call `read(win, 'groupsStore')` and receive the current value. The helper is dev-time only — tests use it, the app does not.

We chose to **place test files under `tests/` adjacent to `index.html`** with a flat `tests/acceptance/<phase>.test.js` layout, rather than (a) collocating tests with the source as `index.test.html` or similar, (b) introducing a `src/` directory for tests parallel to the production artefact, or (c) putting tests under `docs/atdd-logs/`, because the flat layout matches Vitest's defaults (no custom resolver), keeps test files out of the production artefact's directory neighbourhood, and signals at a glance which files are dev-tooling vs which files ship. Collocating would have placed `*.test.js` next to a file (`index.html`) that the user is encouraged to open directly — confusing. A parallel `src/` would have implied the production code lives there too — it doesn't; `index.html` is at the root. Putting tests under `docs/atdd-logs/` would have conflated test artefacts (the RED-gate log outputs) with test source code. The `tests/` directory cleanly separates dev tooling from production.

We chose to **enforce the RED gate via Vitest exit codes** and persist each gate's log under `docs/atdd-logs/0020-category-and-groups-phase-<N>-acceptance-red.log` with a three-line header (`command`, `run_at`, `exit_code`) followed by the captured stdout + stderr. This mirrors the `/phase-atdd` skill's contract and gives the next session (running `/phase-implement`) a concrete, auditable record of what failed and why — replacing the manual DevTools "RED gate" the plan originally documented. The log files are committed alongside the test files at the close of `/phase-atdd`; they are not regenerated during `/phase-implement` (implementation re-runs the tests but writes its own logs, which `/phase-review` consults).

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the single-file constraint applies to the *production app*, not to dev-time tooling — `node_modules/` and `tests/` exist solely to support automated verification and never ship to the user), [ADR-0002](./0002-client-side-only.md) (tests run in Node + JSDOM, not in a real browser; they do not invoke any backend, do not hit `localStorage` because the page doesn't either), [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (tests assert the commit-on-Run discipline — that `editedInitiatives` edits do not re-render charts until Run is pressed), [ADR-0028](./0028-category-as-generalized-moscow.md) and [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) (the seams these tests first exercise — `detectCategoryCol`, `normalizeCategory`, `groupsStore`, `kPerGroup`, `results: GroupResult[]`). The implementation adds `package.json`, `vitest.config.js`, `tests/harness.js`, two acceptance test files for feature 0020 Phases 1 and 2, and the `.gitignore` rules for `node_modules/`.

A future revision could **add Playwright** as a second test layer for true end-to-end coverage of CSV file picker → Run button → chart visible flow, which JSDOM cannot fully simulate (Canvas, real Chart.js rendering, real layout) — additive, would not re-open this ADR. A future revision could **vendor real Chart.js / PapaParse** for tests that need realistic CSV edge cases or chart-internal assertions — would require `canvas` npm package for Chart.js to work, larger footprint, separate trade-off. A future revision could **extract engine code into a standalone module** that both `index.html` and tests import — would re-open [ADR-0001](./0001-single-file-html-app.md) because the single-file constraint would need a documented exemption for engine code; would simplify tests but at the cost of the simulator's "open the file, it works" deployment story. A future revision could **introduce inner-loop unit tests** (separate from acceptance tests) — the harness already supports this; just add `tests/unit/*.test.js`. A future revision could **add coverage reporting** via `c8` or `vitest`'s built-in coverage — would re-open this ADR if a coverage gate is added to CI.
