# Per-team independent Monte Carlo simulations with toggleable historical scope

The **Team Level tab** runs one full Monte Carlo simulation per **Team** present in the selected **Target quarter(s)**, with each team's simulation reading a *team-scoped* count of **Initiatives** (`kMust`, `kMustShould`, `kMustShouldCould` filtered by the team column) but a *user-toggleable* source for the historical parameters (**Poisson λ** and the **Bootstrap pool**) — either team-scoped (initiatives and epics whose team matches the section) or org-wide (every team's historical data pooled). The toggle defaults to *team-scoped* when the team has more than 4 historical **Initiatives** in the selected **Historical quarter(s)**, and to *org-wide* when it has 4 or fewer; a yellow recommendation hint surfaces the org-wide default with the rationale. The implementation is `prepareTeamSimulationData` (`index.html:1802-1904`), which assembles one entry per team, and `renderTeamSection(idx, useOrg)` (`index.html:2422-2439`), which runs `runSimulation` against the chosen parameters and re-renders the team's chart and stats table on every radio toggle.

We chose to **simulate each team independently** — rather than slicing the org-level Run's output by team — because every team has its own *distribution of effort outcomes*, not just its own *expected total*. A slicing approach (run one org-level Run, attribute the iteration totals to teams by some allocation rule) would either fabricate a per-team distribution (by, e.g., apportioning every iteration's epics among teams proportionally) or report only a per-team point estimate (which throws away exactly the property — the right-tail risk — the simulator exists to surface). Running each team's own Monte Carlo means each section's `P(effort > capacity)` is a genuine per-team probability, computed from `iterations` iterations of that team's own sampling, and the section's chart shows that team's actual lognormal-sum shape — narrower for teams with fewer initiatives, broader for the larger ones. The cost is more compute: with `T` teams and `iterations` iterations per team, the Team Level tab does `T × 3 × iterations` epic samples, where the org level only did `3 × iterations`. We accept that; on the cardinalities the simulator is used at (single-digit to low-double-digit teams, 10k–1M iterations), the team Run takes seconds at most, and the alternative (slicing or per-team point estimates) silently loses the property the user came for.

We chose to make the **historical parameter source toggleable** — team-scoped or org-wide — because both modes have honest, *non-overlapping* failure modes. Team-scoped λ is the unbiased fit for *this team's* historical pace, but at small `n` (a team with 3 historical **Initiatives**) it is dominated by sampling noise; the resulting Run reports a pseudo-precise distribution that is actually a fit on three data points. Org-wide λ is the well-fit pace of an *aggregate team*, but it imports the assumption that this team's pace is org-average — which is precisely the assumption the team's stakeholders may want to challenge. Neither default is universally right; the *user's domain knowledge* is what decides. A toggle exposes the choice; an automatic policy (e.g. "always use org-wide when `n ≤ 4`") would smuggle the choice past them. The 4-initiative recommendation hint is the simulator's best guess for *when* the choice usually goes one way, surfaced as a recommendation rather than a mandate.

We chose **4 as the threshold** for the recommendation hint because it lands roughly where the team-scoped λ's standard error becomes comparable to its mean (Poisson with `n=4` has a relative standard error around 50%, beyond which the team-scoped fit is more noise than signal for typical usage). The number is not load-bearing — `5` or `3` would also defend — and the recommendation is *not* enforced. The hint is yellow, not red, and it phrases the count explicitly ("only 3 historical initiatives found for this team") so the user can immediately judge whether their domain knowledge overrides the heuristic. Cranking the threshold would tilt more teams toward org-wide and reduce the chance of a noisy fit; lowering it would respect team-scoped fits at smaller `n`. The current value is a defensible midpoint and is changeable in one line if usage data justifies it.

We chose to keep the **Target quarter** counts (`kMust`, `kMustShould`, `kMustShouldCould`) *always team-scoped*, never toggleable, because the target quarter's initiatives are *what this team has agreed to deliver*; the org-wide count would be a different number measuring a different question. A team's Run forecasts "what will it cost *this team* to deliver *this team's* in-scope **Initiatives**" — there is no honest interpretation under which the K values come from another team's commitments. The toggle is therefore a *historical-only* knob, and the UI labels it `Historical data` for that reason; users who try to interpret it as "which team's plan do I want" hit a wall fast because changing the radio leaves the K values fixed.

We chose to **share the org-level Run's `lambda` and `epicSizingDist`** as the `org` option, rather than recomputing them per-team-section, because they are *already produced* by the run-button handler's `prepareSimulationData` call and shipping them by argument into `prepareTeamSimulationData` is one line. Recomputing would either re-walk the CSVs (wasted work) or risk a second source of truth (an org λ on the team tab that differs from the org λ on the org tab if the two computations drift). The plumb-by-argument pattern is the same discipline applied to capacity and iterations ([ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md)) and keeps every Run reproducible from a single snapshot.

We chose to **re-render only the affected team section** on radio toggle — `renderTeamSection(idx, useOrg)` re-runs `runSimulation` and re-paints exactly the indexed team's chart and stats table — rather than re-rendering every team on every toggle. The other team sections are unaffected by *this* team's radio; re-rendering them would be wasted DOM work and would destroy any markers the user had added to other teams' charts. The targeted re-render also means the toggle is fast (one team's `runSimulation` plus one chart paint), which matters because the toggle is a *comparison gesture* — the user flips it to see the difference, and any latency on the flip erodes the comparison.

We chose to **cache `lastTeamData`, `lastCapacity`, and `lastIterations` in module-scoped vars** so the radio toggle can re-run a team's simulation without going through the full Run pipeline again. The cache is *not* a backup of the live sidebar values (per [ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md), the live sidebar values are the source of truth for the *next* Run); it is a snapshot of the values *as used* by the most recent Run, preserved so the toggle's re-run stays consistent with the Run that produced the team list and the markers. Reading the live sidebar capacity inside the toggle handler instead would let a sidebar edit silently take effect during a marker-or-radio re-render, surprising the user whose mental model is "I clicked a radio, why did the numbers move?".

We chose to **dedupe teams case-insensitively but preserve the first-seen casing** (`teamMap = new Map(); key = raw.toLowerCase(); if (!teamMap.has(key)) teamMap.set(key, raw);`, `index.html:1809-1815`) because real CSVs frequently have `Platform`, `platform`, and `PLATFORM` referring to the same team — a separation that would silently double-count the team in the section list. The first-seen casing wins because that is the casing the user typed first and the casing they will recognise in the section header. Lowercasing the section title would feel like a downgrade; uppercasing or title-casing would be a presumption.

This decision pairs with [ADR-0008](./0008-poisson-epic-count.md) (the team-scoped λ is fit by the same Poisson logic as the org-level λ, just over the team-filtered initiative set), [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (each team's Run produces the same three **Scenario** outputs the org-level Run does), [ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md) (each team's Run reads the same capacity and iterations as the org-level Run; there is no per-team capacity override), [ADR-0015](./0015-capacity-as-auto-managed-chart-marker.md) (every team chart gets the same auto-managed capacity marker as the org chart), and [ADR-0018](./0018-tab-based-results-layout.md) (the per-team sections live in the second tab; they are pre-rendered during the same Run as the org-level chart). The implementation is `prepareTeamSimulationData` (`index.html:1802-1904`), `renderTeamSection` and `renderTeamCharts` (`index.html:2422-2525`), and the radio-toggle delegated change handler (`index.html:2517-2524`).

A future revision could expose a third radio option ("filtered org-wide": every team *except* the current one as the historical pool, for sense-checking whether this team's pace is inflated by including itself in the org average) — additive, would not re-open this ADR. A future revision could surface a per-team capacity field — would re-open [ADR-0014](./0014-capacity-and-iterations-as-run-inputs.md) and this ADR jointly because the "one capacity, all sections" rule would break. Changing the recommendation threshold from 4 to another value is a one-line edit that does not re-open this ADR; the *existence* of the threshold (rather than an automatic policy) is the load-bearing decision. Removing the toggle and forcing one mode (either team-scoped or org-wide) for all teams would re-open this ADR.
