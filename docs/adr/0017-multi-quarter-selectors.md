# Multi-quarter selectors for Historical and Target quarters, via a custom widget

The simulator's two quarter inputs — `Historical Quarter(s) — λ source` and `Target Quarter(s)` — are custom multi-select widgets (`MultiSelect` class, `index.html:1083-1143`) backed by a chip-and-checkbox UI, populated from the union of quarters extracted from the loaded **Initiatives CSV** and **Epics CSV** (`extractQuarters`, `refreshQuarters`; `index.html:1073-1542`) and sorted chronologically by `(year, quarter)` against the canonical `Q[1-4] YYYY` shape. Each widget exposes `getSelected(): string[]` as its read API and fires a bubbling `ms-change` `CustomEvent` on every checkbox toggle and every chip-removal click; downstream consumers (`tryUpdatePreview`, `prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`) read `histMS.getSelected()` and `targetMS.getSelected()` as their canonical source of selection state. There is no underlying `<select multiple>`, no third-party combobox library, and no module-scoped cache of the current selection.

We chose **multi-quarter selection** — rather than a single-quarter text input or a single-quarter dropdown — because every team we forecast for has at least one regular use mode that needs more than one quarter on either side: fitting **Poisson λ** and the **Bootstrap pool** on two or three historical quarters (rather than one, to soften the per-quarter sample variance), or forecasting two-or-three target quarters at once (a half-year plan rather than a single sprint). A single-quarter selector would force those use modes into either a re-run loop or a synthetic "merged quarter" upstream of the simulator, both of which we tried and rejected. The merged-quarter shortcut hides which actuals fed the fit (the user cannot scan the **Data preview**'s `Hist. quarter` cell and verify); the re-run loop multiplies wall-clock time and prevents side-by-side comparison. Multi-quarter selection lets both quarters carry an *ordered set* of labels and pushes the meaning of "set" into the fit pipeline (`prepareSimulationData` filters `parsedInitiatives` by `histQs.includes(row._quarter)` and similarly for epics), where the choice is documented and testable.

We chose to **build a custom widget** rather than use the browser's native `<select multiple>` because the native control's behaviour on `change` is to keep its dropdown open and require ctrl/cmd-click to toggle individual options without losing the others — a UX failure on every touch device, on every browser whose default scroll behaviour conflicts with the option list, and on every novice user who clicks an option without holding the modifier and silently wipes out their other selections. A library combobox (Choices.js, Select2, Tom-Select) would have solved the UX but at the cost of a third dependency loaded from CDN, a ~30 KB chunk on every page load, and a coupling to the library's theme system — all of which conflict with [ADR-0001](./0001-single-file-html-app.md)'s single-file constraint. The `MultiSelect` class is ~60 lines of vanilla DOM; it is cheaper than the library it replaces and the widget it wraps. The chip UI is the affordance that makes the *multi*-ness visible: a user who has picked two quarters sees two chips, and a user who has accidentally picked five sees five chips and a clear `✕` on each.

We chose a **bubbling `ms-change` `CustomEvent`** — rather than a callback registered through the constructor, or a global event bus — because the widget is instantiated twice (`histMS`, `targetMS`) and the two consumers (`tryUpdatePreview` for both selectors, the run-button handler for both) both want the same handler. A `CustomEvent` on the wrapper element lets the event-listener call sites be one-liners (`document.getElementById('hist-ms').addEventListener('ms-change', tryUpdatePreview)`) without touching the widget's internals, and the bubbling flag lets future consumers attach a single listener higher up in the tree if the wiring needs to scale. A constructor-supplied callback would have forced the widget to know who its observer is; a global event bus would have been over-engineering for a two-consumer audience. The event carries no payload — the consumer calls `getSelected()` to read the current state — because the canonical state is the widget's internal `Set`, and re-deriving the payload from the event would have been a second source of truth ready to drift.

We chose to **derive the quarter list from the data, not from a hard-coded calendar** — `extractQuarters` walks each row's `quarter` field, deduplicates, and sorts. The alternative (a hard-coded list of `Q1 2026`, `Q2 2026`, `Q3 2026`, `Q4 2026`, `Q1 2027`, …) would require an update every quarter and would tempt the user to pick a quarter the CSV does not actually contain (producing a silent empty fit). Deriving from the data also means the widget honestly reflects what is loaded — uploading an Epics CSV whose only quarter is `Q4 2025` populates the dropdowns with `Q4 2025` and nothing else, telling the user immediately that their assumed `Q3 2026` quarter is not in this file. The cost is a paint dependency on file-load timing (the widget shows `Upload CSV to populate` placeholder until the first load), which is the correct affordance: the picker is genuinely unusable before data is present.

We chose to **sort chronologically by `(year, quarter)` against the `Q[N] YYYY` regex**, with `localeCompare` as the fallback for non-matching strings, because chronological order is what every reader of the dropdown expects when scanning a quarter list. Lexicographic sort (`Q1 2026` < `Q2 2025`) reads as a defect on every demo. Numeric sort on the year alone would mis-order quarters within a year. The regex (`^Q(\d)\s+(\d{4})$`) is intentionally narrow — it parses the format we emit and forwards anything else to `localeCompare` (which preserves stable order for malformed labels). The same sort lives in both `extractQuarters` and `refreshQuarters` because they sort lists that come from different sources; the duplication is small and the colocation makes the contract obvious.

We chose to **preserve the user's current selection across re-populates** — `refreshQuarters` filters the current selection against the new union and only falls back to defaults (`Q2 2026` for historical, `Q3 2026` for target, with `slice(0,1)` / `slice(1,2)` as a final fallback) when nothing survives. The alternative (wipe on every CSV load) would punish the user who has loaded the Initiatives CSV, picked quarters, and is about to load the Epics CSV — they would have to re-pick after every load. Preserving the selection makes the load order irrelevant to the final state. The default-fallback chain is conservative: it only kicks in when the new union contains *none* of the previously-selected labels, which is the case only when the user has loaded an entirely different dataset.

We chose to keep the **quarter set as an unordered set internally** (`this.selected = new Set(...)`) and only sort at the *chip rendering* stage (`[...this.selected].sort()`). The sort at render time is a *lexicographic* sort on the labels, not chronological — chips are visual breadcrumbs of the user's choices, and a stable lexicographic order is good enough for that affordance. The chronological sort is reserved for the dropdown list (where scanning order matters). This split is intentional: rendering order in the chip strip is a UX choice, not a semantic one, and forcing it to match the dropdown's chronological sort would couple two surfaces that have different jobs.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the widget is vanilla DOM, no library, no build step), [ADR-0003](./0003-csv-input-format.md) (the quarter list is derived from the CSV's `quarter` column, with no canonical schema beyond that column existing), [ADR-0008](./0008-poisson-epic-count.md) (the **Historical quarter** selection is what `prepareSimulationData` filters against to compute λ over in-scope **Initiatives**), [ADR-0010](./0010-three-scenario-moscow-forecasting.md) (the **Target quarter** selection bounds the set of **Initiatives** counted into `K_must`, `K_must+should`, `K_must+should+could`), and [ADR-0016](./0016-live-sidebar-preview.md) (the **Data preview** re-paints on every `ms-change` via the same wiring). The implementation is `extractQuarters` (`index.html:1073-1081`), the `MultiSelect` class (`index.html:1083-1143`), `refreshQuarters` (`index.html:1523-1542`), and the two `ms-change` listeners (`index.html:3271-3272`); the markup is at `index.html:875-895` and the CSS at `index.html:123-152`.

A future revision could add a "select all" shortcut, surface the selected quarters' aggregate row-count next to the trigger (additive — would not re-open this ADR), or support keyboard navigation through the option list (additive, would not re-open this ADR). Replacing the custom widget with a library combobox would re-open both this ADR and [ADR-0001](./0001-single-file-html-app.md). Re-introducing a single-quarter input alongside the multi-select (as a "simple mode") would re-open this ADR because the two-source-of-truth question would become live again. Auto-selecting "the last N quarters from today's date" as the default would re-open this ADR because the `Q2 2026` / `Q3 2026` literals would need a calendar to compute against, a dependency this ADR explicitly avoids.
