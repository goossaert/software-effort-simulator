# Constant work as an optional third CSV with deterministic effort and post-sort distribution shift

> **Amended by [ADR-0033](./0033-constant-work-scoped-to-groups-by-category.md) and [ADR-0034](./0034-editable-constant-work-tab.md).** The deterministic-effort mechanism below (closed-form lognormal mean, post-sort shift preserving sort order, **Synthetic** ↔ **Empirical** toggle) stands. Two things changed: (1) [ADR-0033](./0033-constant-work-scoped-to-groups-by-category.md) replaces the *single global `fixedEffort` scalar applied to every Scenario* with a *per-**Group** vector* scoped by **Category** membership — so the "all constant work, every Group/cell, every surface" behaviour described here (the org/team/projection scope rule, `globalMin = fixedEffort`, and the unconditional appended **Initiative matrix** rows) is superseded by Category-gated scoping on all three surfaces; (2) [ADR-0034](./0034-editable-constant-work-tab.md) makes constant work editable in-app (`editedConstantWork`, an editable **Constant work** tab with add/delete rows, from-scratch authoring, and CSV export), so `parsedConstantWork` is no longer the simulation source of truth.

The simulator accepts an optional third CSV — the **Constant Work CSV** — alongside the required **Initiatives CSV** and **Epics CSV**, describing work that is *guaranteed* to ship in a given (team, quarter) pair. The CSV is uploaded via a dedicated sidebar control (`#constant-work-file`, `index.html:866`), parsed once into `parsedConstantWork` (`index.html:1500`, `1641-1644`), and consumed by three sites only: `getConstantWorkEffort(quarters, teamName?)` (`index.html:1650-1662`) which sums deterministic person-months, `getConstantWorkEpics(quarter, teamName)` (`index.html:1668-1686`) which returns display-formatted rows for the **Initiative matrix**, and the `fixedEffort` shift inside `runSimulation` (`index.html:2086`, `2103-2118`). Constant-work effort is computed *deterministically* from t-shirt size via `tshirtToPersonMonths(size)` (`index.html:1272-1276`), which returns the closed-form lognormal mean `e^(μ + σ²/2)` of the active parameter set — *not* a sample, *not* a P50. The resulting scalar is added to every per-**Iteration** total *after* the Monte Carlo sort, so the **Histogram**, **Stats**, and chart **Global histogram range** of every **Scenario** are translated by exactly `fixedEffort` PM with no reshaping of the distribution. The constant-work rows themselves are appended to the bottom of each **Projection section**'s **Initiative matrix**, tinted soft green (`#f0fdf4`), with the t-shirt size and estimated PM annotated in the name cell (`[M · ~2.12 PM]`).

We chose to **deliver constant work as a third optional CSV with its own fixed schema** — `jira_key`, `epic_name` / `building_block`, `key_result` / `KR` / `kr`, `moscow`, `team`, `quarter`, `tshirt_size` / `t_shirt_size` — rather than as extra rows or a flag on the **Initiatives CSV** or **Epics CSV**, because constant work is a *different kind of entity*: it is an **Epic**-grained commitment with a *team and quarter already assigned*, no parent **Initiative**, and a sampling mode (deterministic, not stochastic) that contradicts the **Initiatives** + **Epics** model the rest of the engine is built on. Folding it into the existing CSVs would force either (a) a parent-initiative placeholder per constant-work row (breaking the "an **Initiative** is the unit the user prioritises" semantics ADR-0006 documents) or (b) a `mode: stochastic | deterministic` column on every **Epic** row (turning a 5-column CSV into a 7-column one and asking every existing export to grow a new field). Keeping the file separate means existing exports continue to work unmodified — the user simply uploads or omits the third CSV — and the column detector family ([ADR-0005](./0005-content-based-column-detection.md), [ADR-0021](./0021-sensible-csv-format-dual-support.md)) does not need to grow a new entry. The constant-work CSV is the user's *own template*, hand-authored to a documented schema, so it bypasses content-scan detection entirely; the column names are hardcoded in `getConstantWorkEffort` and `getConstantWorkEpics` and tolerated variants (`tshirt_size | t_shirt_size`, `jira_key | epic_key`, `epic_name | building_block`, `key_result | KR | kr`) are inline `||` chains. This is intentional — a future revision that converged the constant-work CSV onto the detector family would re-open this ADR jointly with [ADR-0005](./0005-content-based-column-detection.md).

We chose to **compute constant-work effort deterministically as the closed-form lognormal mean** `e^(μ + σ²/2)` rather than (a) sampling once per Run, (b) using a hand-picked PM number on each row, or (c) using the **T-shirt size reference**'s P50, because "guaranteed work" semantically demands a *point estimate* with no Monte Carlo noise — the user has already committed; they want to see how the rest of the plan fits *around* a fixed budget, not what would happen if the constant work itself happened to land at the high or low end of its size band. The closed-form *mean* (rather than the mode or median) is the right point estimate because the same t-shirt size's stochastic samples will, in expectation, average to that value over many iterations; using the mean preserves the property that "running the constant-work load through the stochastic pipeline" and "running the constant-work load through the deterministic pipeline" agree on long-run averages. Using the **T-shirt size reference**'s P50 would understate effort (right-skew of lognormal puts the mean above the median); using a hand-picked PM number per row would push the user into picking a value, which is precisely the friction we are trying to avoid by reusing the existing t-shirt vocabulary. The function `tshirtToPersonMonths` lives in Module 3 next to `T_SHIRT_PARAMS` so it transparently follows the **Synthetic parameters** ↔ **Empirical parameters** toggle: switching to empirical recalibrates every constant-work row on the next Run without any per-row edit.

We chose to **add `fixedEffort` to every per-Iteration total *after* the Monte Carlo sort** — `shifted[i] = sorted[i] + fixedEffort`, `index.html:2105-2113` — rather than (a) injecting a synthetic constant-work "epic" into the bootstrap pool, (b) adding it inside the per-iteration loop, or (c) treating constant work as a separate Scenario, because adding a constant to a sorted array preserves the sort order, which means every downstream consumer — `computeStats` (`index.html:2048`), `buildHistogram` (`index.html:2072`), `percentileBinarySearch` (inside `computeStats`) — works on the shifted array with zero special-casing. Every reported percentile (P10/P25/P50/P75/P90) is then automatically `original_percentile + fixedEffort`; every `P(effort > capacity)` is automatically `P(original > capacity − fixedEffort)`; every bin centre on the **Histogram** sits `fixedEffort` PM to the right of where it would otherwise. Approach (a) — pool injection — would mix a deterministic value into a *stochastic* sampler and require an "always sample, never recurse" gadget; approach (b) — in-loop addition — would do the same `n × iterations` additions instead of `n` after the sort and require touching `runScenario`; approach (c) — separate Scenario — would force the chart to render four datasets instead of three and the stats table to grow a "constant work" row that conveys no per-Iteration information. The post-sort shift is the cheapest, most invariant-preserving option.

We chose to **make `globalMin` equal to `fixedEffort`** rather than holding `globalMin` at `0`, because the resulting distribution no longer has support below `fixedEffort` — every sample is shifted up by the same amount — and leaving `globalMin = 0` would produce a histogram whose leftmost ~`fixedEffort`-wide span is *guaranteed* to be empty bars. Setting `globalMin = fixedEffort` (`index.html:2117`) keeps the 60 fixed-width bins concentrated on the actual support of the shifted distribution, so the chart x-axis reads sensibly: "with `fixedEffort` PM already committed, here is where the rest of the plan lands." The corresponding `globalMax` change (`Math.max(p995(...), fixedEffort + 1)`, `index.html:2118`) ensures the floor-of-1-PM-range guard still holds when the entire shifted distribution sits at exactly `fixedEffort`. A future revision could surface `fixedEffort` as a vertical marker on the chart (e.g. "0 PM is here, constant work eats X PM, stochastic load sits to the right") — additive, would not re-open this ADR.

We chose to **scope constant-work effort differently between the org-level and the team-level Runs** — `getConstantWorkEffort(targetQs)` at the org level sums across *every* team in the target quarters, while `getConstantWorkEffort(targetQuarters, teamName)` at the team level filters to one team — because the org-level **Run** is asking "does the whole org fit?" and must include every committed line item, while a **Team Level tab** section is asking "does this team fit?" and must include only that team's commitments. The third call site, inside `buildTeamProjections`'s **Quick projection Monte Carlo** (`index.html:1980-1981`), uses `cwEffort = cwEpics.reduce((s, e) => s + e.effort, 0)` — a per-team-per-quarter sum — because each **Projection section** renders one effort band per quarter and the band must reflect *that quarter's* constant work for *that team*. The three scope choices encode the same rule: "constant work is summed over exactly the (team, quarter) cells the surface is showing", and the case-insensitive team-name match (`rowTeam !== teamName.toLowerCase()`, `index.html:1658`) tolerates the same dedup convention the **Team Level tab** uses for **Team** ordering ([ADR-0019](./0019-per-team-independent-simulations.md)).

We chose to **render constant-work rows inside the existing Initiative matrix** rather than in a separate table or panel, because constant work is *conceptually a kind of work the team will deliver in the quarter* — the same thing every Initiative-derived row in the matrix represents — and surfacing it on a separate canvas would force the user to read two tables to answer the question "what is this team doing next quarter?". Appending the constant-work rows *after* the sorted Initiative rows preserves the matrix's primary sort (Must → Should → Could → Won't → Unknown, then alphabetical by Jira key) for stochastic work, and uses position alone to convey "below this line is deterministic"; the soft-green tint (`background:#f0fdf4`, `index.html:2630`) is the visual confirmation. The `[size · ~PM]` annotation in the name cell (`[M · ~2.12 PM]`, `index.html:2628`) carries the per-row effort the simulator computed — the user reads "this row contributed 2.12 PM to the deterministic shift". The `isConstant` flag (`index.html:1684`) is the *only* state needed to drive the render-time fork, and is carried on the constant-work row object end-to-end. The matrix's **Effort projection band** footer row still summarises the *stochastic* component plus the `cwEffort` shift, so the user reads the per-row constant-work contribution (in the cell annotation) and the total quarterly effort (in the footer band) without doing arithmetic. The conditional KR column ([ADR-0022](./0022-optional-key-result-column.md)) reads each constant-work row's `kr` field via the parallel `r.key_result || r.KR || r.kr` lookup so constant-work rows participate in the `hasKr = allInits.some(i => i.kr)` gate — see [ADR-0022](./0022-optional-key-result-column.md) for why the lookup list differs from `detectKrCol`'s.

We chose to **expand `buildTeamProjections`'s quarter axis to include constant-work-only quarters** — `activeQuartersForTeam = [...new Set([...allQuarters, ...cwQuarters])]` (`index.html:1940-1947`) — rather than restricting it to the **Initiatives CSV**'s quarters, because a team can legitimately have *only* constant work in a future quarter (e.g. a "Q4 2026: hosting infrastructure renewal, no roadmap commitments yet") and dropping that quarter from the matrix would silently hide a real commitment. The union is computed per-team (so a quarter present only in one team's constant work appears in that team's section only) and the per-cell render gate `if (!qInits.length && !cwEpics.length) continue;` (`index.html:1952`) keeps the cell out when nothing matches. When the **Quick projection Monte Carlo** is skipped (no MSC initiatives, `orgLambda === 0`, or empty bootstrap pool — see [ADR-0020](./0020-team-projections-cross-quarter-view.md)), the band defaults to `{p25: cwEffort, p50: cwEffort, p75: cwEffort}` (`index.html:1985`) — a *flat* band that visually reads as "this cell is constant-work only" on the effort projection chart. This is intentional and preserves the rule that every rendered cell carries an effort number; a triple of zeros would have produced a chart artifact for a cell that *does* carry deterministic effort.

This decision pairs with [ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md) (the Monte Carlo engine that produces the stochastic distribution this feature shifts), [ADR-0007](./0007-lognormal-effort-distribution.md) (the lognormal whose closed-form mean `tshirtToPersonMonths` evaluates), [ADR-0011](./0011-overlapping-histograms-shared-bins.md) (the shared-bin **Global histogram range** whose lower bound this feature moves up to `fixedEffort`), [ADR-0019](./0019-per-team-independent-simulations.md) (the team-scoped Run whose `fixedEffort` is the per-team constant-work sum), [ADR-0020](./0020-team-projections-cross-quarter-view.md) (the **Team Projections tab** that surfaces constant-work rows in the **Initiative matrix** and folds `cwEffort` into the **Quick projection Monte Carlo**), and [ADR-0022](./0022-optional-key-result-column.md) (the parallel `key_result | KR | kr` KR lookup on the constant-work row). The implementation is the sidebar `#constant-work-upload` block (`index.html:861-872`), the `parsedConstantWork` module-scoped cache (`index.html:1500`), the loader `loadConstantWorkCSV` and reset `resetConstantWorkFile` (`index.html:1631-1644`), the two consumer helpers `getConstantWorkEffort` and `getConstantWorkEpics` (`index.html:1650-1686`), the `tshirtToPersonMonths` closed-form mean (`index.html:1266-1276`), the `fixedEffort` parameter on `runSimulation` (`index.html:2086`) and its `shift`/`globalMin`/`globalMax` block (`index.html:2103-2118`), the per-team-quarter `cwEffort` thread through `buildTeamProjections` (`index.html:1940-2003`), the org-level `orgFixedEffort` thread through the run-button handler (`index.html:3343-3344`), the `td.fixedEffort` thread through `prepareTeamSimulationData` / `renderTeamSection` (`index.html:1900-1901`, `2431`), and the constant-work row render fork inside `renderTeamProjections` (`index.html:2626-2634`) with the soft-green CSS swatch (`.mb-constant`, `index.html:651`).

A future revision could **promote constant work to a stochastic-with-tight-CV mode** (still a known commitment, but with a small variance around the size's mean) — would re-open this ADR because the "deterministic, sorted-shift-preserves-order" invariant is the load-bearing decision here. A future revision could **surface a vertical marker at `fixedEffort` on every chart** ("constant work eats X PM") — additive, would not re-open this ADR. A future revision could **expand the constant-work CSV schema** with a `notes` column or a `confidence` field — additive at the inline-lookup level, would not re-open this ADR. A future revision could **converge the constant-work CSV onto the column detector family** ([ADR-0005](./0005-content-based-column-detection.md)) so it tolerates **Quirky format** quirks — would re-open this ADR because the "hand-authored template with hardcoded headers" rule is load-bearing. A future revision could **render constant-work rows in their own section above the Initiative matrix** instead of below it — would re-open this ADR because the "appended-after-sorted-Initiatives, soft-green tinted" rule encodes how the user reads "stochastic above the line, deterministic below". A future revision could **drop the org-level aggregation** ("constant work belongs to a team, never to the org") — would re-open this ADR jointly with the scope rule above, because the org-level **Run**'s `fixedEffort = sum over all teams` is the source of `P(org effort > org capacity)` accuracy when constant work is present.
