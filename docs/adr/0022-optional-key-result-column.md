# Optional Key Result column: header-variant detection, conditional rendering, display-only role

The **Initiatives CSV** can optionally carry a **Key Result** column linking each **Initiative** to the strategic outcome it serves. The column is detected by `detectKrCol(rows)` (`index.html:1436-1450`) — a flat list of header variants (`key_result`, `kr`, `key result`, `keyresult`, `key_results`) checked first by exact match and then by a case-insensitive fallback — and stored on `detectedCols.krCol` as either the matched header name *or* `null` when no variant is present. Per-initiative values are read in `buildTeamProjections` (`index.html:1964`) as `krCol ? (r[krCol] || '').trim() : ''`, and surfaced in the **Initiative matrix** inside each **Projection section** as an extra `KR` column — *only* when at least one row in that section has a non-empty `kr` (the `hasKr = allInits.some(i => i.kr)` gate at `index.html:2615`). The KR string is *never* consumed by the simulation engine: it does not become a sampling key, a grouping axis, or a capacity-allocation input. It is purely a display column that ties the initiative back to its outcome at planning time.

We chose to **detect the KR column by a flat list of header variants**, rather than by the **Content scan** strategy ([ADR-0005](./0005-content-based-column-detection.md)) used for the Initiative key and MoSCoW columns, because Key Results have no content fingerprint. A KR cell can hold `KR-7`, `Activation +10% YoY`, `North Star: weekly active builders`, or any free-text label the org has settled on; there is no regex that distinguishes a KR value from a generic notes column. A content scan would either miss every reasonable KR or false-positive on any short-string column. The header-variants list is the only signal that survives the diversity of KR-naming conventions across exports — and because KRs are conventionally labelled with a small, well-known set of header names (the five we enumerate), the false-negative rate is low. The variants list is deliberately *not* extensible per-CSV: a user whose tooling produces a header outside the list (e.g. `okr`, `okr_key`, `outcome`) can rename the header to one of the five canonical variants, or extend the list in a future revision. We treat the list as the canonical KR vocabulary for the project, not as an exhaustive heuristic.

We chose to **layer header detection in two passes — exact match first, case-insensitive fallback second** — because the **Sensible format** convention is lowercase headers (`jira_key`, `building_block`, `moscow`, `teams`) but the legacy **Quirky format** and some hand-edited exports use mixed case (`Key_Result`, `KR`). Doing only an exact match would miss the latter; doing only a case-insensitive match would conflict with the byte-exact comparisons the other detectors use ([ADR-0021](./0021-sensible-csv-format-dual-support.md)). The two-pass structure preserves the byte-exact discipline of the other detectors as the primary path (so `'kr'` and `'key_result'` are picked up on the cheap branch) while still catching `KR` and `Key Result` without forcing the user to rename. The fallback walks `headers.map(h => h.toLowerCase())` and returns the *original* (raw-case) header name on hit, so downstream `row[krCol]` reads remain valid against PapaParse's case-preserving parse.

We chose to **return `null` when no KR column is present**, rather than returning a sentinel like `'key_result'` (the **Sensible format** default the other detectors fall back to), because the KR column is *optional* in the data model — its absence is a normal state, not a failure. Every other Initiative-side detector (`detectInitKeyCol`, `detectMoscowCol`, `detectNameCol`, `detectTeamCol`) returns a string because the column *must* exist for the simulator to function: a CSV with no Initiative key is unusable, a CSV with no MoSCoW column produces no scenarios. The KR column carries no engine-level meaning, so returning a string when no column exists would force every consumer to either (a) read from a non-existent header and silently get `undefined`, or (b) special-case `''` versus `null` themselves. Returning `null` makes the optionality explicit at the type level: every consumer reads `krCol ? r[krCol] : ''` (`index.html:1964`) and the absent-column case threads through cleanly. The `null` also surfaces in the **Column-detection debug** panel as a visible signal that the simulator looked for the column and did not find it — distinguishing "absent" from "empty values".

We chose to **render the KR column only when at least one initiative in the Projection section has a non-empty `kr`**, gated by `const hasKr = allInits.some(i => i.kr)` (`index.html:2615`), rather than rendering it whenever `detectedCols.krCol !== null`, because the **Initiative matrix** is already a wide table — `Jira Key`, optional `KR`, `Initiative Name`, then one column per quarter in `allQuarters`. Adding an empty KR column to every section regardless of content would (a) waste horizontal space on teams that simply have no KR-tagged initiatives, and (b) confuse the user into thinking the column is required when it is merely available. The two-stage check (column-detected at load time, value-present at render time) lets the matrix degrade gracefully on per-team granularity: a CSV with a KR column populated for the `Platform` team but blank for the `Risk` team renders the column under `Platform`'s section and omits it from `Risk`'s. The gate is *also* applied to the `<tfoot>` `colspan` (`index.html:2692`, `2698`) so the count-chips and effort-band footer rows line up with the column count actually rendered.

We chose to **surface the KR column only in the Team Projections Initiative matrix**, not in the org-level histogram, the Team Level tab, the Initiatives tab ([feature 0019](../../backtracked-features.md#0019)), or the **Data preview**, because the matrix is the single surface where the user reads initiatives as *named, prioritised, time-allocated commitments* — exactly the context in which "what outcome does this serve?" is the next question. The org-level histogram and the Team Level tab show *distributions of effort*, where the KR string would have no analogue; the **Data preview** is a fitted-model summary, not an initiative list; the Initiatives tab is a row-level editor whose schema is the raw CSV (the KR cell is already visible there by default if the column is present, as one of the editable columns — but the tab does not *promote* it to a first-class field). Surfacing the KR everywhere would dilute the signal; surfacing it only where the user is asking the planning question is the targeted choice. A future revision could surface the KR in additional contexts (a per-KR roll-up chart, a per-KR capacity allocation) — see the "Known future ideas" list in `backtracked-features.md` — and would re-open this ADR jointly with [ADR-0020](./0020-team-projections-cross-quarter-view.md).

We chose to **keep the KR string out of the simulation engine entirely** — no per-KR Poisson λ, no per-KR effort distribution, no per-KR capacity bucketing — because the simulator's model ([ADR-0006](./0006-monte-carlo-with-bootstrapped-sizes.md), [ADR-0007](./0007-lognormal-effort-distribution.md), [ADR-0008](./0008-poisson-epic-count.md)) is built on **Initiative** counts and **T-shirt size**–driven epic effort. The KR is a *labelling* axis attached to an Initiative; it has no statistical relationship with epic count or epic size that the simulator can fit. Treating the KR as a sampling dimension would require either (a) a per-KR historical pool — usually too sparse to fit, the same problem [ADR-0019](./0019-per-team-independent-simulations.md) documents at the per-team level — or (b) a synthetic prior, which would introduce yet another hand-maintained parameter table on top of `T_SHIRT_PARAMS`. Neither is justified by the planning value the KR currently delivers (a column in the matrix is enough). Keeping the KR display-only also means it can never *silently* affect a Run output — if a future feature wants to use it, that feature will declare the use explicitly.

We chose to **read the KR field on constant-work rows via a parallel path** — `(r.key_result || r.KR || r.kr || '').trim()` inside `getConstantWorkEpics` (`index.html:1680`) — rather than reusing `detectKrCol` against the **Constant Work CSV**, because the constant-work file has a *fixed* schema with known column names ([feature 0015](../../backtracked-features.md#0015)) and does not go through the detector family. The parallel lookup is a three-element preference list — `key_result`, `KR`, `kr` — and is deliberately *not* the same five-element list as `detectKrCol` because the constant-work CSV is hand-authored to a documented template and is expected to use one of those three exact spellings. Diverging the two lists is intentional: the Initiatives CSV admits more variants because it is exported from disparate tooling; the constant-work CSV is the user's own template. If a future revision converged the two schemas (one shared detector family across both CSVs), that change would re-open this ADR.

This decision pairs with [ADR-0005](./0005-content-based-column-detection.md) (content scan as the canonical detection strategy for content-detectable columns — KR is the explicit exception), [ADR-0020](./0020-team-projections-cross-quarter-view.md) (the **Team Projections tab** is the single surface where the KR column is promoted), and [ADR-0021](./0021-sensible-csv-format-dual-support.md) (the per-detector branch-ordering rules — KR is uniform across formats and so does not need the asymmetric ordering of name and team). The implementation is the `detectKrCol` function (`index.html:1436-1450`), the integration line in `loadInitiativesCSV` (`index.html:1511`), the read in `buildTeamProjections` (`index.html:1964`), the `hasKr` gate and KR cell in `renderTeamProjections` (`index.html:2615-2624`, `2684`, `2692`, `2698`), and the parallel constant-work lookup in `getConstantWorkEpics` (`index.html:1680`).

A future revision could **promote the KR into a sampling dimension** (per-KR Poisson λ, per-KR effort bands, per-KR capacity allocation) — would re-open this ADR jointly with [ADR-0008](./0008-poisson-epic-count.md) because the "count is fit at the org level, then optionally at the team level" rule would need to grow a third axis. A future revision could **render the KR column in the org-level chart's hover tooltip** or in the Team Level tab's stats table — additive, would not re-open this ADR. A future revision could **expand the header-variants list** in `detectKrCol` (`okr`, `okr_id`, `outcome`) — additive at the variant-list level, would not re-open this ADR. A future revision could **render the KR column unconditionally** (even when no initiative has a value) to nudge the user toward filling it in — would re-open this ADR because the "only-when-present" rule is load-bearing. A future revision could **converge the Initiatives-CSV KR detection with the Constant-Work-CSV KR lookup into a single shared helper** — additive at the API level, would not re-open this ADR unless the converged helper changed the variants list materially.
