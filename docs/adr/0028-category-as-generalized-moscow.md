# Category as generalized MoSCoW with backward-compat header cascade

The simulator's per-**Initiative** priority dimension — historically the **MoSCoW** column with the fixed enum `Must | Should | Could | Won't | unknown` — becomes the **Category** column carrying arbitrary free-form strings (`KR1`, `Automation`, `Project ABC`, `Must`, anything). The canonical CSV header is `category`; the **Column detector** family gains a new `detectCategoryCol` whose strategy is a *header-name cascade* — `category` first, then `moscow` (legacy MoSCoW Sensible-format), then `emoji` (legacy **Quirky format** quirk where MoSCoW lived in the emoji column) — with no **Content scan** branch. Empty / whitespace-only values normalise to a reserved **(Blank) sentinel** distinct from any user-typed string and surfaced in the UI as the literal `(Blank)` with parentheses. Equality between two category strings — the lookup the engine performs to test whether an initiative belongs to a **Group** ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)) — is `trim() + case-insensitive`, with the first-seen casing preserved for display (the same convention as team-name dedup in `prepareTeamSimulationData`). The badge rendered in the **Initiative matrix** becomes a uniform neutral grey label — colour is no longer carried by the Category, it is carried by the **Group** the user chooses to put the Category in.

We chose to **generalise the existing column rather than add a new orthogonal axis alongside `moscow` and `key_result`**, because the user's planning vocabulary occupies one axis of categorisation per initiative, not three. The existing `moscow` field models "which priority bucket?"; the existing optional `key_result` field models "which outcome does this contribute to?" ([ADR-0022](./0022-optional-key-result-column.md)); a hypothetical third `category` field would have just been a third way to bucket the same initiative and would have forced the user to pick which axis drives the simulation at every Run. Generalising `moscow` keeps the simulator's input shape stable — one categorical axis per initiative, the user labels it however their planning convention requires — and leaves `key_result` untouched in its display-only role (so a team that already records KR labels separately can keep both: a `category` column for what the simulator runs against, and a `key_result` column for the matrix display). The cost is the column's name no longer matches its semantics for legacy MoSCoW data, which is the entire reason for the rename below.

We chose to **rename the column from `moscow` to `category`** — in the CSV canonical header, in the JS field name (`row.category`), and in the domain glossary ([CONTEXT.md](../../CONTEXT.md)) — rather than (a) keep `moscow` as the header forever, (b) call it `bucket` to match the internal `moscowGroups.must` vocabulary, or (c) call it `group` to match the user-facing Phase-2 noun, because future readers of a CSV holding values like `Automation`, `KR1`, `Project ABC` need the header to honestly describe the data. Keeping `moscow` would have forced every entry of [CONTEXT.md](../../CONTEXT.md) to footnote "the column called moscow may not contain MoSCoW values" — semantic debt that compounds as readers proliferate. `bucket` is internal vocabulary (the engine still talks about `categoryGroups.<name>` internally where MoSCoW once talked about `moscowGroups.must`), but it is a low-information English noun that adds no clarity for the user. `group` collides directly with the new Phase-2 noun ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)) where a **Group** is a named subset *of* Categories — two different things called the same word would be a permanent ambiguity. `category` is the clearest plain-English term, matches how the user described it during the design session, and aligns with the existing Phase-1 mental model.

We chose to **detect the column via a header-name cascade `category → moscow → emoji`, dropping the content-scan regex**, rather than retaining the [ADR-0005](./0005-content-based-column-detection.md) **Content scan** strategy that fingerprinted MoSCoW values via `/must|should|could|won.t/i`, because arbitrary strings cannot be regex-detected. A category of `KR1` matches no pattern; a category of `Automation` matches no pattern; a content scan that fires only on MoSCoW-shaped values would silently fail on every non-MoSCoW dataset and the user would face an opaque "no category column detected" error. Header-name detection is deterministic, fast, and visible in the **Column-detection debug** panel — the user can see exactly which header the detector picked and why. The cost is that a CSV whose category column carries no recognisable header (an extreme **Quirky format** outlier) will no longer auto-detect; the user must rename their header to one of the three accepted names. We accept this because (a) the **Sensible format** ([ADR-0021](./0021-sensible-csv-format-dual-support.md)) is the path forward and the cascade preserves the two header names the Sensible format and Quirky format actually used (`moscow` and `emoji`), and (b) the failure mode is now visible — no column detected → every initiative gets the **(Blank) sentinel** and the Groups tab surfaces this immediately to the user, rather than the previous silent-misclassification failure mode where the content scan picked the wrong column.

We chose to **normalise category values with `trim() + case-insensitive` comparison**, with first-seen casing preserved for display, rather than (a) literal byte-for-byte string equality, (b) lowercase-everywhere internally with title-case display, or (c) inheriting the `normalizeMoscow` strip-non-ASCII step, because real CSVs accumulate trivial whitespace and casing inconsistencies that should not split a category into two ghosts in the chart legend. A user who types `Automation` in one row and `automation` in another almost certainly means one category. The first-seen casing convention is what `prepareTeamSimulationData` already uses for team-name dedup ([CONTEXT.md L54](../../CONTEXT.md)), so a user familiar with that surface gets the same behaviour here. Literal equality would have produced two visually-similar legend entries differing only in case — confusing and almost never intended. Lowercase-internal would have lost the user's chosen casing in every display surface (badge, legend, group editor). The strip-non-ASCII step from `normalizeMoscow` exists historically to handle emoji-prefixed Quirky-format MoSCoW labels (`🔴Must`, etc.) — but a user who deliberately includes an emoji in a free-form category name is doing so intentionally (perhaps `📊 Analytics` is a meaningful Category label), and stripping it silently would split that one Category from any other reference to the same string elsewhere. We preserve emojis in category values; users wanting MoSCoW-emoji compatibility can rename their values on load.

We chose to **render the badge in the Initiative matrix as a uniform neutral grey label** — same neutral background as the existing `?` (unknown) badge — rather than (a) keep per-MoSCoW-value colours (red/orange/green/grey) for those specific strings, (b) hash the category name to a deterministic colour palette, or (c) introduce a per-Category colour-management UI separate from the per-Group colour, because colour now carries the **Group** semantic exclusively ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)). Per-MoSCoW-value colours would have been a special case for the four legacy strings that doesn't generalise — and it would have invited two competing colour systems on the same screen (group colour on the chart, category colour on the matrix) that don't line up because a Category can appear in multiple Groups. Hashing to a palette is aesthetically inconsistent and accessibility-unfriendly. A per-Category colour management UI duplicates the per-Group colour UI for no additional information. Legacy MoSCoW users lose the at-a-glance red/orange/green ranking in the Initiative matrix — they recover priority signal via the new alphabetical-with-(Blank)-last sort and the chart's per-Group colour scheme.

We chose to **represent missing / whitespace category values as a reserved Blank sentinel surfaced as the literal `(Blank)` with parentheses**, rather than (a) the literal string `"Blank"`, (b) `null` everywhere with a context-dependent display, or (c) a per-Group `includeBlank: boolean` flag separate from the member-set, because the parentheses are the disambiguator: a user who legitimately names a category the string `Blank` (perhaps a deliberate "this initiative is in the Blank state" label) gets a Category called `Blank` distinct from the missing-data sentinel rendered `(Blank)`. The literal-string approach would have silently collided. A separate `includeBlank` flag would have created a second concept the user has to learn — two ways to express membership where one suffices. The sentinel approach is the cleanest data model: a Category is either a string or the sentinel, and the **Group**'s `members` array contains a mix of strings and the sentinel.

This decision pairs with [ADR-0005](./0005-content-based-column-detection.md) (the **Content scan** branch of column detection is no longer used for the category column; it remains in place for the **Initiative key** detector), [ADR-0021](./0021-sensible-csv-format-dual-support.md) (the header-name cascade preserves both the **Sensible format** `moscow` and the **Quirky format** `emoji` legacy names — both formats continue to load without intervention), [ADR-0022](./0022-optional-key-result-column.md) (the **Key Result** column is unchanged and remains the display-only outcome label), [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (the Initiatives tab's category cell editor becomes a `<input list="category-options">` datalist combo seeded from `getUniqueColumnValues(parsedInitiatives)`, replacing the constrained `<select>` to accept free-form new values), and [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) (the **Group** concept that consumes Category strings as its member-set). The implementation touches `normalizeMoscow` (rename to `normalizeCategory`, rewrite to `trim + case-fold-for-comparison, preserve first-seen casing`), `detectMoscowCol` (rename to `detectCategoryCol`, rewrite as a header-name cascade with no content-scan branch), `getConstantWorkEpics` (the **Constant Work CSV** schema gains the same `category → moscow → emoji` cascade for its category column), `moscowBadge` (rewrite to `categoryBadge` rendering a single neutral style), `editedInitiatives[rowIdx][col]` write paths (the column key becomes `category`), and every CONTEXT.md / CSS / inline-script reference to the term `MoSCoW`.

A future revision could **accept the `priority` header as a fourth cascade entry** if a third-party export uses that name — additive, would not re-open this ADR. A future revision could **allow per-Category colour assignment** via a new column-management UI — would re-open this ADR because the "Category has no colour, Group has colour" invariant is load-bearing. A future revision could **support multiple categories per Initiative** (e.g., a comma-separated list parsed as a set) — would re-open this ADR because the "one Category per Initiative" cardinality is load-bearing. A future revision could **strip emoji from category values** to recover the historical normaliseMoscow strip-non-ASCII step — would re-open this ADR because the current "emoji are part of the category label" rule is deliberate.
