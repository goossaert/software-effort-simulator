# Editable Constant Work tab with add/delete rows and CSV export

We chose to **give the Constant Work CSV the same editable-tab treatment the Initiatives tab received in [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md)** — a new sixth **Tab**, `Constant work`, rendering every constant-work row in an editable table with an `↓ Export CSV` button, under the same **commit-on-Run** discipline. On load, `parsedConstantWork` is shallow-cloned per row into a new mutable `editedConstantWork`, which becomes the **simulation source of truth**: the consumer helpers `getConstantWorkEffort` and `getConstantWorkEpics` read from `editedConstantWork` instead of `parsedConstantWork`. Field edits, added rows, and deletions commit to `editedConstantWork` immediately, but charts and stats lag until the next **Run Simulation** — the identical rhythm a user has already learned on the **Initiatives tab** and **Groups tab**. `parsedConstantWork` is retained only as the immutable substrate for the tab's dropdown/datalist option pools, mirroring how `parsedInitiatives` survives ADR-0027.

We chose to **give the Constant Work tab add-row, delete-row, and from-scratch authoring — capabilities the Initiatives tab ([ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md)) deliberately does *not* have**. This asymmetry is the surprising part and the reason this ADR exists: a reader comparing the two editable tabs will ask why one grows rows and the other does not. The answer is that the two CSVs are different kinds of input. An **Initiative** row participates in a join — its **Initiative key** is the link target for **Epics CSV** rows — so its structure is owned by the uploaded files, and the Initiatives tab keeps the key and name cells read-only precisely to protect that join; inventing initiative rows in-app would create keys with no epics behind them. A **constant-work** row has *no* join: its effort is computed from its own `tshirt_size` via `tshirtToPersonMonths`, with no parent **Initiative** and no child **Epic** (per [ADR-0023](./0023-constant-work-csv-deterministic-shift.md), constant work is an epic-grained commitment standing alone). The Constant Work CSV is also, by ADR-0023's design, "the user's own template, hand-authored to a documented schema" — often a short list a user would rather build or tweak in-app than round-trip through a spreadsheet. So the Constant Work tab is always present (pre-rendered every Run like the other tabs), and with nothing imported it shows an empty table plus `+ Add row`, letting the user author constant work from zero; an added row carries the **canonical schema** (`jira_key`, `epic_name`, `key_result`, `category`, `team`, `quarter`, `tshirt_size`) when there is no import to mirror, or the imported file's header set when there is, so all rows share columns. Deletion is immediate with no confirmation, matching the Groups tab's per-row delete; re-import or re-add recovers a row.

We chose **smart per-field editors rather than verbatim Initiatives-tab parity**. Because there is no join to protect, *all* cells are editable, including `jira_key` and `epic_name` (a newly added row needs them fillable anyway). Three editors are deliberate: `tshirt_size` is a `<select>` constrained to the seven **Recognised t-shirt sizes** (`2XS`…`XL+`) — closing the ADR-0023 footgun where an unrecognised size silently contributes `0` PM with no warning; `category`, `team`, and `quarter` are HTML5 `<input list>` datalist combos seeded from the observed **union** across `editedInitiatives` and `editedConstantWork`, so the user can pick an existing value (the merge path — typing the same Category as an Initiative makes the two one entry in the **Groups tab**, see [ADR-0033](./0033-constant-work-scoped-to-groups-by-category.md)) or type a new one; `jira_key`, `epic_name`, and `key_result` are free-text inputs. Any unrecognised extra column falls back to a free-text input. New rows are blank and validation is **lenient** — a blank size is `0` PM, a blank quarter is excluded from every simulation, a blank Category is the **(Blank) sentinel** — each mirroring how those blanks already behave elsewhere, so no Run gate is imposed.

We chose to **make Export round-trip stable by preserving the input file's headers verbatim**, exactly as the Initiatives export does. `↓ Export CSV` serialises `editedConstantWork` via `Papa.unparse` and downloads `constant-work-edited.csv`, echoing whatever headers were imported — including alias spellings (`epic_key`, `building_block`, `moscow`, `t_shirt_size`) and any extra columns — so re-importing the exported file reproduces the identical model and Run output. This is the literal reading of "the same column naming convention as the input file." Authoring from scratch (no import) exports the canonical schema. We rejected normalising aliases to canonical names on export because that would *rename* a quirky/legacy input's columns and break the round-trip contract.

This decision pairs with [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (the editable-tab + commit-on-Run + immutable-substrate pattern it copies), [ADR-0023](./0023-constant-work-csv-deterministic-shift.md) (the fixed constant-work schema and `tshirtToPersonMonths` effort it edits), [ADR-0033](./0033-constant-work-scoped-to-groups-by-category.md) (the per-Group scoping the edited rows feed, and the datalist union that drives Category merging), and [ADR-0018](./0018-tab-based-results-layout.md) (the tab bar this adds a sixth entry to). The constant-work consumer helpers move their read from `parsedConstantWork` to `editedConstantWork`; `parsedConstantWork` stays for option pools.

A future revision could **add a Duplicate-row action** (as the Groups tab has) — additive, would not re-open this ADR. A future revision could **back-port add/delete rows to the Initiatives tab**, which would require solving the orphaned-key problem with the Epics CSV — would re-open ADR-0027, not this ADR. A future revision could **persist `editedConstantWork` to browser storage** rather than relying on CSV export — would re-open this ADR jointly with the client-side-only stance of [ADR-0002](./0002-client-side-only.md).
