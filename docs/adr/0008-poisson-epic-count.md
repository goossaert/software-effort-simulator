# Poisson distribution for epic count per initiative

For each of the `K` **Initiatives** in a **Scenario**, the number of **Epics** that **Initiative** will spawn in an **Iteration** is sampled from `Poisson(λ)`, where `λ` is the *mean number of epics per initiative* in the **Historical quarter** (computed as `total_epics_in_scope / count_of_initiatives_in_scope` over the user-selected historical quarters; see `prepareSimulationData`, `index.html:1705`).

We chose Poisson because it is the canonical discrete distribution for *count of independent arrivals*, requires only one parameter (which we can fit from historical data with a single mean), and produces the realistic spread we observe: most initiatives generate a handful of epics, a few generate many, and an occasional one generates none. A negative-binomial would model overdispersion more faithfully when the empirical variance exceeds the mean, but it requires two parameters and a fit step that is awkward to do in a single-file client-side app ([ADR-0001](./0001-single-file-html-app.md), [ADR-0002](./0002-client-side-only.md)) without bringing in a stats library; the resulting forecast tightness is also bounded by the lognormal's right tail, so Poisson's underdispersion is in practice the smaller error.

We chose to sample *per initiative* rather than once at the scenario level (e.g. `numEpics = Poisson(K · λ)`) because the two are not equivalent for sums of independent samples: per-initiative sampling lets the epic-count variance flow through independently for each initiative, which is what matches reality and what makes the simulator's tail honest. Lumping epics into a single Poisson at scenario level would shrink the tail and produce dishonestly narrow forecasts.

The implementation switches between two samplers at `λ = 30`: Knuth's multiplicative method below (exact, but `O(λ)` per draw) and a Gaussian approximation above (`O(1)`, bias < 0.1%; see `samplePoisson`, `index.html:1319`). This is an internal optimisation; the *distribution* sampled from is Poisson in both regimes, and the switch point is not user-visible. A future revision could move to negative-binomial if we find a robust per-team fit; that would not change the per-initiative sampling discipline, only the distribution family.
