# JSON persistence for Groups via user-driven `groups.json` save / load

The **Groups tab** ([ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)) gains a toolbar with `↓ Save groups (JSON)` and `↑ Load groups (JSON)` buttons that read and write a single user-driven file — `groups.json` by default — carrying the entire `groupsStore: Group[]` array under a versioned wrapper. Schema: `{ schemaVersion: 1, groups: [{ name: string, color: string, members: (string | null)[], isProjection: boolean }, ...] }`. The **(Blank) sentinel** ([ADR-0028](./0028-category-as-generalized-moscow.md)) serialises as the JSON `null` literal — a token JSON has natively and that no user-typed Category string can produce. Save serialises `groupsStore` via `JSON.stringify(..., null, 2)`, wraps in a `Blob`, and triggers a download via the same `URL.createObjectURL` + anchor-click pattern the **Marker CSV** ([ADR-0025](./0025-per-context-marker-system.md)) already uses for `markers-${contextKey}.csv`. Load opens a `<input type="file" accept=".json">` picker, reads via `FileReader.readAsText`, parses, validates the wrapper shape and `schemaVersion`, and replaces `groupsStore` **wholesale** — every previously-defined Group is dropped. A confirmation modal interposes the wholesale-replace if `groupsStore` is non-trivial (more than just the auto-default `All` Group); the modal also surfaces on a parse error with the raw error string. The version policy is **read-permissive within major**: on load, `schemaVersion === 1` parses the known fields and *silently ignores any unknown fields* (forward-compat for additive evolution); `schemaVersion > 1` surfaces an inline error `"This file was saved by a newer version of the simulator."` and does not load; missing `schemaVersion` is treated as `1` for back-compat with hand-authored files. After a successful load, the loaded Groups land in `groupsStore` immediately but no chart re-renders until the user presses **Run Simulation** — matching the commit-on-Run pattern from [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) and [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md). Categories referenced in the loaded `members` arrays but absent from the currently-loaded **Initiatives CSV** are kept verbatim in the Group's member-set and silently match zero initiatives at Run time (the same lenient-validation rule as in-app group editing — [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md)).

We chose **JSON over CSV** for this persistence layer — the first non-CSV file format the simulator handles — rather than (a) a multi-row CSV with one row per Group and a delimited members column, (b) one CSV per Group with a separate index CSV, or (c) embedding Groups into the existing **Initiatives CSV** as additional columns, because the Group data shape is naturally *hierarchical*: a Group has a name and a colour and a *list of strings* (members) — and CSV does not natively express list values without a separator-escaping convention. The existing **Marker CSV** ([ADR-0025](./0025-per-context-marker-system.md)) already paid this cost — `label,value,color,is_capacity` with `label` JSON-encoded so commas and quotes survive — and adding a second collection-of-strings field would have doubled the encoding awkwardness. JSON parses natively in vanilla JS (`JSON.parse` / `JSON.stringify`) with no library dependency, handles nested structure trivially, and produces a file that humans can read and edit. A multi-row CSV would have forced the loader to reconstruct the `members[]` array from delimited substrings — and the moment a Category contains a comma or a pipe character, the round-trip breaks. One CSV per Group would have meant uploading N files for N Groups — friction with no upside. Embedding Groups into Initiatives CSV would have conflated two unrelated concerns: the Initiatives CSV is the *input data*, the Groups file is *user planning preferences* — they have different lifecycles, different owners, and should not live in the same file.

We chose to **persist Groups only** — not capacity, iterations, parameter mode, **Quarter selector** picks, **Markers**, or the editable initiatives — rather than (a) emit a single `simulator-state.json` snapshot covering the entire session, (b) bundle Groups + Markers into one file, or (c) defer the Groups file format pending a future "save full workspace" feature, because the user explicitly framed Phase 3 as "save the groups to a json file". Broadening the scope would have invited disagreement over which other surfaces should be co-persisted — sidebar inputs (capacity / iterations / parameter mode) are session-ephemeral by design ([ADR-0002](./0002-client-side-only.md)); Markers are already persisted per-context via the **Marker CSV** ([ADR-0025](./0025-per-context-marker-system.md)) and bundling them into the Groups file would have duplicated the persistence path; the editable initiatives already round-trip through `initiatives-edited.csv` ([ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md)) and should not be co-persisted because their lifecycle is tied to the **Initiatives CSV** upload, not to the Groups configuration. Keeping Groups in their own file gives the user a single-purpose artefact — "this is my planning convention" — that they can version-control, share with a colleague, or check into a project repo independently of the Initiatives CSV. The cost is that the user reloading the page must still re-upload the Initiatives CSV and re-set their sidebar inputs *and* re-load the Groups JSON — three separate gestures — but each artefact is independently meaningful and reusable.

We chose to **replace `groupsStore` wholesale on load**, with a confirmation modal interposed when the current store is non-trivial, rather than (a) additively merge loaded Groups into the existing store (dedup-by-name with numeric suffix on collision), (b) require the user to clear Groups before loading, or (c) load into a "preview" state the user explicitly accepts, because wholesale-replace is the simplest mental model — "load is what the file says, exactly". An additive merge introduces the question "is `Must` the same Group as `Must`?" — by name? by member-set? by colour? — and the answer is unclear; the user who actually wants to combine two Group sets is better served by loading one, manually adding the others, and saving the union as a new file. Requiring the user to clear Groups first adds a friction step that the confirmation modal already covers when needed. A preview state would introduce a new application mode ("loaded but not committed") that no other surface uses and that conflicts with the simulator's commit-on-Run pattern. The wholesale-replace matches the **Marker CSV** load semantics ([ADR-0025](./0025-per-context-marker-system.md)) verbatim ("Loading replaces `markerStore[contextKey]` *wholesale*") — uniform behaviour across the two persistence paths.

We chose to **serialise the (Blank) sentinel as the JSON `null` literal in `members`** rather than (a) a magic string like `"__BLANK__"`, (b) a structured object `{ blank: true }`, or (c) omitting the sentinel from the `members` array and tracking a separate `includeBlank: boolean` on the Group, because `null` is a token JSON natively supports, no user-typed Category string can be `null` (`JSON.parse('"null"')` produces the string `"null"`, distinct from `JSON.parse('null')`), and the serialiser / deserialiser logic is one-line each (`members.map(m => m === BLANK ? null : m)` and the reverse). A magic string would have introduced a one-day-someone-will-name-their-category-`__BLANK__` bug; a structured object would have broken the otherwise uniform `members: string[]` typing; a separate `includeBlank` field would have re-introduced the very asymmetry [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) explicitly removed (Blank is just another addressable member). The hand-edit experience is also clean — a user opening `groups.json` in a text editor sees `"members": ["Must", "Should", null]` and the `null` reads as "the missing-data bucket" without needing a glossary lookup.

We chose **`schemaVersion: 1` as an integer field with read-permissive evolution** within the major version (parse known fields, ignore unknown, treat missing as `1`), rather than (a) no version field at all relying on field-presence detection, (b) a semver string like `"1.0.0"`, or (c) strict-equality version matching that errors on any deviation, because the simulator is a small single-file app where the cost of carrying an integer version field is one line and the benefit is a clean error path when a future v2 introduces a breaking schema change. No version at all would have forced the loader to detect the file's vintage from which fields it carries — fragile, and the failure mode is silent. A semver string is overkill for a single-file app where there is no library dependency contract to manage; integer versions match the **Marker CSV** convention (which itself uses no version but has not yet needed one — Markers may eventually adopt the same `schemaVersion` field for symmetry, see future-revisions below). Strict-equality on every load would have forced a version bump for every additive change (e.g., adding a `description` field to a Group) — and a version bump on additive evolution forces every existing `groups.json` file the user has on disk to error on load. Read-permissive lets additive fields land without a version change; only genuinely breaking changes (rename of an existing field, change in semantics) bump the version and earn a migration shim in the loader.

This decision pairs with [ADR-0001](./0001-single-file-html-app.md) (the save / load implementation lives inline in `index.html` — `JSON.stringify`, `Blob`, `FileReader`, no library dependencies), [ADR-0002](./0002-client-side-only.md) (no backend; persistence is user-driven file download / upload; the file lives on the user's filesystem or in their cloud-sync folder of choice), [ADR-0025](./0025-per-context-marker-system.md) (the Marker CSV save / load pattern this echoes — same anchor-click download trigger, same `FileReader.readAsText` load, same wholesale-replace semantics), [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) (the commit-on-Run discipline that load semantics honour — loaded Groups land in `groupsStore` immediately but no chart re-renders until Run), [ADR-0028](./0028-category-as-generalized-moscow.md) (the Category strings that `members` reference), and [ADR-0029](./0029-user-defined-groups-supersede-cumulative-moscow.md) (the Group data model being persisted, and the lenient-validation rule that loaded categories absent from the current CSV are silently zero-K). The implementation adds two toolbar buttons to the Groups tab markup, two handlers (`saveGroupsJSON()` and the file-load `change` event handler), the version-validation logic at the top of the load handler, and the BLANK ↔ `null` mapping in both serialise and deserialise paths.

A future revision could **add a `description: string` field per Group** for free-text notes — additive, would not bump `schemaVersion` and would not re-open this ADR. A future revision could **add a `tags: string[]` field per Group** for cross-cutting categorisation — additive, would not bump `schemaVersion`. A future revision could **persist Groups in `localStorage`** alongside the explicit JSON save — would re-open [ADR-0002](./0002-client-side-only.md) jointly with this ADR because the "no implicit state across sessions" rule is shared. A future revision could **bundle Markers, Groups, and editable initiatives into a single `workspace.json`** — would re-open this ADR jointly with [ADR-0025](./0025-per-context-marker-system.md) and [ADR-0027](./0027-editable-initiatives-tab-with-csv-export.md) because the "each artefact persists independently" rule is load-bearing. A future revision could **introduce `schemaVersion: 2`** for a breaking change (e.g., renaming `members` to `categories`); the v2 loader would carry a one-shot `v1 → v2` migration that runs on `schemaVersion === 1` files — additive at the loader level, would not re-open this ADR (the read-permissive policy already anticipates this). A future revision could **support an additive-merge load mode** behind a toggle in the load dialog — would re-open this ADR because the "wholesale-replace" rule is the load-bearing decision here. A future revision could **drag-drop a `groups.json` file onto the Groups tab** as an alternative to the file-picker button — additive UI, would not re-open this ADR. A future revision could **migrate the Marker CSV to a `markers.json` with the same versioning policy** — would re-open [ADR-0025](./0025-per-context-marker-system.md) but not this ADR (it would simply gain a sibling).
