# Software Effort Simulator

A Monte Carlo simulator that, given a historical quarter's body of work and a target quarter's planned initiatives, projects the probability that delivery effort will fit within available team capacity.

## Language

### Work items

**Initiative**:
A unit of planned work that belongs to one **Quarter**, one **Team**, and one **Category**. Initiatives are sampled by *count* (Poisson) and *generate* zero or more Epics.
_Avoid_: project, ticket, story, feature (in the product-management sense).

**Epic**:
A child of an Initiative carrying a t-shirt size. The size is what the simulator actually samples to produce effort, via a lognormal distribution. An Epic without a parent Initiative is an *orphan epic* (currently unhandled).
_Avoid_: task, sub-task, work item.

**Initiative key**:
The Jira-style identifier (e.g. `INIT-123`) that links an Epic to its parent Initiative. Stored on the Epic as the link column, surfaced internally as `_initiative_key`.
_Avoid_: parent id, ticket id.

**Constant work**:
Deterministic, guaranteed work carrying a **Team**, a **Quarter**, and a **Category**, uploaded via the optional **Constant Work CSV** or authored/edited in the **Constant work tab**. Bypasses Monte Carlo sampling — its effort is computed from **T-shirt size** as the closed-form lognormal *mean* (`e^(μ + σ²/2)`) via `tshirtToPersonMonths` against the *active* parameter set (so it follows the **Synthetic parameters** ↔ **Empirical parameters** toggle — see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md)). As of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) it is folded in **per Group**: for each **Group**, the effort of constant-work rows whose **Category** ∈ `group.members` (compared `trim()` + case-insensitively) is summed and added as *that Group's* deterministic shift to every **Iteration**'s output *after* the Monte Carlo sort — replacing the single global `fixedEffort` scalar [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md) applied uniformly to every **Scenario**. Constant work never contributes to **K** or **Poisson λ**. The Category-gated rule holds on all three surfaces: the org-level **Run** sums (Category ∈ members) across *all* teams in the **Target quarter(s)**; the **Team Level tab** additionally filters by team; the **Quick projection Monte Carlo** scopes each (team, quarter) cell to the **Projection group**'s members. Constant work whose Category is in *no* Group is **excluded** from the simulation but surfaced in the **Data preview** (never silently dropped); **(Blank)**-categorised constant work follows the **(Blank) sentinel** rule. Surfaces as appended soft-green-tinted rows at the bottom of each **Projection section**'s **Initiative matrix** with `[<size> · ~<PM> PM]` annotated in the name cell (now filtered to the Projection group's Categories).
_Avoid_: fixed work, baseline work.

### Sizing and effort

**T-shirt size**:
The categorical effort label on an Epic — one of `2XS`, `XS`, `S`, `M`, `L`, `XL`, `XL+`. Each size maps to a lognormal distribution over person-months. The seven sizes are listed in their canonical order (smallest band on top); `2XS` was added later as a downward extension of the original six-size set — see [ADR-0024](docs/adr/0024-2xs-t-shirt-size-extension.md).
_Avoid_: estimate, story points, complexity.

**Person-month (PM)**:
The unit of effort throughout the system: one engineer working for one calendar month. All capacities, samples, and statistics are expressed in PM.
_Avoid_: man-month, FTE-month, sprint-points.

**Synthetic parameters**:
The documented-baseline lognormal parameter set, fit to the documented P10/P90 of each **T-shirt size** band via the closed-form `μ = (ln min + ln max) / 2`, `σ = (ln max − ln min) / (2 · Φ⁻¹(0.9))` formula — see [ADR-0007](docs/adr/0007-lognormal-effort-distribution.md). Lives at `T_SHIRT_PARAMS` (`index.html:1229`). The user-facing **T-shirt size reference** panel mirrors *this* table's bands and does not re-render when the **Empirical parameters** ↔ **Synthetic parameters** radio flips — see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md). No longer the page-load default as of [ADR-0035](docs/adr/0035-default-to-empirical-lognormal-parameters.md) — now the one-click alternative to the **Empirical parameters**, selectable via the **Lognormal Parameters** radio (`<input name="param-mode" value="synthetic">`).
_Avoid_: theoretical, default, calibrated.

**Empirical parameters**:
An alternative parameter set, bias-corrected from realised effort in Q1 2026 (`n = 36` epics across `XS`/`S`/`M`/`L`). Calibration preserves the synthetic `σ` per size and shifts `μ` by `ln(avg_ratio)`, where `avg_ratio = mean(actual / synthetic_mean)` per size; sizes lacking Q1 data (`2XS`, `XL`, `XL+`) carry the synthetic `(μ, σ)` through unchanged so `Object.keys(T_SHIRT_PARAMS_EMPIRICAL)` matches `Object.keys(T_SHIRT_PARAMS)` exactly — see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md). Lives at `T_SHIRT_PARAMS_EMPIRICAL` (`index.html:1253-1261`). Selected via the **Lognormal Parameters** sidebar radio (`<input name="param-mode" value="empirical">`); the `change` handler (`index.html:3293-3300`) reassigns the module-scoped `activeParams` reference (`index.html:1264`) so every sampler (`sampleLognormal`, `tshirtToPersonMonths`) reads through the new table on the *next* call. The **default** value of `activeParams` on every page-load as of [ADR-0035](docs/adr/0035-default-to-empirical-lognormal-parameters.md). Swapping is global, ephemeral (no `localStorage`, reset to **Empirical parameters** on every reload), and does *not* re-run the simulation or repaint any pre-Run surface — see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md) and [ADR-0035](docs/adr/0035-default-to-empirical-lognormal-parameters.md).
_Avoid_: actual, real-world, fitted.

**Poisson λ**:
The mean of the Poisson distribution from which each **Initiative**'s epic count is drawn. Fitted in `prepareSimulationData` as the average count of in-scope **Epics** per in-scope **Initiative** over the **Historical quarter**. One scalar per Run at the org level; one per team in the Team Level tab.
_Avoid_: rate, mean, average epics.

**Bootstrap pool**:
The flat array of historical **T-shirt size** labels — one entry per in-scope **Epic** in the **Historical quarter** — that each **Iteration**'s epic sizes are drawn from uniformly with replacement. Size labels that are not a **Recognised t-shirt size** are excluded from the pool.
_Avoid_: empirical distribution, sample pool, size population.

### Organisational unit

**Team**:
The owning unit of an **Initiative**, drawn from the initiative's team column (the **Sensible format** uses `teams`, the **Quirky format** uses the value-detected variant). Within `prepareTeamSimulationData` (`index.html:1802`), teams are deduplicated case-insensitively with first-seen casing preserved and sorted alphabetically case-insensitively. The **Team Level tab** renders one section per team present in the selected **Target quarter(s)**.
_Avoid_: squad, group, tribe, owner.

### Planning vocabulary

**Quarter**:
A label like `Q3 2026`. The unit of time the simulator works in. Initiatives and Epics each carry a quarter.
_Avoid_: sprint, period, cycle.

**Historical quarter**:
The quarter (or set of quarters) used to fit the model: Poisson λ over initiative count and the bootstrap pool of t-shirt sizes. Selected via the **Quarter selector** `#hist-ms` in the sidebar.
_Avoid_: source, baseline, reference quarter.

**Target quarter**:
The quarter (or set of quarters) being forecast. Initiatives in the target quarter are bucketed by **Category** and counted as `K` for each **Scenario** — one count per **Group** in `groupsStore`; **Constant work** in the target quarter is summed per Group as a deterministic shift ([ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)). Selected via the **Quarter selector** `#target-ms` in the sidebar, whose options — as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) — include quarters present only in the **Constant Work CSV**.
_Avoid_: forecast quarter, projected quarter, future quarter.

**Quarter selector**:
The sidebar multi-select widget the user picks **Historical quarter**(s) or **Target quarter**(s) with — a custom-built combo box (`MultiSelect` class, `index.html:1083`) showing a chip strip plus a checkbox dropdown. The simulator instantiates two: `#hist-ms` (Historical) and `#target-ms` (Target). `#hist-ms` is populated by `refreshQuarters` from the chronologically-sorted union of quarter labels present in the loaded **Initiatives CSV** and **Epics CSV** (`extractQuarters`); as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) `#target-ms` *additionally* includes quarters present in `editedConstantWork`, so the two selectors deliberately diverge in source — constant work is forward-looking and cannot inform the historical fit (**Poisson λ**, **Bootstrap pool**). Read via `histMS.getSelected()` / `targetMS.getSelected()`, both returning `string[]`. Every checkbox toggle and every chip-`✕` click dispatches a single bubbling `ms-change` `CustomEvent` on the wrapper; the canonical observation contract — see [ADR-0017](docs/adr/0017-multi-quarter-selectors.md). Selecting zero quarters is allowed at the widget level; downstream consumers (the **Data preview**'s guard, the run-button handler's gate) decide whether to act on the empty selection.
_Avoid_: quarter dropdown, quarter picker, multi-select, combo box.

**Category**:
The free-form label on an **Initiative** or a **Constant work** row, carrying whichever bucketing convention the user's planning uses — `Must`/`Should`/`Could`/`Won't` for legacy MoSCoW, `KR1`/`KR2`/`KR3` for key-result-driven planning, `Automation`/`Project ABC` for project-driven planning, or any arbitrary string. Stored in the canonical CSV header `category` and detected by `detectCategoryCol` via a header-name cascade `category` → `moscow` → `emoji` (no **Content scan** branch — see [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)). Comparison between two Category values is `trim() + case-insensitive`, with first-seen casing preserved for display (same convention as the team-name dedup in `prepareTeamSimulationData`). Empty / whitespace-only values normalise to the **(Blank) sentinel**. As of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md), the Category set offered in the **Groups tab** Members popover and the **Constant work tab** editors is the case-insensitive **union** of Categories observed across `editedInitiatives` and `editedConstantWork`; a Category present in *both* sources is **merged** into one entry (Initiative casing wins; a constant-work-only Category keeps its own casing). Categories carry no colour and no inherent ordering — colour is carried by the **Group** the user chooses to put the Category in ([ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)). Inclusion of a Category-labelled **Initiative** in a **Scenario** is purely a function of **Group** membership; there is no hardcoded `Won't` or `unknown` exclusion ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)).
_Avoid_: priority, tier, importance, bucket, label, tag.

**MoSCoW**:
A specific convention of **Category** values — the strings `Must`, `Should`, `Could`, `Won't` — historically the only categorisation the simulator supported. With [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md), MoSCoW is no longer a first-class engine concept; it is one possible Category vocabulary among arbitrarily many. Legacy MoSCoW CSVs continue to load (the header `moscow` is the second entry in the `detectCategoryCol` cascade), and the user can recover the legacy cumulative three-scenario behaviour by defining three overlapping **Groups** `{Must}`, `{Must, Should}`, `{Must, Should, Could}` ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)).
_Avoid_: priority (which now means Category in general).

**(Blank) sentinel**:
The reserved value the **Category** column resolves to when an **Initiative**'s category field is empty, missing, or whitespace-only. Distinct from any user-typed string (including the literal string `"Blank"`) because the UI surfaces it as `(Blank)` *with parentheses*. In a **Group**'s `members` array the sentinel appears alongside Category strings; in the JSON persistence layer ([ADR-0030](docs/adr/0030-json-persistence-for-groups.md)) it serialises as the JSON `null` literal — a token no Category string can produce. The (Blank) sentinel is just another addressable member; whether (Blank)-categorised initiatives participate in a Run is decided by **Group** membership exactly like any other Category ([ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)).
_Avoid_: blank, empty, missing, unknown, null.

**Key Result**:
An optional free-text label on an **Initiative** linking it to the strategic outcome it serves (`KR-7`, `Activation +10% YoY`, etc.). Detected by `detectKrCol` (`index.html:1436-1450`) from a small canonical vocabulary of header variants — `key_result`, `kr`, `key result`, `keyresult`, `key_results` — first by exact match, then by case-insensitive fallback returning the *raw-case* header. The detector returns `null` when no variant is present, which is the *only* `null`-able key in `detectedCols`. Display-only: never read by the simulation engine; surfaced only as a conditional `KR` column in the **Initiative matrix** inside each **Projection section**, gated per-section by `hasKr = allInits.some(i => i.kr)` so that teams whose initiatives carry no KRs render the matrix without the column. **Constant work** rows participate in the same gate and read their own KR via a parallel `r.key_result || r.KR || r.kr` lookup inside `getConstantWorkEpics` (the **Constant Work CSV** has a fixed schema and does not go through the detector family) — see [ADR-0022](docs/adr/0022-optional-key-result-column.md).
_Avoid_: OKR, outcome, objective, KR label.

**Scenario**:
One of the `|groups|` forecasts the simulator runs side-by-side in a single **Run**, defined by exactly one **Group** — `K = #initiatives in editedInitiatives whose Category ∈ group.members`, plus that Group's deterministic **Constant work** shift (the summed effort of `editedConstantWork` rows whose Category ∈ `group.members`, scoped by target quarter and — at team/projection scope — team; see [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)). Each Scenario shares the per-Run **Poisson λ**, **Bootstrap pool**, **Capacity**, and PRNG seed with every other Scenario in the same Run ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)). Historically (pre-[ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)) the three Scenarios were hardcoded as cumulative MoSCoW unions; that shape is now reproducible by the user defining three overlapping Groups.
_Avoid_: case, projection, model.

**Group**:
A user-defined named subset of **Category** values that determines exactly one **Scenario** in a **Run** — selecting both which **Initiatives** count toward the Scenario's `K` and (as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)) which **Constant work** effort folds into it. Each Group carries `{ name: string, color: string, members: (string | BlankSentinel)[], isProjection: boolean }` — a flat, independent set (not nested-by-construction; not cumulative; overlap with other Groups is allowed and is exactly how cumulative semantics is recovered). Lives in module-scoped `groupsStore: Group[]`; the *array order* of `groupsStore` is the user-controlled **Group order** (see below) and is itself part of the model — it drives the left-to-right dataset draw order in every chart and the left-to-right column order in every stats table. Created / edited / duplicated / removed / *reordered* via the **Groups tab**; the chip-strip + **MultiSelect** popover populates the Members cell from observed Categories plus a `(Blank)` row plus a free-text input. Edits that *can change a Scenario's distribution* (name, color, members, `isProjection`, add/delete/duplicate) commit to `groupsStore` immediately but charts and stats lag until the next **Run Simulation** press — same commit-on-Run contract as the **Initiatives tab** ([ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md), [ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)). The single exception is **Group order** (the array-index permutation): a drag-reorder re-paints the org-level chart legend, the **Summary Statistics** columns, and every cached **Team Level tab** chart immediately, with no resimulation — see [ADR-0032](docs/adr/0032-drag-to-reorder-groups.md). Validation is **lenient** — empty / duplicate names, zero-member Groups, and references to categories absent from the loaded CSV are all allowed; the only strict rule is exactly-one `isProjection`, enforced via single-select radio in the Groups tab. Soft cap ~5 (not enforced); above that, the overlapping-datasets chart and N-column stats table degrade in legibility as a self-evident signal. On first load with no JSON yet, a single auto-default Group named `All` is created, with members = every unique Category across both `editedInitiatives` and `editedConstantWork` including (Blank), flagged `isProjection` — so a fresh import captures all constant work by default ([ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)).
_Avoid_: bucket (which is internal pre-[ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md) vocabulary), scenario (which is the *result* of running a Group through the engine), set, partition.

**Group order**:
The array-index permutation of `groupsStore: Group[]` — top-to-bottom in the **Groups tab** equals left-to-right in every chart legend and stats-table column. User-controlled via a **Drag handle** (the `⋮⋮` glyph nested in the Name cell of each Group row) that makes the `<tr>` `draggable="true"`; an above/below Y-midpoint drop test selects the insertion edge on the hovered row. On drop, `reorderGroup(fromIdx, beforeIdx)` splices `groupsStore` into the new order *and* applies the same index permutation in lockstep to every cached per-Group array — `lastTeamData[*].kPerGroup` and `lastRenderState[*].results.results` — preserving the invariant *"index `i` in `groupsStore` is index `i` in every per-Group array of the same Run"*. Every cached `lastRenderState` context (`org` plus every rendered `team-{idx}`) is then re-painted in place via `rerenderContext`; no **Iteration** is re-sampled, no PRNG re-seed happens. This is the *only* Groups-tab interaction that re-paints immediately — see [ADR-0032](docs/adr/0032-drag-to-reorder-groups.md). The **Team Projections tab** is deliberately *not* re-painted on reorder (its `projData` is not cached today); it catches up on the next **Run Simulation** press. Persistence: the order survives **Groups JSON** save / load round-trip with no schema change, because the wrapper already serialises Groups as an ordered JSON array ([ADR-0030](docs/adr/0030-json-persistence-for-groups.md)).
_Avoid_: row order, group sort, group ranking.

**Projection group**:
The single **Group** in `groupsStore` with `isProjection === true` — exactly one at any time. Drives the per-(Team, Quarter) **Quick projection Monte Carlo** ([ADR-0020](docs/adr/0020-team-projections-cross-quarter-view.md)) and the **Effort projection band** in every **Projection section**; as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) its members also select which **Constant work** folds into each projection cell. The selection is surfaced as a single-select radio column in the **Groups tab**; deleting the current projection Group transfers the flag to the next-available Group. If `groupsStore` is empty (no Groups at all), the projection band falls back to the constant-work-only triple `(cwEffort, cwEffort, cwEffort)` — the same fallback [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md) already specifies.
_Avoid_: default group, primary group, MSC scenario.

**Capacity**:
The PM budget the team commits to deliver in the target quarter. Configured via the sidebar (`#capacity`), default 120 PM. Every scenario's risk is reported as `P(effort > capacity)`. Rendered on every chart as an auto-managed **Marker** (red dashed vertical line, label `Capacity: {value} PM`) via `ensureCapacityMarker` — see [ADR-0015](docs/adr/0015-capacity-as-auto-managed-chart-marker.md) and [ADR-0025](docs/adr/0025-per-context-marker-system.md); the user can recolour or relabel the line from the **Marker dialog**, but cannot delete it, and its `value` is overwritten from the sidebar input on every **Run**.
_Avoid_: budget, headcount, throughput.

**Iteration**:
One draw of the Monte Carlo loop: sample `numEpics ~ Poisson(λ)` for each of the `K` initiatives, sample each epic's effort from `Lognormal(μ, σ)` by t-shirt size, sum to a total. The iteration count is user-configured via `#iterations` (default 10,000 in the HTML, 1,000,000 if the field is empty when **Run** is pressed), clamped to `[1000, 10000000]`.
_Avoid_: trial, sample, run (which means the whole batch).

**Run**:
A single press of `Run Simulation`. Re-seeds the PRNG and executes `iteration` × `|groups|` **Scenarios** — one Scenario per **Group** in `groupsStore` ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)). Historically (pre-[ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)) the multiplier was hardcoded `× 3`.
_Avoid_: simulation, batch.

**Quick projection Monte Carlo**:
The per-(**Team**, **Quarter**) Monte Carlo run inside `buildTeamProjections` that produces the **Effort projection band** for the **Team Projections tab**. Uses the same `runSimulation` engine as the headline Run but with four differences: (a) only the **Projection group**'s **Scenario** is read out (P25/P50/P75), (b) the iteration count is *capped* at `projIterations = Math.min(iters, 3000)` (set once in the run-button handler), (c) **Capacity** is passed as `0` so no risk-gate row is computed and no marker is drawn, and (d) **Constant work** is folded in deterministically as `fixedEffort = cwEffort` — the per-team-per-quarter sum, scoped as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) to constant work whose **Category** ∈ the **Projection group**'s members — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md). Reads the org-wide **Poisson λ** and **Bootstrap pool** verbatim — never the team-scoped ones — see [ADR-0020](docs/adr/0020-team-projections-cross-quarter-view.md). Skipped (band defaults to the flat triple `(cwEffort, cwEffort, cwEffort)`) when the team-quarter cell has `K === 0` for the **Projection group**, when `orgLambda === 0`, when `orgEpicSizingDist.length === 0`, or when `groupsStore` is empty.
_Avoid_: projection run, mini simulation, quick MC.

### Inputs

**Initiatives CSV**:
A user-supplied file listing one row per Initiative. Carries the Jira key, name, **Category**, team, quarter, and optionally a **Key Result**. Required. The **Category** column is detected by `detectCategoryCol` via a header-name cascade — `category` → `moscow` → `emoji` — see [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md). The optional **Key Result** column is detected by `detectKrCol` from a fixed vocabulary of header variants — see [ADR-0022](docs/adr/0022-optional-key-result-column.md). Parsed once into the immutable `parsedInitiatives` and *immediately* shallow-cloned per-row into `editedInitiatives` — the latter is the **simulation source of truth** that `prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`, and the projection-axis `extractQuarters` call all read from, so user edits made on the **Initiatives tab** flow into the next **Run** automatically — see [ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md). `parsedInitiatives` is preserved only as the *dropdown-option enumeration substrate* (`getUniqueColumnValues(parsedInitiatives)`); migrating that read to `editedInitiatives` would collapse the option pool to whatever the user had typed last — see [ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md). The Initiatives-tab category cell editor is an HTML5 `<input list="category-options">` datalist combo seeded from `getUniqueColumnValues`, accepting both pick-from-existing and type-new entries.
_Avoid_: initiative file, input file.

**Epics CSV**:
A user-supplied file listing one row per Epic. Carries the t-shirt size, the parent initiative key, and a quarter. Required.
_Avoid_: epic file, sizing file.

**Constant Work CSV**:
An optional third CSV describing constant-work epics (see **Constant work**). Hand-authored to a *fixed* schema — `jira_key | epic_key`, `epic_name | building_block`, `key_result | KR | kr`, `category | moscow | emoji`, `team`, `quarter`, `tshirt_size | t_shirt_size` — read inline by `getConstantWorkEffort` and `getConstantWorkEpics`; the category column is resolved by the same `category` → `moscow` → `emoji` cascade as the **Initiatives CSV** ([ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)) but inline-coded against the fixed schema (no `detectXxxCol` invocation). Parsed once into the module-scoped `parsedConstantWork` (three observable states: `null` no CSV ever loaded, `[]` loaded but empty, `RowObject[]` loaded) and *immediately* shallow-cloned per row into `editedConstantWork`, the **simulation source of truth** the consumer helpers read from — `parsedConstantWork` is retained only as the **Constant work tab**'s dropdown/datalist option substrate, mirroring `parsedInitiatives` ([ADR-0034](docs/adr/0034-editable-constant-work-tab.md)). Rows are matched against the team-name column case-insensitively and against the quarter column exactly (trimmed). Constant-work entries do *not* participate in **K** counts for any **Group** — but as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) their **Category** *does* determine which Group(s) their deterministic shift folds into (it is no longer display-only); the effort is still added as a per-Group shift after the Monte Carlo sort ([ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md)). Unrecognised `tshirt_size` values silently contribute `0` PM via `tshirtToPersonMonths`'s miss branch — no warning, no error — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md).
_Avoid_: fixed-work file.

**Sensible format**:
The CSV layout where headers match their semantics: `jira_key`, `building_block`, `category` (canonical; legacy `moscow` still detected — see [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)), `teams`, `quarter`. The recommended layout going forward. The `building_block` and `teams` headers are read by `detectNameCol` and `detectTeamCol` via a *direct header-name match* placed *before* the positional fallback (see [ADR-0021](docs/adr/0021-sensible-csv-format-dual-support.md)); the `jira_key` header is picked up by the **Content scan** in the normal case and by the **Detection fallback** in degenerate-data cases; the `category` header is the first entry in the `detectCategoryCol` cascade and matches before `moscow` or `emoji`.
_Avoid_: new format, target format, modern format.

**Quirky format**:
The legacy CSV layout exported by an older internal tooling, where `teams` actually held Jira keys and `emoji` actually held MoSCoW priority (a specific **Category** convention). Still parseable: `detectInitKeyCol` continues to use **Content scan** to find the Jira-key column, the positional fallbacks inside `detectNameCol` / `detectTeamCol` read the header immediately before `initKeyCol` and `categoryCol`, and `detectCategoryCol`'s header cascade (`category` → `moscow` → `emoji`) picks up the `emoji` header as its third entry. Both **Sensible format** and **Quirky format** remain first-class inputs — see [ADR-0021](docs/adr/0021-sensible-csv-format-dual-support.md) and [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md). Note: with [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md) the Content-scan branch is no longer used for the category column — header matching is the only path.
_Avoid_: old format, legacy format (acceptable in prose, but the canonical term is **Quirky format**).

**Groups JSON**:
The user-driven persistence file for `groupsStore`, named `groups.json` by default. Saved via the `↓ Save groups (JSON)` button on the **Groups tab** and loaded via `↑ Load groups (JSON)` — both implemented inline using `JSON.stringify` / `JSON.parse`, `Blob` + `URL.createObjectURL` for save, and `<input type="file" accept=".json">` + `FileReader.readAsText` for load. Fixed schema: `{ schemaVersion: 1, groups: [{ name: string, color: string, members: (string | null)[], isProjection: boolean }, ...] }` — the **(Blank) sentinel** serialises as the JSON `null` literal. Load semantics are **wholesale replace** with a confirmation modal when `groupsStore` is non-trivial (more than just the auto-default `All` Group); referenced categories absent from the currently-loaded **Initiatives CSV** are kept verbatim in members and silently match zero initiatives. Version policy is **read-permissive within major**: `schemaVersion === 1` parses known fields and ignores unknown for forward-compat; `schemaVersion > 1` is rejected with an inline error; missing `schemaVersion` is treated as `1` — see [ADR-0030](docs/adr/0030-json-persistence-for-groups.md).
_Avoid_: groups file, scenarios file, planning file.

### Summary statistics

**Stats**:
The per-**Scenario** tuple `{ p10, p25, p50, p75, p90, mean, pExceed }` produced by `computeStats` from a sorted `Float64Array` of per-**Iteration** totals. Exactly one Stats tuple per Scenario per **Run** — `|groups|` per Run at the org level ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)). Read by the stats table at `#stats-table` and by any per-team or per-**Marker** row that displays a percentile or a **Probability of exceedance**.
_Avoid_: summary, metrics, KPIs.

**Percentile (Pxx)**:
The value at fractional rank `xx/100` in the sorted distribution, computed as `sorted[min(n − 1, floor(p · n))]`. Every reported percentile is an *actual realised Iteration* (not an interpolated value). The simulator reports P10, P25, P50, P75, and P90 — see [ADR-0012](docs/adr/0012-percentile-summary-and-probability-of-exceedance.md). P50 is the **Median**.
_Avoid_: quantile, percentile estimate, interpolated percentile.

**Tail percentile**:
P75 and P90 specifically — the *upper* points used to read overrun risk. The simulator deliberately does *not* report P95 or P99: at the default 10,000 **Iterations** the latter are dominated by Monte Carlo noise and are the territory custom **Markers** are designed for.
_Avoid_: extreme percentile, upper bound.

**Probability of exceedance**:
The fraction of **Iterations** whose total effort *strictly* exceeds a threshold, in `[0, 1]`. The headline cell in the stats table is `P(effort > capacity)`, but the same metric is computed against every non-capacity **Marker** value. Computed by a single binary search over the sorted `Float64Array` via `computePExceed` (or the inline form inside `computeStats`).
_Avoid_: overrun probability, breach probability, P(breach), risk score.

**Risk tier**:
The colour class applied to a **Probability of exceedance** cell in the stats table: `ok` (green) when `pExceed ≤ 0.25`, `caution` (orange) when `0.25 < pExceed ≤ 0.5`, `warn` (red) when `pExceed > 0.5`. Hard-coded; see [ADR-0013](docs/adr/0013-three-tier-risk-colouring.md). Applies identically to the capacity row and to per-**Marker** rows.
_Avoid_: risk level, severity, status, RAG.

### Result tabs

**Tab**:
One of six named views of a **Run**'s output: `Organization Level`, **Team Level tab**, **Team Projections tab**, **Initiatives tab**, **Groups tab**, **Constant work tab**. The tab bar holds one `.tab-btn` per tab; exactly one carries the `.active` class at any time. After every Run, the active tab resets to `Organization Level` — see [ADR-0018](docs/adr/0018-tab-based-results-layout.md). The **Initiatives tab** ([ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md)), the **Groups tab** ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)), and the **Constant work tab** ([ADR-0034](docs/adr/0034-editable-constant-work-tab.md)) are the three tabs that mutate simulator state on user interaction (via inline edit handlers); the other three are read-only.
_Avoid_: view, panel (which is the container the tab points to), section (which is the per-**Team** block inside the Team Level tab or **Projection section** inside the Team Projections tab).

**Tab panel**:
The container `<div class="tab-panel">` that holds a Tab's content. Identified by `id="tab-${dataTab}"`; exactly one panel has `display: flex` at a time, the others are `display: none`. Pre-rendered during the Run, not lazy-rendered on tab switch.
_Avoid_: view, page, screen.

**Team Level tab**:
The second **Tab**, `#tab-teams`. Renders one section per **Team** in the selected **Target quarter(s)**, with the team's title row, a `Historical data` radio toggle, a chart card, and a Stats table. Each section runs an *independent* Monte Carlo via `runSimulation`, with K values always team-scoped and **Poisson λ** + **Bootstrap pool** toggleable between team-scoped and org-wide — see [ADR-0019](docs/adr/0019-per-team-independent-simulations.md). Section ordering is the alphabetical case-insensitive sort produced by `prepareTeamSimulationData`.
_Avoid_: per-team tab, teams view, drilldown.

**Historical data toggle**:
The per-section radio pair on the **Team Level tab** that selects which historical parameters to feed the section's Run: `This team only` (uses `teamLambda` + `teamEpicSizingDist`) or `All teams — org-wide` (uses the org-level `lambda` + `epicSizingDist` carried by the same Run). Defaults to *org-wide* when the team has 4 or fewer historical **Initiatives** (`useOrgByDefault === histInitCount <= 4`), surfaced via a yellow `Recommended: only N historical initiatives found for this team` chip. Toggling re-runs *only that team's* Monte Carlo via `renderTeamSection(idx, useOrg)`; the other sections, the org tab, and per-section **Markers** are untouched. K values are *not* toggleable — only the historical scope is.
_Avoid_: source selector, parameter switch, scope toggle.

**Team Projections tab**:
The third **Tab**, `#tab-projections`. A *cross-quarter*, *per-team* view: a single **Summary table** of **Initiative** counts per **Team** per **Quarter** followed by one **Projection section** per team appearing anywhere in the loaded **Initiatives CSV**. The team list is bounded by the *entire CSV*, never by the **Quarter selector**'s **Historical quarter(s)** or **Target quarter(s)** selection — see [ADR-0020](docs/adr/0020-team-projections-cross-quarter-view.md). Built by `buildTeamProjections` (`index.html:1917`) and `renderTeamProjections` (`index.html:2550`); the team-list dedup is the same case-insensitive rule with first-seen casing as the **Team Level tab**. Unlike the Team Level tab, this tab does *not* surface a **Historical data toggle** — every projection runs with the org-wide **Poisson λ** and **Bootstrap pool**.
_Avoid_: projections view, cross-quarter view, forecast tab.

**Projection section**:
The per-**Team** block inside the **Team Projections tab** (`.proj-team-section`). Contains the team's title, a `proj-charts-row` holding two charts side-by-side (the stacked-bar `Initiatives by Quarter` count chart and the `Effort Projection by Quarter` P25/P50/P75 bar chart), and a `.proj-init-table` **Initiative matrix** with a per-quarter **Category** badge column per initiative plus footer rows for the per-quarter cumulative count chips and the **Effort projection band**. Sections are appended to `#proj-teams-container` in alphabetical case-insensitive order. Each section's quarter axis (`displayQuarters`) is *always* the full `allQuarters` from `extractQuarters(editedInitiatives)`, regardless of which quarters that team has data in — see [ADR-0020](docs/adr/0020-team-projections-cross-quarter-view.md).
_Avoid_: team projection card, projection panel, drilldown (which is the **Team Level tab**'s per-team block).

**Initiatives tab**:
The fourth **Tab**, `#tab-initiatives`. Renders every row of `editedInitiatives` as a wide horizontally-scrolling table with one column per CSV header (except the hidden columns `jira_link` and `emoji`, which are still serialised on export) and one `↓ Export CSV` toolbar button. Each cell is one of three kinds, decided per-render by `renderInitiativesTable`: (a) **non-editable plain text** for the detected **Initiative key** column and the detected name column (`nonEditable = new Set([detectedCols.initKeyCol, detectedCols.nameCol].filter(Boolean))`) — protecting the **Initiative key**'s join contract with the **Epics CSV**; (b) **free-form `<input type="number" step="any">`** for the hardcoded `numericCols` set `{'added_value_impact', 'added_value_cost_saving'}` — those two value-impact fields are continuous-numeric in spirit, not categorical; (c) for the detected **Category** column an **HTML5 `<input list>` datalist combo** ([ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)) seeded from `getUniqueColumnValues(parsedInitiatives)` — accepts both pick-from-existing and type-new entries; (d) for every other column a **`<select>` dropdown** of unique observed values, where the option list comes from `getUniqueColumnValues(parsedInitiatives)` (the *immutable* parsed array, so the pool stays stable across edits — see [ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md)) and always includes the row's current value at the head when absent from the unique set. Inline `onchange` (and `oninput` for numeric inputs) handlers write `this.value` — always a string — directly into `editedInitiatives[rowIdx][col]`. Edits are **commit-on-Run**: they land in `editedInitiatives` immediately but the other tabs' charts and stats lag until the user presses **Run Simulation**. The tab is pre-rendered as part of every Run; the table is rebuilt wholesale via a single `innerHTML` assignment. The `↓ Export CSV` button calls `exportInitiativesCSV` which serialises `editedInitiatives` verbatim via `Papa.unparse` and downloads `initiatives-edited.csv` — the export is round-trip stable (re-uploading produces the same `detectedCols` and the same Run output as the original-plus-edits). Empty `editedInitiatives` renders the inline `No initiatives loaded.` message in place of the toolbar+table — see [ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md).
_Avoid_: editable initiatives view, initiatives table, initiative editor.

**Groups tab**:
The fifth **Tab**, `#tab-groups`. Renders one editable row per **Group** in `groupsStore` with columns `[Name | Color | Members | Projection | Duplicate | Delete]` plus a `+ New group` row at the bottom and a toolbar carrying `↓ Save groups (JSON)` and `↑ Load groups (JSON)` ([ADR-0030](docs/adr/0030-json-persistence-for-groups.md)). Every Group row is `draggable="true"` and carries a **Drag handle** (`⋮⋮`) nested in the Name cell; dragging a row sets the **Group order**, which propagates immediately to the chart legend, the **Summary Statistics** columns, and every cached **Team Level tab** chart with no resimulation — see [ADR-0032](docs/adr/0032-drag-to-reorder-groups.md). The trailing `+ New group` row is *not* draggable. The Name cell is a free-text `<input>` (alongside the drag handle); the Color cell opens the same 80-colour palette swatch (`COLOR_PALETTE`) the **Marker dialog** uses; the Members cell is a horizontal chip strip with `×` per chip and a `+` button that opens a **MultiSelect** popover ([ADR-0017](docs/adr/0017-multi-quarter-selectors.md)) listing every observed **Category** in the **union** of `editedInitiatives` and `editedConstantWork` ([ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)) plus a `(Blank)` row plus a free-text input for adding categories not yet in either source; the Projection cell is a single-select radio enforcing exactly-one `isProjection` across the table; the Duplicate button clones the row with name `"<original> (copy)"`; the Delete button removes the row (and, if it was the projection Group, transfers the flag to the next-available row). Edits that *can change a Scenario's distribution* (Name, Color, Members, Projection, Duplicate, Delete, `+ New group`) commit to `groupsStore` immediately but charts and stats lag until the next **Run Simulation** — same commit-on-Run contract as the **Initiatives tab** ([ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md), [ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)). **Group order** is the documented exception (re-paints immediately — see above). Validation is lenient — empty names, duplicate names, zero-member Groups, and references to categories absent from the loaded CSV are all allowed. Pre-rendered as part of every Run. On first load with no JSON yet, a single auto-default Group named `All` is created with members = every unique Category across both `editedInitiatives` and `editedConstantWork` including the **(Blank) sentinel**, flagged `isProjection` ([ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)).
_Avoid_: scenarios tab, planning tab, group editor.

**Constant work tab**:
The sixth **Tab**, `#tab-constant-work`. Renders every row of `editedConstantWork` as an editable table mirroring the **Initiatives tab** ([ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md)) but with three deliberate differences ([ADR-0034](docs/adr/0034-editable-constant-work-tab.md)): (a) *all* cells are editable — there is no **Epics CSV** join to protect, so even the key and name cells accept input; (b) the tab carries a `+ Add row` control and a per-row delete (the Initiatives tab has neither) and is **always present** even with no **Constant Work CSV** loaded, so a user can author constant work from scratch — an added row gets the canonical schema (`jira_key, epic_name, key_result, category, team, quarter, tshirt_size`) when there is nothing to mirror, else the imported header set; (c) `tshirt_size` is a `<select>` constrained to the seven **Recognised t-shirt sizes** (closing the silent-`0`-PM footgun), `category` / `team` / `quarter` are `<input list>` datalist combos seeded from the observed **union** across `editedInitiatives` and `editedConstantWork`, and `jira_key` / `epic_name` / `key_result` are free-text inputs. Edits, additions, and deletions are **commit-on-Run** — they land in `editedConstantWork` immediately but charts and stats lag until the next **Run Simulation**, the same discipline as the **Initiatives tab** and **Groups tab**. New rows are blank and validation is lenient (a blank size → `0` PM, a blank quarter → excluded from every simulation, a blank Category → the **(Blank) sentinel**); delete is immediate, no confirmation. The `↓ Export CSV` button serialises `editedConstantWork` via `Papa.unparse` to `constant-work-edited.csv`, preserving the imported headers verbatim (canonical schema when authored from scratch) for a round-trip-stable re-import.
_Avoid_: constant work view, fixed-work tab, CW editor.

### Visualisation

**Histogram**:
The per-**Scenario** tuple `{ counts, binCenters, binWidth }` produced by `buildHistogram` from a sorted `Float64Array` of per-**Iteration** totals. Exactly one Histogram per Scenario per **Run** — `|groups|` per Run at the org level ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)).
_Avoid_: chart data, frequency table, bar data.

**Bin**:
One fixed-width interval over the effort (person-month) axis. There are exactly 60 Bins per Run, shared across all **Scenarios** in the Run so their bars are directly comparable.
_Avoid_: bucket, band, slot.

**Global histogram range**:
The shared `(globalMin, globalMax)` interval used to compute every Scenario's Bins in a Run. `globalMin` equals the **Constant work** shift (`fixedEffort`; `0` when none is loaded — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md)) — and as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md), since each **Scenario** now carries its *own* per-Group shift, the shared `globalMin` becomes the **minimum** per-Group shift across the Run (often `0`); `globalMax` equals `max(Outlier clip of each Scenario, globalMin + 1)` over all `|groups|` Scenarios. Computed once per Run in `runSimulation`.
_Avoid_: chart range, x-range, plot range.

**Outlier clip**:
The P99.5 of a Scenario's sorted distribution, used as that Scenario's contribution to the upper bound of the Global histogram range. Purely a *display* decision — it does not change the simulated distribution and does not affect any percentile or `P(effort > capacity)` value reported in the stats table.
_Avoid_: chart clip, tail clip, P99.5 cap.

**Marker**:
A user-defined (or auto-managed) threshold on the effort (PM) axis of a per-context Histogram chart, surfacing simultaneously as a labelled dashed vertical line on the chart and as an auto-inserted row in the corresponding Stats table reporting **Probability of exceedance** against every **Scenario**. Stored as `{ id, label, value, color, isCapacity }` in the **Marker store**. The **Capacity** line *is* a Marker — the entry with `isCapacity: true` — drawn by the same `markersPlugin` `afterDraw` hook (`index.html:2174-2243`) as every other marker; the only asymmetry is that the **Marker dialog** disables the value-input and hides the delete button on capacity markers, and `ensureCapacityMarker` overwrites `value` on every **Run** while preserving `label` and `color` (see [ADR-0015](docs/adr/0015-capacity-as-auto-managed-chart-marker.md)). User markers are added via the per-context `＋ Add marker` button and edited via click-to-edit on the chart (10 CSS-pixel hit-target) — see [ADR-0025](docs/adr/0025-per-context-marker-system.md). The per-marker stats-row block reuses the **Risk tier** colouring uniformly with the capacity row.
_Avoid_: threshold (the abstract concept), annotation, line, indicator.

**Marker store**:
The module-scoped `markerStore: contextKey → Marker[]` map (`index.html:2975`) — keyed by the chart's `contextKey` (`'org'` for the org-level chart, `'team-{idx}'` for each team's chart in the **Team Level tab**). Each context's array is lazily initialised on first read via `getMarkers(key)`. Markers do *not* cross contexts: a marker added on the org chart never appears on a team chart, and vice versa. The auto-managed **Capacity** marker exists exactly once per context, created on first **Run** by `ensureCapacityMarker(contextKey, value)` and updated in place on subsequent Runs — see [ADR-0025](docs/adr/0025-per-context-marker-system.md). Persisted across browser sessions only via the per-context **Marker CSV** (`markers-${contextKey}.csv`, `saveMarkersToCSV` / `triggerLoadMarkers`).
_Avoid_: marker registry, marker list, threshold store.

**Marker dialog**:
The modal overlay (`#marker-overlay`) opened by `openMarkerDialog(contextKey, markerId)` (`index.html:3020-3039`) to add or edit a **Marker**. Carries a name input, a value (PM) input, an 80-colour swatch palette (`COLOR_PALETTE`, `index.html:2954-2972`) arranged in 8 rows × 10 columns (grays, reds, oranges, yellows, greens, teals, blues, purples), and `Save` / `Delete` / `Cancel` controls. Opens on the per-context `＋ Add marker` button click *or* on a chart canvas click within 10 CSS-pixels of an existing marker's vertical line. The value-input is disabled and the delete button hidden when the target marker has `isCapacity === true` — see [ADR-0015](docs/adr/0015-capacity-as-auto-managed-chart-marker.md). Closes on `Save`, `Delete`, `Cancel`, or backdrop click; on save and delete, `rerenderContext(contextKey)` rebuilds the chart and stats table in lockstep.
_Avoid_: marker modal, threshold editor, add-marker form.

**Marker CSV**:
The per-context export/import file (`markers-${contextKey}.csv`) used to persist a chart context's **Marker store** across browser sessions, written by `saveMarkersToCSV` (`index.html:3204-3219`) and read by the marker-load `change` handler (`index.html:3231-3265`). Fixed schema: `label,value,color,is_capacity`, with the label JSON-encoded (so commas/quotes survive the round-trip) and the booleans written bare. Loading replaces `markerStore[contextKey]` *wholesale*; the capacity entry's `id` is regenerated deterministically to `cap-${contextKey}` and non-capacity ids are regenerated randomly. A loaded capacity row whose `value` parses as a positive number also writes back to the sidebar `#capacity` input and triggers a **Run** — see [ADR-0025](docs/adr/0025-per-context-marker-system.md).
_Avoid_: marker file, threshold export, marker dump.

**Initiative matrix**:
The wide table inside each **Projection section** (`.proj-init-table`) listing every in-scope **Initiative** for the team as a row, with columns `Jira Key`, optional `KR` (the **Key Result**; only present when at least one row in the section — Initiative or **Constant work** — has a non-empty value, gated per-section by `hasKr = allInits.some(i => i.kr)` — see [ADR-0022](docs/adr/0022-optional-key-result-column.md)), `Initiative Name`, and one column per **Quarter** in `allQuarters`. A row's **Category** badge appears in *exactly* the column matching that initiative's `quarter`; other quarter columns are blank. The badge renders as a uniform neutral grey label carrying the Category text (no per-Category colour — colour is carried by **Group**, not Category; see [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)); a (Blank)-categorised initiative renders the badge as `(Blank)` in italic grey. Rows are sorted case-insensitively by Category alphabetically with the **(Blank) sentinel** always last, then by Jira key. **Constant work** rows — as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md), only those whose **Category** ∈ the **Projection group**'s members — are appended after the sorted Initiatives, tinted soft green (`#f0fdf4`) with the t-shirt size and estimated PM in the name cell (`[M · ~2.12 PM]`) — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md). The `<tfoot>` holds two summary rows per section: `Initiatives count` (per-quarter Category count chips) and `Effort P50 (P25–P75)` (the **Effort projection band** values for each quarter, driven by the **Projection group**); both rows' first `<td>` carries `colspan="3"` when the KR column is present and `colspan="2"` otherwise.
_Avoid_: initiative table, projection table, initiative grid.

**Effort projection band**:
The triple `{p25, p50, p75}` produced by the **Quick projection Monte Carlo** for one **Team** in one **Quarter** on the **Projection group**'s **Scenario** (with `fixedEffort = cwEffort`, where `cwEffort` as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) counts only constant work whose **Category** ∈ the Projection group's members). Surfaced two ways: (a) as a paired bar on the `Effort Projection by Quarter` chart — a semi-transparent indigo P25–P75 range bar overlaid by a narrower opaque P50 median bar; (b) as a footer row in the **Initiative matrix** showing `~{p50} PM` with `{p25}–{p75}` underneath. When the Quick projection is skipped (no initiatives match the Projection group's members, zero org λ, empty bootstrap pool, or empty `groupsStore`), all three values default to `cwEffort` — surfacing a flat band that reads as "constant-work only" in the chart — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md). The band is deliberately a *three-point summary*, not P10/P90: the 3000-iteration cap on the **Quick projection Monte Carlo** is calibrated to those three quartiles only — see [ADR-0020](docs/adr/0020-team-projections-cross-quarter-view.md).
_Avoid_: projection range, P25–P75 range (which is just one half of the band), percentile band.

### Column detection

**Column detector**:
A function that, given the parsed rows of a CSV, returns the header name corresponding to a known semantic column (Initiative key, **Category**, team, name, Epic→Initiative link, **Key Result**). Downstream code reads `row[headerName]`; nothing else interprets headers. Every detector returns a `string` *except* `detectKrCol` and `detectCategoryCol`, which return `string | null` because their columns are optional (the **Key Result** column is fully optional — see [ADR-0022](docs/adr/0022-optional-key-result-column.md); the **Category** column is *practically* required but yields `null` when missing, in which case every initiative's category resolves to the **(Blank) sentinel** — see [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)).
_Avoid_: parser, mapper, resolver.

**Content scan**:
The detector strategy that iterates over each column and computes the fraction of non-empty values matching a fixed regex (Jira-key pattern for keys). The first column whose match ratio exceeds the **Detection threshold** wins. Historically used for the MoSCoW column too (regex `/must|should|could|won.t/i`); [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md) removed that use because arbitrary **Category** strings can't be regex-fingerprinted — `detectCategoryCol` uses a header-name cascade exclusively.
_Avoid_: regex match, content sniff, value-based detection.

**Detection threshold**:
The fraction of non-empty values that must match the regex for a **Content scan** to claim a column. Currently `> 0.5` for the Initiative key column and `> 0.4` for the Epic→Initiative link column. Header-name fallback applies when no column clears the threshold. (The MoSCoW threshold of `> 0.5` is retired as of [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md) — the Category detector uses header matching only.)
_Avoid_: confidence, score.

**Detection fallback**:
The header-name lookup branch inside a **Column detector**, used when the **Content scan** finds no winning column or — for some detectors — checked *before* any content-based path. Per-detector branch order is asymmetric and *intentional*:
- `detectInitKeyCol` runs **Content scan** first, treats the **Sensible format** `jira_key` header (and the **Quirky format** `teams`) as a fallback.
- `detectCategoryCol` does not use **Content scan** at all (post-[ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)) — it runs a pure header-name cascade `category` → `moscow` → `emoji` and returns the first match (or `null`).
- `detectNameCol` and `detectTeamCol` check the **Sensible format** header *first* (`building_block`, `teams`) and only fall through to positional inference when it is absent — see [ADR-0021](docs/adr/0021-sensible-csv-format-dual-support.md).
- `detectKrCol` runs an exact-then-case-insensitive header lookup over a fixed vocabulary (`key_result`, `kr`, etc.) and returns `null` when nothing matches — see [ADR-0022](docs/adr/0022-optional-key-result-column.md).
_Avoid_: default column, header lookup.

**Recognised t-shirt size**:
A normalised size string (output of `normalizeSize`) that exists as a key in the active parameter table — the one `activeParams` (`index.html:1264`) currently references, **Synthetic parameters** or **Empirical parameters**. The two tables share the same key set by invariant (see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md)), so a size's recognition status does *not* change when the user flips the **Lognormal Parameters** radio. Used as the tie-breaker when two Epic rows share an `_epic_key` during within-file dedup — the row with a recognised size wins.
_Avoid_: valid size, known size.

### Pre-Run sidebar surfaces

**Data preview**:
The live sidebar block (`#data-preview`) that surfaces the *fitted* model inputs the simulator believes about the currently-loaded CSVs and quarter selection: **Historical quarter**, count of historical **Initiatives**, **Poisson λ**, **Bootstrap pool** size with its per-**T-shirt size** breakdown, **Target quarter**, and one per-**Group** initiative count `K` (one row per Group in `groupsStore`, labelled with the Group's name and colour) — and, as of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md), the **Constant work** PM folded into each Group beside its `K`, plus a line reporting any constant work whose **Category** is in *no* Group (`Constant work in no group: X PM across N rows — excluded`). Painted by `renderPreview` from the `preview` field of `prepareSimulationData`'s return value. Hidden until both **Initiatives CSV** and **Epics CSV** are loaded *and* at least one **Historical quarter** and one **Target quarter** are selected; thereafter monotonically visible. Re-paints automatically via `tryUpdatePreview` on every file-load, every epics-reset, every multi-select change, every Groups-tab edit, and (as of [ADR-0034](docs/adr/0034-editable-constant-work-tab.md)) every Constant-work-tab edit.
_Avoid_: summary, sidebar dashboard, input preview.

**T-shirt size reference**:
The collapsible `<details>` panel in the sidebar showing a static three-column table — `Size`, `Min PM`, `Max PM` — listing the documented P10/P90 band of each **T-shirt size** from `2XS` (`0.10`–`0.25` PM — see [ADR-0024](docs/adr/0024-2xs-t-shirt-size-extension.md)) to `XL+` (`10`–`11` PM). Hand-maintained mirror of the synthetic `T_SHIRT_PARAMS` map (see [ADR-0007](docs/adr/0007-lognormal-effort-distribution.md)); does *not* re-render when the **Empirical parameters** ↔ **Synthetic parameters** radio is flipped — the panel's contract is band-as-definition, not band-as-current-sampling-window, see [ADR-0026](docs/adr/0026-empirical-lognormal-parameters-mode-toggle.md). Always available, regardless of CSV state.
_Avoid_: size legend, sizing guide, t-shirt key.

**Column-detection debug**:
The collapsible `<details>` panel (`#debug-details`) whose `<pre id="debug-pre">` block shows two JSON sections — `Detected columns` (the live `detectedCols` map written by the **Column detectors**) and `Target Category breakdown` (the `preview.categoryBreakdown` map: category-string → count over `editedInitiatives` in the **Target quarter**). The trust surface for [ADR-0005](docs/adr/0005-content-based-column-detection.md)'s content-based detection *and* for [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md)'s header-cascade Category detection: lets the user verify which header was claimed for each semantic column and what categories are present in the loaded data. Hidden until the first successful **Data preview** paint; written via `textContent` (never `innerHTML`) to neutralise HTML-special characters in user-supplied headers.
_Avoid_: diagnostics, detection log, parser output.

## Relationships

- An **Initiative** belongs to exactly one **Quarter**, exactly one **Team**, and carries exactly one **Category** (which may be the **(Blank) sentinel**).
- An **Epic** belongs to exactly one **Initiative** (via **Initiative key**) and carries exactly one **T-shirt size**.
- A **Group** carries a flat set of **Category** values (the `members` array, including possibly the **(Blank) sentinel**); membership is the *sole* determinant of which Initiatives the engine includes in the Group's **Scenario** — no hardcoded exclusion based on Category value ([ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md)).
- A **Scenario** corresponds one-to-one with a **Group** in `groupsStore` for the duration of a **Run**: `K = #initiatives whose Category ∈ group.members` over the **Target quarter**'s `editedInitiatives`.
- Exactly one **Group** in `groupsStore` has `isProjection === true` (the **Projection group**); the Projection group drives the **Quick projection Monte Carlo** and every **Effort projection band**.
- A **Run** executes `iteration` independent **Iterations** per **Scenario**, producing one distribution of total effort per Scenario — `|groups|` distributions at the org level.
- A **Constant work** entry produces a deterministic PM shift applied to every **Iteration** *after* sorting — see [ADR-0023](docs/adr/0023-constant-work-csv-deterministic-shift.md). As of [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) the shift is **per Group**: a constant-work entry shifts a Group's **Scenario** only if its **Category** ∈ that Group's `members` — Category membership is the sole binding between a Group and its constant work. On top of that, each surface adds its own scope: org-level **Run** across *all* teams in the **Target quarter(s)**, **Team Level tab** filtered to one team, **Quick projection Monte Carlo** scoped to the **Projection group** per team-quarter. Constant work whose Category is in *no* Group is excluded (and surfaced in the **Data preview**). Constant work never participates in K counts for any Group.
- A **Run** produces one **Histogram** per **Scenario**; all `|groups|` Histograms in a Run share the same **Global histogram range** and the same Bin count.
- A **Run** also produces one **Stats** tuple per **Scenario**, computed from the same sorted distribution the **Histogram** is built from. The Stats tuple's `pExceed` field is the **Probability of exceedance** against the configured **Capacity**; its **Risk tier** classifies that probability into one of three colour bands.
- A **Marker** (when present) adds one extra row to the stats table whose value is `P(effort > marker.value)` — the same **Probability of exceedance** metric as the capacity row, classified into the same **Risk tier** bands. The capacity row itself is the hardcoded row produced by `renderStatsTableInto`, *not* a row in the per-marker block (which filters `!isCapacity`) — see [ADR-0025](docs/adr/0025-per-context-marker-system.md).
- The **Data preview** is read pre-**Run** and reflects the fitted inputs (**Poisson λ**, **Bootstrap pool**, per-**Group** `K`) the engine *would* consume if the user pressed **Run** now; it is the upstream-of-the-engine companion to the post-Run **Stats** table.
- The **Column-detection debug** panel is the user-visible audit of `detectedCols` — the same map the **Column detectors** write and that every downstream reader (`prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`, `renderInitiativesTable`) consumes; surfacing it as JSON in the sidebar makes the otherwise-opaque detection outcome reviewable before any **Run**.
- A **Run** produces one **Projection section** per **Team** appearing in the **Initiatives CSV** (not bounded by the **Quarter selector**'s selection — contrast with the **Team Level tab**). Each Projection section contains one **Effort projection band** per quarter where the team has in-scope **Initiatives** or **Constant work**.
- An **Effort projection band** is the `{p25, p50, p75}` output of a **Quick projection Monte Carlo** for one (Team, Quarter) cell — the same engine as the headline Run, with a smaller iteration budget and the **Projection group**'s **Scenario** only. **Constant work** for the cell — scoped to the Projection group's **Category** members ([ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)) — is folded in deterministically as a `fixedEffort` shift, so the band always sits at or above that scoped `cwEffort`.
- The **Initiative matrix** inside a **Projection section** lists every in-scope **Initiative** for the team (including **Constant work** rows — those matching the **Projection group**'s Categories, [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md) — at the bottom) with the **Category** badge in exactly the column matching the initiative's quarter; per-quarter footer rows surface the cumulative Category count chips and the **Effort projection band** as `~{p50} PM` with `{p25}–{p75}` underneath.
- The **Groups JSON** file is the persistent representation of `groupsStore`; it is the *only* persistence layer for **Group** definitions across browser sessions, and follows the same wholesale-replace load semantics as the per-context **Marker CSV** ([ADR-0030](docs/adr/0030-json-persistence-for-groups.md)). Loaded groups are committed to `groupsStore` immediately but charts and stats lag until the next **Run**, matching the **Initiatives tab**'s commit-on-Run discipline.
- An **Initiative** carries an optional **Key Result** ([ADR-0022](docs/adr/0022-optional-key-result-column.md)) — a free-text label detected by `detectKrCol` from a fixed vocabulary of header variants. The **Key Result** never reaches the simulation engine; it surfaces only as a conditional column in the **Initiative matrix**, gated per-section by `hasKr = allInits.some(i => i.kr)` so that a team whose initiatives carry no KRs renders the matrix without the column. **Constant work** entries carry their own **Key Result** via a parallel `r.key_result || r.KR || r.kr` lookup against the **Constant Work CSV**'s fixed schema, and participate in the same per-section gate.
- Every initiative-consuming surface (`prepareSimulationData`, `prepareTeamSimulationData`, `buildTeamProjections`, the projection-axis `extractQuarters` call) reads from `editedInitiatives` — the per-row shallow-clone of `parsedInitiatives` that the **Initiatives tab** mutates in place via inline edit handlers; the immutable `parsedInitiatives` is retained *only* to power the **Initiatives tab**'s dropdown option pool (`getUniqueColumnValues(parsedInitiatives)`) so the pool stays stable across user edits — see [ADR-0027](docs/adr/0027-editable-initiatives-tab-with-csv-export.md). Edits commit to `editedInitiatives` immediately but the other four tabs' charts and stats lag until the next **Run**.

## Example dialogue

> **Dev:** When the user picks `Q2 2026` as the **Historical quarter** and `Q3 2026` as the **Target quarter**, what are we fitting and what are we forecasting?
> **Modeller:** We fit Poisson λ from the **Initiative** count in Q2, and we build the bootstrap pool of **T-shirt sizes** from the Q2 **Epics**. We forecast Q3 by counting how many Q3 **Initiatives** belong to each **Group** in `groupsStore` — one count per Group — and running one **Scenario** per Group on those counts.
> **Dev:** What if the user wants the classic Must / Must+Should / Must+Should+Could view?
> **Modeller:** They define three overlapping **Groups** in the **Groups tab**: `Must` with members `{Must}`, `Must+Should` with `{Must, Should}`, `Must+Should+Could` with `{Must, Should, Could}`, and flag one of them as the **Projection group**. Three Groups, three Scenarios, same chart shape as before — but now under explicit user control, not hardcoded. They can save the set as `groups.json` and reload on the next session.
> **Dev:** What if an **Initiative** in Q3 has no **Epics** yet?
> **Modeller:** That's fine — we don't read Q3 epics for the forecast at all. The number of epics per initiative comes from λ (fit on the historical quarter), and the sizes come from the bootstrap pool. The target-quarter epics only appear in the **Team Projections** view, not in the org-level forecast.
> **Dev:** What about an **Initiative** whose **Category** is empty?
> **Modeller:** It resolves to the **(Blank) sentinel** — surfaced in the UI as `(Blank)` with parentheses. Whether it participates in any **Scenario** depends purely on whether any **Group** has the (Blank) sentinel in its `members` array. The auto-default `All` Group includes it; user-defined Groups include it only if the user adds it via the Members popover's `(Blank)` row.

## Flagged ambiguities

- "team" was used to mean both the **owning team of an Initiative** and the **team-scoped simulation context** in the **Team Level tab** — resolved: the column is the **Team** (the Initiative's owning team); the tab runs a per-team Run filtered by that column.
- "size" was used to mean both **T-shirt size** (label) and **person-months** (number) — resolved: T-shirt size is always the label; PM is always the number.
- "iteration" was used for both a single Monte Carlo draw and a Jira-style sprint — resolved: only the Monte Carlo meaning is used in this project. Use **Run** for "one press of the button."
- "quarter" can refer to a single quarter or the user's multi-quarter selection — resolved: the historical and target **Quarter selectors** are both multi-selects; "quarter" in domain talk usually means the selected set unless explicitly singular.
- "detection" was used for two distinct steps: identifying which header carries a semantic column (a **Column detector** via **Content scan** / **Detection fallback**) versus normalising a raw value once the column is known (`normalizeCategory`, `normalizeSize`) — resolved: *detection* picks the column, *normalisation* transforms the value.
- "range" was used for both a **T-shirt size**'s P10/P90 band and the chart's **Global histogram range** on the effort axis — resolved: a size's range is its *band* (input to the lognormal fit); the chart's range is the **Global histogram range** (output of `runSimulation`).
- "bin" had been informally used both for one of the 60 effort-axis intervals (**Bin**) and for a Category/MoSCoW bucket — resolved: Bin is reserved for the histogram; Category bucketing happens via **Group** `members`.
- "probability" was used for two distinct things: the per-**Iteration** sampling draws inside the Monte Carlo loop, and the reported fraction `P(effort > capacity)` in the stats table — resolved: the latter is **Probability of exceedance**, an *empirical fraction* over the completed **Run**, never an in-loop sampling probability.
- "exceed" / "overrun" / "breach" were used interchangeably for the same metric — resolved: the published term is **Probability of exceedance**; the cell label is `P(effort > capacity)`; the code field is `pExceed`. The comparison is *strictly* greater (`>`), not `≥`.
- "risk" was used both for the **Risk tier** colour class and informally for "any of the orange/red cells in the table" — resolved: Risk tier is the named classifier; individual cells *carry* a tier, they are not themselves "the risk".
- "MoSCoW" was used to mean both the *engine concept* of a hardcoded priority axis and the *user-facing label vocabulary* `Must|Should|Could|Won't` — resolved by [ADR-0028](docs/adr/0028-category-as-generalized-moscow.md): MoSCoW is no longer an engine concept; it is one possible **Category** vocabulary. Use **Category** when referring to the engine's axis, MoSCoW when referring to those four specific label strings as a convention.
- "group" — the canonical noun for a **Group** in `groupsStore` post-[ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md). Older code retains the variable name `moscowGroups` for the per-MoSCoW-bucket count map used by `prepareSimulationData`; that historical name is being renamed to `categoryCounts` as part of the [ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md) rewrite. Use **Group** strictly for the user-facing, persistent, named subset; use *bucket* or *count map* for the engine's internal per-value tally.
- "scenario" was used to mean *one of the hardcoded three cumulative MoSCoW forecasts* — resolved by [ADR-0029](docs/adr/0029-user-defined-groups-supersede-cumulative-moscow.md): a **Scenario** is now exactly one **Group**'s simulation run; the cumulative-three shape is recoverable by explicit Group definitions but no longer the engine's contract.
- "blank" vs "(Blank)" — the literal string `Blank` is a valid **Category** value a user may type; the **(Blank) sentinel** is the reserved value missing/whitespace categories normalise to. The UI always renders the sentinel with surrounding parentheses to keep the two distinguishable; in JSON the sentinel serialises as `null`.
- "projection" — the noun-phrase **Projection group** identifies the single Group flagged `isProjection`; the standalone **Team Projections tab** and **Projection section** identify cross-quarter team views. The two senses are related (the Projection group drives the **Effort projection band** inside every **Projection section**) but distinct — when in doubt, prefer the full noun phrase.
- "constant work scope" — pre-[ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md), **Constant work** was added uniformly to *every* **Scenario** (a single global `fixedEffort` shift) and its **Category** was read only for the **Initiative matrix** badge. Resolved by [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md): constant work folds into a **Group** *only* when its **Category** ∈ that Group's `members`; "all constant work in every group" is retired. When discussing constant work, always state the Group scope.
- "exactly the same" (Category match between **Initiative** and **Constant work**) — informally read as a case-*sensitive* match. Resolved: **Category** comparison is `trim()` + case-insensitive everywhere ([ADR-0028](docs/adr/0028-category-as-generalized-moscow.md), [ADR-0033](docs/adr/0033-constant-work-scoped-to-groups-by-category.md)), so `Backend` (Initiative) and `backend` (Constant work) **merge** into one Category in the **Groups tab** and bucket together; there is no case-sensitive path.
